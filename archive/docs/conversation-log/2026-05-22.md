### Mini-decision: Markdown vs JSON for `tasks.md`

User asked why the choice matters. Honest answer: **for our human-in-loop workflow it matters less than Anthropic's argument suggests.** The "structural gravity" benefit prevents autonomous agents from prematurely flipping a checkbox without verification. Our flow is *Claude implements one task → Claude reports → user reviews → next task*. The wrong-flip can't propagate past the human check.

**Resolution**: stay with Markdown for `tasks.md`. Consistent with rest of `specs/v0.X/`. Human-readable at a glance. Cleaner diffs. Revisit if v0.2+ adds an autonomous "run-all-tasks-no-review" mode.

### Mini-decision: log auto-extract errors as structured JSON (refinement of FR-10)

Hightower's structured-error pattern is a real improvement to our `auto-extract-memory.sh` logging. Current: writes exit codes and freeform text to `context/sessions/{date}.extract.log`. Better: each entry is a JSON line with `{ts, success, error_category, observation_count, skipped_reason}`. Easier to parse later for analytics. Will fold into design.md as a refinement, not a new FR.
