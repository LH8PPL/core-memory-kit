# Google Cloud's "Always-On Memory Agent" — continuous LLM consolidation on Gemini 3.1 Flash-Lite

**Date:** 2026-07-21
**Source:** `<local-wiki>/raw/Google Cloud's Always-On Memory Agent Replaces RAG and Embeddings With Continuous LLM Consolidation on Gemini 3.1 Flash-Lite.md` (marktechpost.com, published 2026-07-18, byline Michal Sutter) — reporting on `GoogleCloudPlatform/generative-ai`, path `gemini/agents/always-on-memory-agent` (README + `agent.py` read directly from GitHub `main`, 2026-07-21).

## What it claims

A marktechpost writeup of a sample Google Cloud added to its official `generative-ai` GitHub repo: a "lightweight background agent that never stops," built on Google ADK + Gemini 3.1 Flash-Lite, storing memory in SQLite with "no vector database and no embeddings." Three sub-agents (Ingest / Consolidate / Query) sit behind one orchestrator. IngestAgent extracts summary/entities/topics/importance from any of 27 file types (text/image/audio/video/PDF) dropped in `./inbox`. ConsolidateAgent runs "every 30 minutes," reviews unconsolidated memories, "finds connections," and "merges related memories," generating a synthesized summary + one insight while idle. QueryAgent reads memories + consolidation insights and answers with cited memory IDs. The article's own comparison table names the system's main limitation as "Query reads up to 50 recent memories."

## What the evidence actually shows

I read the article's markdown in full (no images embedded in the clipping — verified by grep, 0 image links present) and then fetched the actual source from GitHub (`README.md`, `agent.py`, `requirements.txt`, `docs/architecture.png`) since the article is a marketing rewrite of the README and the code is the authoritative mechanism source. The code confirms most claims literally but the word "consolidate/merge" in the README and diagram does **not** mean fact-level merging in the database — see Mechanism detail. `requirements.txt` (`streamlit`, `google-genai`, `google-adk`, `aiohttp`, `requests`) confirms the "no vector DB, no embeddings" claim — there is no FAISS/Chroma/sentence-transformers dependency; retrieval really is "read the last N rows and let the LLM read them."

One provenance detail worth flagging: the README's own install instructions clone `https://github.com/Shubhamsaboo/always-on-memory-agent.git`, not a `GoogleCloudPlatform` URL — i.e. the artifact as shipped inside Google's official samples repo textually points at what looks like a third-party origin repo. I did not verify whether that repo is the true upstream, a fork, or an unrelated coincidence of naming — recorded as unverified, not asserted as fact.

## Mechanism detail

**Storage schema** (`agent.py::get_db`, SQLite, 3 tables):
- `memories(id, source, raw_text, summary, entities JSON, topics JSON, connections JSON, importance REAL, created_at, consolidated INT)`
- `consolidations(id, source_ids JSON, summary, insight, created_at)`
- `processed_files(path PRIMARY KEY, processed_at)` — dedupes the file watcher by path only (not content hash).

**Ingest** (`ingest_agent`, tool `store_memory`): one Gemini call per input (text or multimodal bytes ≤20MB) returns `summary` (1-2 sentences), `entities[]`, `topics[]` (2-4 tags), `importance` (0.0-1.0 float), then INSERTs one `memories` row verbatim (`raw_text` = the full description/content, kept forever, never screened). There is **no injection/secret/PII screen anywhere in the ingest path** — a dropped file's bytes go straight into the multimodal Gemini call and the LLM's own description is stored as-is.

**Consolidate** (`consolidate_agent`, timer default 30 min, or manual `POST /consolidate`):
1. Timer fires → code checks `COUNT(*) WHERE consolidated=0`; skips if `<2`.
2. LLM tool-calls `read_unconsolidated_memories()` → `SELECT * FROM memories WHERE consolidated=0 ORDER BY created_at DESC LIMIT 10` (hard cap of 10 per cycle — only the newest 10 unconsolidated rows are ever visible to a consolidation pass).
3. The LLM (single turn, no deterministic pre-filter, no BM25/embedding candidate-narrowing) freely decides connections/patterns/insight and calls `store_consolidation(source_ids, summary, insight, connections[])` **once** per cycle.
4. `store_consolidation` does three things: (a) INSERT one row into `consolidations`; (b) for each `{from_id, to_id, relationship}` in `connections`, append a `{linked_to, relationship}` JSON object onto **both** endpoints' `memories.connections` column — this is the system's only graph-edge structure, and it lives *only* in SQLite (no markdown source, not rebuildable from a text corpus); (c) `UPDATE memories SET consolidated=1 WHERE id IN (source_ids)`.
5. **Critically, nothing here edits, merges, or deletes the original `memories` rows' content.** "Consolidation" = write one summary/insight row + flip a boolean + append edge JSON. The README's "merges related memories" and the architecture diagram's "Merge related memories" label are not backed by any UPDATE/DELETE on `raw_text`, `summary`, `entities`, or `topics` — the originals persist unchanged and unbounded forever.
6. `source_ids` supplied by the LLM are **not validated against the corpus**: the `connections` loop silently no-ops on any id that doesn't `SELECT` a row, and the `consolidated=1` UPDATE using `id IN (...)` simply matches nothing for a hallucinated id — no error, no log entry, no rejection.
7. Once `consolidated=1`, a row is permanently excluded from all future `read_unconsolidated_memories()` calls — it can never be re-linked against memories that arrive later. Cross-time connections only ever form within a single 30-minute cohort of up to 10 items.

**Query** (`query_agent`): one LLM call reads `read_all_memories()` (`ORDER BY created_at DESC LIMIT 50`) + `read_consolidation_history()` (`ORDER BY created_at DESC LIMIT 10`), synthesizes an answer citing `[Memory N]`. Both reads are hard, silent recency caps — no relevance ranking, no keyword/semantic filter, no signal to the user or the log that anything beyond the cap became unreachable. A memory or an insight simply falls out of query-visibility once 50 newer memories / 10 newer consolidations exist, with zero notice.

**Deletion**: `DELETE FROM memories WHERE id=?` (hard delete, `/delete` endpoint) and `/clear` (wipes all three tables **and** deletes the inbox folder's files). No tombstone, no archive-before-delete, no soft-delete of any kind.

**Scheduling/resumability**: a single `asyncio` timer loop (`consolidation_loop`), no persistence of "was a consolidation in progress," no batching-with-checkpoints — safety comes only from the fact that `store_consolidation` is one atomic multi-statement transaction, so a process kill mid-cycle just means "nothing was written, retry next timer tick with the same (still-unconsolidated) rows." This happens to be crash-safe at this small scale (≤10 items) but is not a designed resumability model — no artifact-derived resume point, no per-batch persistence, no bound on how large a backlog could grow before a cycle if the process were down for a long time (the `read_unconsolidated_memories` LIMIT 10 means a large backlog is processed 10-at-a-time per 30-min tick regardless, so a big backlog just drains slowly with no user-visible progress signal).

**`importance` field**: captured at ingest (0.0-1.0) but never read back anywhere in `agent.py` — not in any `WHERE`, `ORDER BY`, or prompt-side filtering logic. It is display-only metadata that plays no functional role in this build.

**Model choice rationale** (README "Why Gemini 3.1 Flash-Lite?"): explicitly chosen for latency + cost of *continuous* background operation over raw intelligence — the same tradeoff class the kit already makes with Haiku for `compress-session`/`daily-distill`/`weekly-curate` (design §8.5). This is one additional data point (from Google's own official sample, not a random third-party clone) that "cheap/fast model for background consolidation" is a converging industry choice — corroborating, not novel.

I also fetched and read `docs/architecture.png` (the repo's own diagram, not embedded in the source article) — it shows exactly the code's shape: one root orchestrator fanning out to Ingest/Consolidate/Query boxes, all three writing into one SQLite icon. It matches the code precisely except for the "Merge related memories" label under Consolidate Agent, which — per the code above — the implementation does not actually do.

## Relevance to core-memory-kit

**Task 95 / design.md §21 (dream re-curation engine) — graph_relevance: strong (as a negative exemplar); the anti-scope list in §21.6 reads almost like a direct rebuttal of this system's actual behavior.** Point-by-point:

- §21.2's "the LLM proposes; CODE decides... hallucinated ids rejected (D-230 bridge discipline)" is exactly the gap this system lacks: `store_consolidation`'s `source_ids` are LLM-supplied and unvalidated, silently no-op on a bad id rather than being rejected/logged. Confirms why §21's floor-then-one-call-then-code-decides shape is worth keeping strict.
- §21.2's deterministic floor ("nothing the floor can decide reaches the LLM") has no counterpart here — every consolidation decision, including which items are even "related," is 100% LLM judgment with no dedup/hash pre-filter.
- §21.6's anti-scope line "no LLM-decided which-wins... no LLM-counted recurrence... no unscreened raw-verbatim to the LLM" is validated by this system doing precisely those things: raw multimodal bytes go straight to Gemini uncredited, and there is zero screening equivalent to the kit's `screenBeforeCommittedWrite` (§21.3, Task 216).
- §21.2's Dreams contract ("archive-not-delete... source tier pruned only AFTER adoption") is the opposite of this system's `/delete` and `/clear` (hard DELETE, no tombstone, inbox files also destroyed).
- §21.5's "never-silent-skip: every skipped item carries a stated reason in the run log" is exactly the gap in this system's `LIMIT 50` / `LIMIT 10` query caps — older memories and older insights become unreachable with no log entry, no user notice, nothing like the kit's queue/audit-log discipline.
- The one place this system's shape *does* resemble §21: a fixed-interval batch pass that reviews "unconsolidated" items and writes derived output while idle — structurally the same rhythm as §21.4's "runs against the today→archive roll... on idle/cron machinery." But it lacks §21.4's resumability contract (ADR-0020: artifact-derived resume point, per-batch durable units) — its "resumability" is incidental (small atomic writes), not designed.

**Task 232 / ADR-0023 (graph-recall activation) — graph_relevance: strong (cautionary).** The `connections` JSON blob (`{linked_to, relationship}` appended per memory row on each consolidation) is a real, if crude, typed-edge graph — LLM-authored, stored only in SQLite, with no markdown/text source of truth and no way to rebuild it from anything but the DB itself. This is a direct counter-example to the hard constraint ADR-0023 holds (edges must be derivable/rebuildable, not a second, DB-only source of truth) — useful as a "don't do this" reference point when 232 designs how activated `links`/`superseded_by` edges get exposed to traversal.

**Tasks 161/203 (session-buffer compression timeouts / daily-distill starvation) — task95_relevance / graph_relevance: weak-to-none directly, but one adjacent point.** This system's consolidation trigger is a bare `asyncio.sleep(interval*60)` loop with no crash-recovery beyond "nothing persisted, retry with the same backlog next tick," no health-check equivalent to HC-10, and no protection against a long-down process facing a huge backlog beyond the passive `LIMIT 10`-per-tick drain. It's a smaller-scale illustration of exactly the class of "silent starvation, no outcome-check" bug D-298/Task 203 fixed for the kit's own cron path — not new information, but a real-world instance confirming the failure class is common when a background job's only signal is "did the timer fire," not "did the work finish."

## Borrow candidates

- **The ingest schema tuple (summary + entities[] + topics[] + importance) as a compact structured-extraction shape** for a single LLM call over arbitrary multimodal input — reasonable as a reference shape for a from-scratch extractor, though the kit's own auto-extract already does comparable structured extraction and this system's own `importance` field is unused dead weight (see Reject below), so borrow the *shape*, not the field list verbatim.
- **File-watcher + `./inbox` + `processed_files` dedupe-by-path pattern** as a lightweight reference if the kit ever wants a non-conversational "drop a file, get it captured" ingestion path (currently out of scope for core-memory-kit, which is conversation-transcript-driven) — worth a mental note, not an active task.

## Reject candidates (with reasons)

- **LLM-only consolidation with no deterministic floor and no id-hallucination guard** — reject. Directly contradicts §21.2's floor-then-code-decides discipline; this system's silent no-op on a bad id is exactly the failure mode D-230 was written to close.
- **Hard delete, no tombstone/archive** — reject. Contradicts §21.2/§21.6's archive-not-delete Dreams contract.
- **Unscreened raw multimodal ingest straight to the LLM** — reject. Contradicts §21.3's shared-screen requirement (`screenBeforeCommittedWrite`, Task 216); this system has no equivalent to Poison_Guard anywhere.
- **Fixed recency-window (`LIMIT 50`/`LIMIT 10`) as the entire retrieval mechanism, with no logged signal when items fall out of range** — reject as a recall strategy (the kit's FTS5+vector hybrid, Task 65, is strictly more capable) and reject the silent-cap behavior specifically against §21.5's never-silent-skip gate.
- **DB-only graph edges with no markdown/text source of truth** — reject against the markdown-as-truth + rebuildable-index invariant ADR-0023 holds as a hard constraint.
- **Capturing a scoring field (`importance`) that is never consumed anywhere in the code** — reject as a pattern; a caution against adding a metadata field to a schema without wiring it into any actual gating/ranking decision (the kit's Task 70.5 source-trust tags are the counter-example done right — they gate which class of op a claim can promote through).
- **Marketing the operation as "merge related memories" (README text + diagram label) when the actual DB operation never edits/merges/deletes the source rows** — reject as a documentation practice; noted here as a caution for how the kit itself describes Task 95's behavior once built, not as a mechanism to imitate.
