## What it claims

- README/ARCHITECTURE.md market it as "graph-first, not folder-first" memory: "Your agent maintains this graph — linking work notes to people, decisions, and competencies automatically," "part of the graph," "the graph does the rest," with an explicit `📊 Performance Graph` README section and CLAUDE.md sections titled "Graph-First Thinking."
- Also claims persistent, self-organizing, vault-first memory that "compounds" across sessions, with QMD badged as "semantic search" (`![QMD](...semantic%20search...)`).

## What the evidence actually shows

**No persisted/traversed graph store exists in code — "graph" means Obsidian's own backlink visualization, not a queryable structure the agent walks.** Every "graph" hit in the codebase (`grep -rli graph`) resolves to one of: (a) prose describing Obsidian's native graph view (`ARCHITECTURE.md:33` `Obsidian["Obsidian app<br/>(graph, Bases, CLI)"]`), (b) doc prose about wikilinks-as-organization, or (c) unrelated diagramming skills (excalidraw/mermaid). There is no `graph.ts`, no adjacency list, no node/edge table, no traversal function (`grep -rniE "adjacen|traversal|neighbou?r|graphdb|networkx"` across all `.ts` returns zero hits that are actual graph code — only comments using "traversal" to describe *absence* of one, e.g. `hint-state.ts:16` "no traversal surface").

`.claude/scripts/lib/wikilinks.ts` (216 lines) is the closest thing to graph machinery: it extracts `[[target]]` links, strips code regions, resolves aliases from frontmatter, and returns a **boolean** `Resolver(target, sourceRelPath) => boolean`. It exists solely to back a broken-link lint gate (`tests/vault-wikilinks.test.ts`, referenced in the file's own docstring) — it answers "does this link resolve," never "what does this note connect to" as a queryable/traversable structure. Nothing persists an edge list; nothing does BFS/DFS; nothing computes backlinks programmatically (Obsidian's own app does that, in its own UI, outside this repo's code).

This matches the D-153 discipline exactly: a "graph-first" flagship (3.4k★) ships a link-resolution boolean, not a graph.

**Retrieval is BM25 + local embeddings via a third-party CLI (`@tobilu/qmd`, github.com/tobi/qmd), not anything in this repo.** `README.md:125-153` is explicit: "QMD is where most of the agent's retrieval intelligence comes from... falls back to grep + the Obsidian CLI." `.claude/scripts/lib/qmd.ts` only resolves/spawns the external binary cross-platform; the actual index/search/embed logic lives entirely outside this repo, in `@tobilu/qmd`'s own SQLite store (`~/.cache/qmd/<index>.sqlite`, "First-time `qmd embed` downloads a ~328MB embedding model" per README). If `qmd` isn't installed, the repo's own fallback is literally `Grep`/`Glob`/`Read` (confirmed in `.claude/skills/qmd/SKILL.md`'s preference-order list).

**No offline/scheduled consolidation ("dream") pass exists anywhere in the shipped code.** Every mutation-adjacent mechanism found is one of two kinds, and neither is an LLM-driven batch re-curation:

1. **Zero-LLM deterministic detectors** (`lib/active-hygiene.ts`, 561 lines) — pure pattern/regex/frontmatter scanning for: completed-but-still-in-active notes, ungrouped topic clusters (document-frequency-guarded token overlap), oversized notes (25KB threshold, split-never-trim), stale open loops (`waiting on` / unchecked checkboxes 14+ days old), meetings-inbox pressure. Explicitly documented as "pattern-based, no LLM, conservative... NUDGES — it never moves files" (file docstring, lines 31-36). `tidy-fix.ts` (326 lines) is the one place that *acts* — but only on the two zero-judgment finding classes (archive completed notes, migrate misplaced memory files), and explicitly *refuses* every judgment-shaped finding with a pointer back to the agent tier: "The refusal list is what makes this safe to run from cron" (line 8).
2. **Live, user-invoked, agent-judgment slash commands** (`/om-tidy`, `/om-vault-audit`, `/om-project-archive`, `/om-weekly`) — markdown prompt files (`.claude/commands/*.md`) that instruct the *conversational* Claude Code agent to walk a checklist (frontmatter completeness, orphan detection, duplicate tags, status/folder alignment) live, in-session, asking the user on ambiguity ("Don't delete anything without asking"). `om-vault-audit.md` explicitly delegates structural work to two subagents (`vault-librarian`, `cross-linker`) but these are Claude Code subagent *prompts* (`.claude/agents/*.md`), not code — no batching, no typed op contract, no deterministic-floor/LLM-call split, no resumability, no provenance tracking, no trust tiers.

Nothing in the repo runs unattended on a schedule against the corpus. `tidy-fix.ts` is *designed to be cron-safe* (docstring says so) but the repo ships no cron registration — `om-tidy.md:41` even says the "scheduled home" of the hygiene pass is `/om-weekly`, itself a user-invoked slash command, not an installed scheduler. There is no equivalent of a deterministic-floor → one-batched-LLM-call → AUTO/QUEUE-op-split → gated-and-audited pipeline anywhere.

**Memory capture is agent-judgment-at-write-time, not Stop-hook auto-extraction.** `.claude/scripts/stop-checklist.ts` (85 lines, the Stop hook) prints a **reminder checklist** for the human/agent to act on ("Archive completed projects?", "New notes linked?", "Run `/om-vault-audit` if many notes were created/modified") plus concrete hygiene-drift findings from the same detector `active-hygiene.ts` uses at SessionStart — it does not extract or write any fact itself. The actual capture mechanism is: `classify-message.ts` (91 lines, `UserPromptSubmit` hook) regex-matches the user's raw prompt against signal patterns (`lib/matcher.ts`/`lib/signals.ts`) and injects a **hint** telling the agent to consider writing a note ("Content classification hints... Remember: use proper templates, add `[[wikilinks]]`, follow CLAUDE.md conventions") — the agent then decides, in-conversation, whether/what to write via normal `Write`/`Edit` tool calls, which `validate-write.ts` (PostToolUse hook) then lints post-hoc for location/frontmatter/link hygiene. There is no subagent or code path that extracts durable facts from the transcript without the live agent choosing to write them.

**The stored fact format is unstructured prose, not typed records.** `brain/Memories.md`, `brain/Key Decisions.md` etc. ship as near-empty templates: frontmatter has only `description`/`tags`; content is a flat bullet list under a heading (e.g. `## Recent Context` / a bare `-`). No id, no trust tier, no source-trust tag, no timestamps-union, no supersession pointer beyond an optional prose `[[Decision Record]]` wikilink. Dedup/contradiction-resolution is entirely human/agent judgment at write time — no deterministic ID/hash floor exists anywhere in the repo.

## Mechanism detail (HOW, precisely)

- **Wikilink resolution** (`lib/wikilinks.ts`): `stripCodeRegions` (fence-aware line scan + backtick-run regex) removes literal `[[...]]` examples from docs before matching; `extractWikilinkTargets` regex `/!?\[\[([^\[\]\n]+?)\]\]/g` pulls raw targets, strips alias tails (`|`, `\|`, `&#124;`) and `#fragment`s, drops asset extensions and `{{template}}` placeholders; `buildResolver(files, aliasesByFile)` builds three lookup sets (full path, basename, alias) and returns a closure `(target, sourceRelPath) => boolean` handling `./`/`../` relative resolution (with a `normalizeRelative` that rejects `..` past root), path-suffix matching, and case-insensitive bare-name/alias matching.
- **SessionStart injection** (`session-start.ts`, `lib/session-start.ts`): reads `hookInput.source` (`startup`/`clear`/`resume`/`compact`) into a `mode` via `injectionMode`. When `mode === "full"` it emits North Star excerpt (anchored at `## Current Focus`, struck-through completed bullets dropped, 30-line cap), the brain-topic index (`brainIndex()` — filename + frontmatter `description`, no content), and a full vault file listing (high-volume dirs collapsed to one count line via `COLLAPSED_DIRS`). On `resume`/`compact` those three static sections are replaced by one `### Context Pointer` line saying they're unchanged and already in-conversation; only volatile sections (recent git changes, open tasks, active work, hygiene drift, QMD self-heal notes) re-inject. Every emission ends with `formatInjectionSize(Buffer.byteLength(body))` — a literal "here's what this cost in bytes" meter on stdout.
- **QMD self-heal**: before emitting anything, spawns a detached, unref'd, `stdio:"ignore"` re-index child pinned to `tmpdir()` cwd (never the vault dir, so a later `rm -rf` of the vault can't race a stale process handle on Windows) via `buildQmdCommand`; separately runs a bounded (`timeout:5000`) `qmd status` preflight and, if the native `better-sqlite3` binding is ABI-mismatched (Node upgrade broke it), spawns `npm rebuild better-sqlite3` (bounded `timeout:20000`) and reports success/failure inline.
- **Hygiene detectors** (`lib/active-hygiene.ts`): five independent pure functions (`findCompletedInActive`, `findUngroupedClusters`, `findOversizedNotes`, `findOpenLoops`, `findInboxPressure`) composed by `scanActiveHygiene`, shared verbatim between `session-start.ts` (read-only report), `stop-checklist.ts` (read-only report), and `validate-write.ts` (write-time trigger via `newNoteClusterCandidate` — same cluster function, called with just-written file's path). Topic clustering: tokenize titles (≥4 chars, strip date prefix + stopwords), group by token, keep only tokens present in 2..half of root files (document-frequency guard against generic words).
- **tidy-fix.ts's ACT/REFUSE split**: re-runs the same `scanActiveHygiene` output; for `completedInActive` it `git mv`s to `work/archive/<year>/` (year from frontmatter `date`, but skips whole clusters with mixed completion status); for misplaced-memory it copies → regenerates `MEMORY.md` index → byte-verifies the copy → only then deletes the original. Every other finding class is pushed into a `report.refused[]` array with a pointer to `/om-tidy`, never acted on. `--apply` flag gates real mutation; default is dry-run.
- **Once-per-session hint dedup** (`classify-message.ts` + `lib/hint-state.ts`): a JSON state file keyed by `session_id`, self-pruning at 7-day age / 200-session cap, tracks which signal hints already fired this session so the same nudge doesn't repeat every turn; missing/invalid `session_id` fails OPEN (every hint fires) rather than silently suppressing.

## Relevance to core-memory-kit

- **Task 95 (dream re-curation engine, design.md §21):** `task95_relevance: none` for mechanism-borrowing — obsidian-mind has no analog to §21.2's three-stage pass (deterministic floor → one batched LLM call → AUTO/QUEUE op split with provenance + gates). Its closest thing is `tidy-fix.ts`'s ACT/REFUSE split, which is a *structurally convergent instinct* (separate zero-judgment automatable ops from judgment-requiring ones, make the automatable subset cron-safe) implemented with **zero LLM and zero batching** — worth citing in §21 as an independent confirmation that "deterministic-floor-first, judgment-refuses-loudly" is a sound split, but it offers no mechanism for §21's actual hard part (the batched dedup/contradiction-resolution LLM call, id validation, θ=0.80 similarity gating). Its "judgment tier" is a *live agent conversation* (`/om-vault-audit`), not an offline pass — the opposite of §21.4's "runs against the today→archive roll... on idle/cron" scheduling requirement.
- **ADR-0023 / Task 232 (graph edges — activate `related:`/`[[slug]]` into an `edges(src,dst,type)` table, recursive CTE for supersession chains):** `graph_relevance: weak — borrowable code pattern, not an architecture`. `wikilinks.ts`'s `extractWikilinkTargets` + `extractAliases` + `buildResolver` is a clean, tested, dependency-free implementation of exactly the parsing primitive Task 232 needs (`[[slug]]` extraction with alias resolution) — but it's wired to a boolean lint check, not to a persisted edges table, so it's a *reference for the parser*, not for the graph itself. The repo independently confirms ADR-0023's own finding (mem0/MemOS/Letta ship no real graph despite the label) — a 10th data point for the "graph memory flagships under-deliver in shipped code" pattern the ADR's research sweep already documented across 9 other systems.
- **Task 233 (recall-trigger hybrid — existence advertisement + gated hint):** directly relevant and the strongest match in this repo. `brainIndex()`'s pointer-list (`- [[brain/Foo]] — <frontmatter description>`, content never loaded) and `formatInjectionSize`'s per-injection byte meter are working, shipped implementations of exactly Task 233's "existence advertisement... one live metadata line" bullet — a second concrete precedent alongside the Letta borrow already cited in the task. The **source-aware injection split** (`mode: "full"` vs resume/compact swapping static bulk for a one-line pointer) is a technique not currently named in Tasks 161/203/233 and is a plausible borrow for SessionStart injection-budget efficiency generally (see honest gap below on whether Claude Code's hook `source` field is available/reliable the same way).
- **Tasks 161/203 (context-compaction / session-buffer compression):** `task95_relevance`-adjacent but weak — `pre-compact.ts` is a raw `.jsonl` transcript **backup-only** hook (copy to `thinking/session-logs/`, retain 30, prune older), with no summarization/distillation logic at all. It is meaningfully *shallower* than core-memory-kit's own PreCompact roll (which distills, per design §8.5) — useful mainly as a negative data point (a popular flagship's PreCompact hook does the minimum: preserve raw data, do nothing smart with it).
- **Poison_Guard / screening (Task 216, ADR-0019):** no analog found — `validate-write.ts` lints structure/location/hygiene, never scans for secrets or prompt-injection content. Not investigated further (out of scope for this pass, flagged as a gap below).

## Borrow candidates

1. **Injection-size meter** (`formatInjectionSize`, one line appended to every SessionStart/hook emission reporting the actual byte cost of what was just injected) — cheap, deterministic, directly checkable transparency device; relevant to any of core-memory-kit's own snapshot-injection surfaces if not already present.
2. **Existence-advertisement-as-pointer-list pattern** for Task 233: instead of (or alongside) the planned single "fact count / scopes / last-write recency" line, `brainIndex()`'s per-topic one-line-description list is a concrete second reference for "tell the agent what exists without loading content" — worth a look when 233 is built.
3. **Source-aware injection split by hook `source`** (`startup`/`clear` = full; `resume`/`compact` = pointer-only for static sections) — a token-economy technique for avoiding re-injection of unchanged bulk content across a compact/resume boundary within one session. Worth checking whether Claude Code's SessionStart payload's `source` field is already read/used anywhere in core-memory-kit's own hook code; if not, this is a candidate for the injection-budget class of work (§7.1).
4. **Deterministic-floor / judgment-refuses split as a cron-safety pattern** (`tidy-fix.ts`): the "act only on zero-judgment findings, refuse everything else loudly with a pointer to the agent tier, so the deterministic subset is safe to run unattended" shape is a clean, minimal reference for how to keep an automated pass honest about what it will/won't decide — independently convergent with design §21.6's "no LLM-decided which-wins" anti-scope line.
5. **`wikilinks.ts`'s parsing regex + code-region-stripping + alias-resolution logic** — directly reusable *reference* (not code, license/dependency-free reimplementation) for Task 232's `[[slug]]` extraction, specifically the fence-aware `stripCodeRegions` (so literal `[[...]]` syntax examples in docs aren't parsed as edges) and the table-escaped-pipe alias handling (`\|` and `&#124;`), which are easy-to-miss edge cases.
6. **Cron-worker cwd pinned to `tmpdir()`, never the working tree, for detached fire-and-forget spawns** — a specific, named Windows-compat technique (avoids a stale child process holding a directory handle during a later `rm -rf`/test-cleanup race) that's directly applicable to core-memory-kit's own detached-spawn sites given the project's documented Windows spawn-layer pain (CLAUDE.md verification-rules section, multiple precedents).

## Reject candidates

- **Adopting "graph" as a marketing/architecture claim without code to back it** — reject outright; this repo is itself the counter-example (see D-153 note above). No action needed beyond citing it as a data point.
- **QMD as a dependency** — reject for core-memory-kit: it's a third-party CLI with a ~328MB embedding-model download and its own SQLite store outside the project; core-memory-kit already has its own semantic backend decision (ADR-0015, sqlite-vec + local ONNX). Not a fit, not evaluated further.
- **Live-agent-judgment slash commands as the consolidation mechanism** — reject for Task 95's purpose specifically: core-memory-kit's design explicitly wants an *offline*, unattended, resumable pass (§21.4), and obsidian-mind's model requires an attentive human in a live session to invoke `/om-vault-audit` and answer ambiguous cases. Fine for obsidian-mind's PM/1:1-tracking domain (judgment calls about org/people context genuinely may need a human), wrong shape for what Task 95 is trying to build.
- **Unstructured prose fact storage (no id/trust/timestamps)** — reject; core-memory-kit's fact-file schema (id, type, trust, source-trust tag, timestamps-union) is already a deliberate improvement over exactly this shape, and obsidian-mind's own hygiene detectors can't do content-level dedup as a direct consequence (they only catch structural drift, never "these two notes say contradictory things").
- **ShardMind as a packaging/distribution model** — not evaluated for adoption; it's a separate, general-purpose install/update lifecycle tool by the same author with its own managed/unmanaged-file and fingerprinted-hook-rerun concepts. Interesting in isolation (bootstrap/personalize/post-update hook lifecycle split) but out of scope for a memory-mechanism pass and not investigated at code level here — flagged as an honest gap, not rejected on evidence.
