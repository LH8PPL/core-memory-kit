## Method note — Track A vs Track B, and why it matters for this synthesis

Design §8.0 draws a hard line the kit already committed to: **Track A** (`sessions/now.md → today-*.md → recent.md → archive.md`, §8.2.x — "what happened when") and **Track B** (`MEMORY.md` → `context/memory/*.md` fact graduation + Task 95's dream re-curation, §21 — "what's true"). Most of this pass's corpus (the four "Dreaming" articles, Google's Always-On agent, EverOS's Reflection) markets itself as a single undifferentiated "consolidation" feature that does BOTH at once — summarize old conversation history AND merge/dedupe the durable fact store in the same pass. The kit deliberately does not: §8.2.x's Haiku compress is Track-A-only (lossy summarization of a session diary, never touches fact provenance), and Task 95's three-stage floor→LLM→AUTO/QUEUE pass (§21.2) is Track-B-only (dedup/merge/supersede over facts, raw-transcript-grounded per D-44/D-352, never in-place-rewrites the diary). A technique lifted from a Dreaming-style article without checking which track it actually targets is a category error — several of the sources' own internal descriptions blur this line (Google's ConsolidateAgent operates over one flat `memories` table that is simultaneously the session log AND the fact store, because it has no Track A/B split at all). This note keeps the split explicit throughout; where a technique is genuinely Track-B/Task-95 territory it is named as such and NOT folded into the §8.2.x mapping the task brief asked for.

## 4. Special scrutiny — Google's "replaces RAG and embeddings with continuous LLM consolidation" claim

**The claim, as marketed:** a background agent "replaces RAG and embeddings" via "continuous LLM consolidation" — implying consolidation is doing the WORK retrieval used to do.

**What the code (`agent.py`, read directly from `GoogleCloudPlatform/generative-ai`) actually shows, mechanism by mechanism:**

1. **"No vector DB, no embeddings" is literally true** (`requirements.txt` has no FAISS/Chroma/sentence-transformers) — but this is not evidence retrieval got REPLACED by something better; it's evidence retrieval was **removed**.
2. **QueryAgent does not rank or retrieve at all.** It reads `SELECT * FROM memories ORDER BY created_at DESC LIMIT 50` + `LIMIT 10` consolidations and hands the whole batch to one LLM call to synthesize an answer. There is no BM25, no cosine similarity, no keyword match, no relevance score of any kind deciding WHICH 50 rows the model sees — only recency. Once the corpus exceeds 50 memories, older content becomes **silently, permanently unreachable** to Query with zero signal to the user or the log — exactly the failure §21.5's "never-silent-skip" gate is designed to prevent.
3. **"Consolidation" does not consolidate the fact store's content.** `store_consolidation` never `UPDATE`s or `DELETE`s a source row's `raw_text`/`summary`/`entities`/`topics` — it only inserts one new `consolidations` row, appends a `{linked_to, relationship}` JSON blob to two rows' `connections` column, and flips a `consolidated` boolean. The README's own "merges related memories" claim and the architecture diagram's "Merge related memories" label are **not backed by any code that merges anything** — verified by reading the full function body, not inferred from absence.
4. **Consolidation cannot compound over time by construction.** Once a row's `consolidated` flag is set, `read_unconsolidated_memories()` permanently excludes it from every future cycle — so a memory can only ever be linked to the ≤10 other memories that happened to arrive in its own 30-minute cohort, never to anything ingested later. The marketing's "builds new understanding while idle, like sleep cycles" framing is not supported by a scheduler that provably cannot connect memory #3 (ingested Monday) to memory #47 (ingested Thursday) once #3 has already been through one consolidation cycle.
5. **The one graph-shaped artifact this system produces (`connections` JSON) is DB-only** — no markdown/text source of truth, not rebuildable — which is exactly the shape ADR-0023 already rejects for the kit's own graph-recall design.
6. **No screening, no validation of LLM-supplied ids, no tombstone on delete.** Raw multimodal bytes go straight into the extraction call with nothing resembling Poison_Guard; a hallucinated `source_id` in a consolidation call silently no-ops rather than being rejected/logged (violates the D-230 bridge discipline §21.2 already enforces); `/delete` and `/clear` are hard deletes with no archive step.

**Verdict: the headline does not survive contact with its own implementation.** What actually happened is not "consolidation replaced retrieval's job" — it's "retrieval was deleted and replaced with an unranked, silently-capped recency window," while the "consolidation" step produces a side artifact (one summary+insight row, a DB-only edge blob) that never feeds back into what Query actually reads beyond its own `LIMIT 10`. This is measurably weaker than the kit's own FTS5+vector RRF hybrid (0.941 R@5 on the Task-99 benchmark, D-360) on every axis the kit's design already treats as load-bearing: rankedness, reachability past a size threshold, and honest failure signaling. It is a genuine, code-verified counter-example worth citing precisely because it's an official Google Cloud sample repo, not a random clone — the gap between the marketing headline and the shipped mechanism is the most extreme in this entire 29-source pass.
