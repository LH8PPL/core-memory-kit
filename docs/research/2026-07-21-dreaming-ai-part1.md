# Dreaming AI, Part 1 — "Your AI Should Dream. Mine Does."

**Date:** 2026-07-21
**Source:** `<local-wiki>/raw/Part 1 of Dreaming AI — Your AI Should Dream. Mine Does.md` (Medium, Jason Croucher, published 2026-04-08) — <https://medium.com/@jason.croucher/part-1-of-dreaming-ai-your-ai-should-dream-mine-does-fa47ed08e742>

## What it claims

Two ideas, arrived at independently, describe the same four-step cycle for maintaining an LLM's accumulated knowledge over time:

1. **Andrej Karpathy's "LLM Wiki" gist** — treat the LLM as a librarian that maintains a structured, cross-linked markdown wiki over raw source material, rather than re-deriving answers from scratch each query.
2. **Anthropic's "Auto Dream"** — an *undocumented, currently-gated* Claude Code feature: a background subagent that runs between sessions, reads accumulated memory/session-transcript signal, and rewrites the project's memory/`CLAUDE.md` files — merging related notes, resolving contradictions, deleting stale entries, updating the index. The author frames it as "REM sleep for your AI."

The author had Karpathy's wiki pattern running against an Obsidian vault, but found Auto Dream itself switched off (`/memory` reports `Auto-dream: off · never` — Anthropic controls the rollout). Rather than wait, he reconstructed the other half himself: a **DIY cron-triggered dream**, using the actual Auto Dream system prompt (extracted from the Claude Code binary by third parties) adapted to his own paths, run nightly via infrastructure he'd already built for an unrelated project (`claude-channels`).

He then names two open problems with what he built: (1) the same agent that wrote the memories is the one "grading its own homework" when it consolidates them (the self-evaluation trap, citing Anthropic's own research), and (2) the source-material inbox (`raw/`) keeps growing, so each nightly run spends more time re-figuring-out what's already processed than doing new work. Both are promised fixes "in the next post" (Part 2).

## What the evidence actually shows

- **Auto Dream's existence is third-party-reverse-engineered, not Anthropic-documented.** The chain of evidence in the article: Marco Kotrotsos ran `strings` on the Claude Code macOS binary and found internal identifiers `"autoDream"` and `"tengu_onyx_plover"`, plus a system-prompt fragment describing a background agent that rewrites `CLAUDE.md` files "without asking you first." Piebald-AI separately published the full extracted system-prompt text in a GitHub repo (`claude-code-system-prompts`). Neither source is an official Anthropic announcement or doc page — I did not find or fetch any Anthropic-primary confirmation (out of the 5-link cap, and not linked from the article itself). Treat "Auto Dream is real and gated-off" as **the article's sourced claim**, not an independently-verified fact.
- **The four-phase claim is real and specific**, verified by directly reading two of the article's embedded images (WebFetch cannot decode webp binaries as text, but the Read tool renders them):
  - A comparison table titled "Operation | Auto Dream (Agent Memory) | Wiki Compile (User Knowledge)" with five rows: **Orient, Gather, Consolidate, Prune, Index** (the running prose calls it "four phases" but the table separates Prune from Index into five operations).
  - A flowchart of the LLM-Wiki architecture: `Raw Sources` --"drop in raw/"--> `LLM Librarian`; `CLAUDE.md` (schema+conventions) --"instructs"--> `LLM Librarian`; `User Query` --"ask"--> `LLM Librarian`; `LLM Librarian` --"compile / check shelves first / update article"--> `Curated Wiki`; and a feedback edge `Curated Wiki` --"gap found → search raw/"--> `LLM Librarian`.
- **The claude-channels infrastructure numbers are as claimed by the repo/README (fetched via WebFetch's AI summary, not by reading the raw source files myself)**: 254 total lines replacing a 20,000-commit OpenClaw fork; the webhook-channel HTTP server is 93 lines on port 8788 (article text says the dream crons hit port 8790 — a numeric inconsistency between the repo's own port and the article's stated port, unresolved, flagged as a gap below); CLAUDE.md routing layer 67 lines; crontab config 13 lines.
- **The self-evaluation trap claim is grounded in a real, separately-fetched Anthropic-attributed finding** (via the author's own earlier post): "an agent will identify legitimate problems and then talk itself into believing they aren't that bad" — and the fix that same post proposes is a **structurally separate second agent** (an Evaluator, differently prompted for skepticism, that interacts with the live artifact rather than trusting self-report) — not merely "try harder." The Part-1 dream system as built does **not** yet have this separation; it's a single Claude instance reviewing its own prior output.

## Mechanism detail (the HOW)

**LLM Wiki (Karpathy pattern, as implemented by the author):**
- Three folders: `raw/` (immutable source material — LLM reads but never modifies), `wiki/` (LLM-owned, cross-linked markdown: summaries, entity pages, concept pages), `output/` (ephemeral exports).
- `CLAUDE.md` is the schema file: tells the LLM the wiki's structure, conventions, and workflows.
- Two infrastructure files: `index.md` (content-oriented catalog of every wiki page) and `log.md` (append-only chronological record of ingests/queries/maintenance).
- Three workflows: **Ingest** (new source → extract → integrate across ~10-15 existing wiki pages → update entity pages, revisions, cross-refs), **Query** (search the wiki, answer with citations; good answers themselves become new wiki pages), **Lint** (periodic health pass: contradictions, stale claims, orphaned pages, missing cross-refs).

**Auto Dream's four-phase system prompt** (as extracted/summarized from the Piebald-AI-published text):
1. **Orient** — list the memory directory, read the index file, skim existing topics to avoid duplication, review recent activity logs by date.
2. **Gather Signal** — priority order: session logs (most recent 1–3 days) → contradicted existing memories → transcript search via **narrow grep terms**, with the explicit instruction "grep narrowly, don't read whole files."
3. **Consolidate** — update/create memory files in a specified format; **merge new signal into existing topic files rather than creating near-duplicates**; convert relative dates ("yesterday") to absolute dates; **delete contradicted facts by fixing the source** (i.e., an in-place overwrite, not an append-and-mark).
4. **Prune and Index** — keep the index file under a line/character budget; the comparison-table image states this concretely as **"Keeps MEMORY.md under 200 lines"** (I'm treating the image as the more reliable read of this number over WebFetch's separately-fetched textual summary of the same source page, which gave an inconsistent "~150 character entries" figure — likely referring to per-entry length rather than the file-wide budget, but I could not reconcile the two numbers from Part 1 alone); remove stale pointers; shorten verbose lines; resolve cross-file contradictions; return a summary of what was consolidated/updated/pruned.

**DIY reproduction, concretely:**
```
crontab (2:03 AM) → curl :8790/dream-memory → webhook → Claude → /dream-memory skill
crontab (3:33 AM) → curl :8790/dream-wiki   → webhook → Claude → /dream-wiki skill
```
- `/dream-memory`: takes the extracted Auto Dream system prompt, adapts it with local paths, runs the four-phase cycle against Claude's own memory files.
- `/dream-wiki`: runs the CLAUDE.md wiki-compile workflow against `raw/` — inventory what's new, triage each file, compile wiki articles, cross-link, update indices. A "manifest file for tracking what's already been compiled" is mentioned as existing infrastructure, but the article's own "problem #2" (the `raw/` inbox spending more time on already-processed files each night) is stated as **unresolved** despite the manifest — so its actual resumability mechanics are unclear/unverified from Part 1.
- Both jobs push their results to Telegram (push notification, not a pull-based check).
- The webhook is a *persistent-session pusher*, not a one-shot spawn: it pushes an event into an already-running Claude Code session, which reads `meta.path` and consults CLAUDE.md's routing table to decide what to execute — the inversion the author calls out ("channels actively push events" vs. passive tool-call waiting).

## Relevance to core-memory-kit

I read [design.md §21](../../specs/design.md) (the Task-95 dream re-curation engine's mechanism contract, D-352) before writing this section, per the task instructions.

**graph_relevance: weak** — the LLM-Wiki diagram's cross-linked `[[wiki links]]` and the Obsidian graph-view screenshot are an existence proof that a linked knowledge base is browsable/valuable, which lightly supports the *value* of Task 232 (activating `related:`/backlinks), but the article gives no new mechanism for graph edges — no edge schema, no traversal method, nothing Task 232 doesn't already have from its own ADR-0023 sweep.

**task95_relevance: strong** — the article's four/five-phase Auto Dream contract is a close structural cousin of design §21's three-stage pass (deterministic floor → one batched LLM call → AUTO/QUEUE op application), and several concrete points either validate or usefully diverge from the kit's already-settled design:

- **Validates the self-evaluation-trap concern design §21 already structurally answers.** The article names exactly the failure the kit's AUTO/QUEUE op-class split (§21.2 step 3) is built to contain: non-destructive/deterministic/reversible ops apply automatically, but anything lossy, generative, or touching a high-trust fact is forced into a **human-reviewable** `context/queues/recuration.md` diff. Where the article's fix (promised for Part 2) looks to be a **second, structurally-separate LLM evaluator**, the kit's answer is a **human arbiter via queue** — a different mitigation for the same named problem. Worth an explicit note in §21 (not a re-design) that this alternative (a second differently-prompted LLM pass ahead of the human queue, to cut queue volume/noise) was considered against the article's framing and is not currently in scope.
- **Directly contradicts §21.6's anti-scope on one concrete point, in the kit's favor.** Auto Dream's own Consolidate-phase instruction is "delete contradicted facts by fixing the source" — an in-place overwrite. Design §21.6 explicitly forbids this class of operation ("No DELETE op exists... No in-entry trail-lossy merges" — archive-not-delete, both sides retained via the Task-66 validity-window close). This is a genuine, citable point where the kit's settled contract is stricter than what Anthropic's own (leaked) prompt tells its own dream process to do.
- **"Grep narrowly, don't read whole files"** (Gather phase) is the same instinct as §21.2's top-K=5 candidate narrowing (FTS5 BM25 / embedding cosine θ≥0.80) before the one batched LLM call — independent confirmation of the narrow-then-batch shape, not a new technique.
- **Scheduling divergence, kit's favor.** Auto Dream (as DIY-reproduced) fires on a fixed wall-clock cron (2:03/3:33 AM) regardless of whether there's new signal. Design §21.4 runs "against the today→archive roll," composed with idle/cron + the Haiku cooldown, and Task 198's `no-new-facts` short-circuit proves idle sessions spawn zero judge calls. The kit's artifact/roll-triggered scheduling is the more principled answer to exactly the "runs the same files every night" complaint the article raises as problem #2.
- **Resumability.** The article's unresolved "raw/ keeps growing, dream re-scans it" problem is the same class ADR-0020/Task 204 (derive the resume point from artifacts, never a fresh sentinel) already targets in this kit, with concrete reference implementations (Task 148's byte-offset watermark, Task 225's committed import ledger) that are more specific than the article's undetailed "manifest file."
- **Provenance gap in their design, already closed in ours.** Auto Dream's four phases as described carry no mention of preserving source pointers through a merge/consolidate. Design §21.3 requires source-trust tags and source-pointer preservation (session ids + absorbed fact ids) through every condensed claim. Not a borrow — a confirmation the kit's contract already covers ground the article's system doesn't mention.

**Task 233 relevance: weak.** The wiki diagram's "ask → LLM Librarian → check shelves first" is an always-consult-on-query pattern, closer to always-search than the kit's judgment-pulled hybrid (ADR-0024) — the kit's own Task 149 recall-trigger survey already covered this design space more broadly; nothing new here.

**Task 161/203 (context-compaction/timeout family): weak-to-none.** The article doesn't discuss compression latency, timeouts, or input-size growth for the compaction step itself (its "growing raw/" problem is about *inbox triage*, not about a single compress call timing out) — no direct bearing on the D-171/D-174/D-298 timeout-and-starvation lineage.

## Borrow candidates

- **The Orient/Gather/Consolidate/Prune/Index column framing itself**, as a documentation device — a short comparison table mapping Task-95's three-stage pass onto this same five-operation vocabulary could make design §21 easier for a new contributor to onboard onto, if such a table doesn't already exist there. (Documentation suggestion only, not evaluated as a code change.)
- **The explicit design question** raised above: should the QUEUE class in §21.2 ever get a second, differently-prompted LLM pass before the human review, as a noise-reduction step — worth recording as a considered-and-currently-rejected alternative in §21 rather than leaving it unaddressed, since the article's own Part 2 promises a concrete answer to the identical problem for its system.
- **Push-notify-on-completion** (their Telegram report) as an observed pattern-gap: the kit currently journals nightly-run outcomes to NDJSON logs (`audit.log`, `curate.log`) that the user must pull via `cmk doctor`/`cmk search`, rather than push a "here's what last night's distill did" summary. Noted as a real UX pattern seen elsewhere, not a recommendation to build it — low priority, no existing task claims this ground.

## Reject candidates

- **Fixed-clock-time cron as the dream trigger** — the kit's artifact/roll-triggered scheduling (§21.4) plus Task 198's no-new-facts short-circuit is already the more principled answer; adopting a bare fixed-time cron would be a regression against work already done.
- **"Delete contradicted facts" as a literal operation** — §21.6 already and deliberately rejects this; the kit's archive-not-delete, validity-window-close approach is chosen specifically to preserve what Auto Dream's own prompt discards. Reject re-affirms an existing, already-correct decision, not a new finding.
- **Single self-reviewing-agent consolidation as sufficient** — this is the article's own self-critique of its Part-1 system; the kit's AUTO/QUEUE human-gated split is already a partial answer to the same problem, so importing the raw single-agent shape (with no second reviewer of any kind, human or AI) would be a regression from the kit's current design.
- **The unverified "manifest file" as a resumability mechanism to copy** — cannot recommend borrowing a mechanism whose operation I could not verify; the article states the problem it's meant to solve as still unresolved.
