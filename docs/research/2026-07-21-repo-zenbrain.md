The 6 PMA components are proprietary and run in the production system. This open-source package ships the algorithm library: 10 core algorithms + 10 advanced research modules (20 modules)."* This admission is buried in a collapsed detail-adjacent blockquote right before the comparison table that still lists all the checkmarks as if they're live product features.

## What the evidence actually shows

**Inventory:** TypeScript monorepo (npm workspaces + turbo), Apache-2.0, 4 packages (`core`, `algorithms`, `adapters/{postgres,sqlite}`) + `apps/playground`. ~7,733 LOC in `packages/core` + `packages/algorithms` (`.ts` src, excl. tests); adapters add another ~550. 31 `*.test.ts` files found; the "528 tests" figure was not independently re-run/verified. CI (`.github/workflows/ci.yml`) builds+type-checks on Node 18/20/22 (matrix); stars 14, forks 4, open issues 6, created 2026-03-23, last commit in the 50-commit shallow log 2026-07-17.

**The 7 layers are real** — `packages/core/src/layers/{working,short-term,episodic,semantic,procedural,core,cross-context}.ts`, each a real class wired into `MemoryCoordinator` (`packages/core/src/coordinator.ts`), each backed by a real SQL table in `packages/adapters/sqlite/src/index.ts::initSchema()` (`episodic_memories`, `learned_facts`, `procedural_memories`, `core_memory_blocks`, `cross_context_links`, plus a `knowledge_entities` stub). This part of the claim holds.

**Most of "15/20 neuroscience mechanisms" are NOT wired into the live system.** `MemoryCoordinator.store()`/`recall()`/`consolidate()` (the only entry points; `packages/core/src/coordinator.ts`) import from exactly 4 algorithm modules: `emotional` (tagEmotion/computeEmotionalWeight, for routing+decay), `context-retrieval` (encoding-context similarity boost), `similarity` (Jaccard, for cross-context entity dedup), and `fsrs` (spaced-repetition scheduling in `layers/semantic.ts`). Grepped every other exported module (`sleep-consolidation`, `sleep-simulation-selection`, `personalized-pagerank`, `hebbian`, `hebbian-two-factor`, `bayesian`, `dopamine-routing`, `hopfield-stm`, `spectral-health`, `ib-budget`, `surprise-gradient-memory`, `temporal-multi-route`, `fsrs-vmPFC`) for importers outside their own file + their own `__tests__/*.test.ts` + the package `index.ts` re-export: **zero hits in `packages/core` or `apps/playground`'s functional demo path.** `apps/playground/src/index.ts` (the only executable "demo") calls FSRS, Emotional, raw Hebbian scalar math, Bayesian, Negation, WorkingMemory, ShortTermMemory as isolated `console.log` snippets — it never constructs a `MemoryCoordinator`, never touches a graph, never runs sleep consolidation. These 13+ modules are pure, well-tested, standalone math libraries (each independently unit-tested — genuinely good code) that the product never calls.

**Sleep consolidation is a disconnected simulation, not a running pipeline.** `packages/algorithms/src/sleep-consolidation.ts` (`selectForReplay` / `simulateReplay` / `pruneWeakConnections`) is pure-in-pure-out: takes an in-memory array of `MemoryForConsolidation` (with a caller-supplied `hebbianEdges` array), returns stability/edge deltas — it never reads or writes storage. `MemoryCoordinator.consolidate()` (coordinator.ts:403-444) is a **completely separate, simpler** implementation that does NOT call any function from `sleep-consolidation.ts`: it re-scans up to 100 recent episodes every call, promotes any with `emotionalWeight > 0.5` to a semantic fact via one **unbatched LLM call per episode** (not the batched-pool pattern), and has **no idempotency guard** — `episodic_memories` has no `consolidated`/`promoted` column, so calling `consolidate()` twice re-promotes the same episodes into duplicate facts.

**No real graph store.** `knowledge_entities` (id, name, type, embedding) is `CREATE TABLE`'d in the SQLite adapter but has **zero INSERT statements anywhere in the codebase** — nothing populates it. `cross_context_links` (entity_a, entity_b) + `CrossContextMemory.createLink()` (`packages/core/src/layers/cross-context.ts:77`) exist and would write one edge row per call, but `createLink` is called **only from its own test file** — no auto-population path from `store()`/`recall()`. `CrossContextMemory.detectMergeCandidates()` does pairwise Jaccard-name comparison across a hardcoded context list, not graph traversal — no BFS/DFS, no recursive CTE, anywhere in the SQL or TS. `packages/algorithms/src/personalized-pagerank.ts` is a genuinely correct, well-documented power-iteration PPR implementation (317 lines, real math) — but it is a **pure function taking a `PPRGraph {nodes, edges}` as a plain argument**; nothing in the repo constructs that graph from storage. Its own doc comment is the smoking gun: it references `services/knowledge-graph/hebbian-dynamics`, `hybrid-retriever.ts`, and `services/knowledge-graph/hebbian-ppr.ts ("when implemented")` — **files that do not exist anywhere in this repository.** This confirms the algorithm was extracted from the (closed-source) production platform and the graph-construction bridge either lives only in the proprietary half or was never built. `hebbian.ts`'s own docstring says it outright: *"This module contains ONLY the pure mathematical functions. Database operations are left to the consuming application."* No consuming application in this repo does that.

## Mechanism detail (the HOW, precisely)

- **Store routing** (`coordinator.ts::resolveStoreType`, lines 600-627): explicit `type` > procedural-pattern-regex > `emotionalWeight > 0.5` → episode > `confidence > 0.9` → core (pinned) > default semantic fact. All stored content is also pushed into `WorkingMemory` (7±2 slots, decay 0.05).
- **Recall** (`coordinator.ts::recall`, lines 299-353): parallel per-layer search (episodic/semantic via embedding cosine through `formatForPgVector`/`<=>`; procedural via its own recall; core via literal keyword substring match with a flat 0.5/0.8 score), optional encoding-context boost, Jaccard-word-overlap dedup at 0.9 threshold, sort-and-slice top N. No cross-layer fusion algorithm (no RRF, no learned re-ranker) — just per-layer scores concatenated and sorted.
- **Consolidation** (`coordinator.ts::consolidate`, lines 403-444): `episodic.getRecent(100)` → filter `emotionalWeight > 0.5` → per-episode LLM call (`"Extract the key factual insight... one concise sentence"`, maxTokens 100) → `semantic.storeFact(summary, 'consolidation', 0.7)`. No dedup floor, no batching, no idempotency tracking, no queue, no audit trail.
- **FSRS** (`layers/semantic.ts`, real + wired): `storeFact` inits FSRS state via `initFromDecayClass('normal_decay')`; `recordRecall` reads current `fsrs_difficulty/stability/next_review`, branches `grade>=3 ? updateAfterRecall : updateAfterForgot`, writes back. This is a genuinely live spaced-repetition loop.
- **SQLite adapter** (`packages/adapters/sqlite/src/index.ts`): translates Postgres-style `$N` params + `<=>` pgvector operator into a registered `zb_cosine_dist` SQLite UDF doing full-scan cosine distance over JSON-array-string embeddings — no ANN index, explicitly documented as fine for dev/single-user, not production scale.

## Relevance to core-memory-kit

**Task 95 (dream re-curation, design.md §21) — reject-as-architecture, weak-borrow-as-formula.** Our design's three-stage pass (deterministic floor → ONE batched LLM call with `action ∈ {add,update,supersede,none}` → AUTO/QUEUE op-class split with screening + audit) has no analog here. zenbrain's "sleep consolidation" module is architecturally the opposite of what we need as a reference: it's a stateless scoring formula never wired to persistence, with no resumability (ADR-0020), no idempotency (our design's whole point is exactly the gap zenbrain's `consolidate()` falls into — duplicate-promotion on every re-run), no screening/provenance (D-352 F2), no op classes. The one thing worth carrying forward as a **secondary reference, not a copy**: `selectForReplay`'s priority formula (`accessCount*0.3 + emotionalWeight*0.3 + recency*0.2 + instability*0.2`, all normalized/decayed) is a clean, documented example of a multi-signal ranking formula for "what to consolidate first" — useful only as an idea-source if Task 95 ever needs a secondary prioritization signal inside the recall top-K candidate pool; it is not benchmarked or measured against anything (no citation of a lift number, just asserted), so it would need our own validation, not adoption on faith.

**ADR-0023 (graph edges: activate/defer/reject) — reinforces the REJECT lines, adds a 10th "graph memory flagship ships no graph" data point.** This is not in the D-361 9-system sweep (graphiti/mem0/cognee/MemOS/MemoryOS/iwe/sift-kg/pulse8/Memora) — zenbrain is a fresh instance of the exact pattern the ADR already names: "the field's own 'graph memory' flagships under-deliver in shipped code." zenbrain's `knowledge_entities`/`cross_context_links` schema is weaker than even our REJECTED "hand-authored graphs (iwe model)" alternative — iwe at least has a real editor UI; zenbrain has an unpopulated table and one untested-in-production `createLink()` method. It does **not** resemble our ADOPTED activation slice either (parse `related:`/`[[slug]]` at reindex into an `edges(src,dst,type)` table, recursive CTE for supersession chains, `mk_links`) — there is no reindex-from-source-of-truth rebuild anywhere in zenbrain; the entity/link tables are just empty CRUD surface. `personalized-pagerank.ts` is genuinely well-built PPR math and could be a reference implementation *if and only if* our DEFERRED LLM-edge-derivation trigger ever fires (multi-hop scoring beyond simple recursive-CTE chain-walking) — but that's speculative and pushed well past our current activation-slice scope.

**Tasks 233/161/203 (context-compaction family) — weak relevance.** No PreCompact-analog, no rolling-window/daily-distill-analog in this repo; `consolidate()` is the closest concept and it's a flat re-scan, not incremental/windowed.

## Borrow candidates

- `selectForReplay`'s weighted multi-signal priority formula (access/emotion/recency/instability) — as an idea to validate, not code to import; unmeasured, no benchmark backing.
- Nothing else rises to "borrow" — the wired product surface (FSRS loop, encoding-context boost) is standard and we already have equivalents (Task 66 validity engine, our own recall ranking); the unwired 13+ modules are architecturally interesting math but not proven-in-production references (they're proven-in-unit-test only).

## Reject candidates (with reasons)

- **The "graph memory" claim wholesale** — `knowledge_entities` unpopulated, `cross_context_links` write path untested-in-product, no traversal query anywhere. Reinforces ADR-0023's existing REJECT line on hand-authored/unpopulated graph schemas; adds zenbrain as a named instance if the ADR's evidence list is ever extended.
- **`MemoryCoordinator.consolidate()` as an architecture reference for Task 95** — unbatched per-item LLM calls (the exact anti-pattern our design.md §21.2 cites mem0 for abandoning), zero idempotency (duplicate-fact bug on repeat calls), zero resumability, zero screening/audit. A cautionary example, not a template.
- **`sleep-consolidation.ts` as a consolidation ENGINE** — it's a pure scoring/replay-simulation library with no I/O; adopting it would still require building the entire persistence/resumability/screening/queue layer our design already specifies, so it saves no engineering effort over designing from scratch.
- **The README comparison table's checkmark framing** — treats unwired algorithm-library exports as equivalent to Mem0/Letta/Zep's actual shipped runtime behavior; not a reliable signal for evaluating "does X system do Y in production."
