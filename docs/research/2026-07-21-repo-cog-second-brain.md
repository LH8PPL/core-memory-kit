## What the evidence actually shows

The "no database" claim is **literally true and the whole architecture**, verified at code level:

- **No graph store.** `grep -ri graph` across the whole tree returns 6 files, and every hit is one of: (a) Mermaid `graph LR`/`graph TD` flowchart syntax in README diagrams (not a data structure), (b) a mention of GitHub's `resolveReviewThread` **GraphQL** mutation in the PR-etiquette rules (CLAUDE.md/CHANGELOG), (c) one line in `onboarding/SKILL.md` noting markdown files "benefit from Obsidian's... graph view" (an Obsidian app feature, not something COG builds), and (d) the roadmap's own unchecked "knowledge graph visualization" item. There is no `edges`/`nodes` table, no graph library import, nothing resembling ADR-0023's `edges(src,dst,type)` shape. Unlike the three "graph memory" repos the 2026-07-19 sweep (ADR-0023's basis) found shipping no graph behind a graph claim, **COG does not actually claim to be a graph-memory system in its core pitch** — the graph word appears only in an honestly-unchecked roadmap box. This is a fifth "no graph despite the word appearing" data point, but a cleaner one: no false claim, just an absent, honestly-labeled aspiration.
- **No retrieval index of any kind.** There is no FTS, no embeddings, no BM25, nothing. "Retrieval" is entirely the phrase `05-knowledge/**` and `04-projects/<project>/` combined with a written instruction ("Brain-First Knowledge Protocol," `CLAUDE.md` lines 62-69: "Before answering any question about people, projects, strategy, decisions, or historical context: 1. Read relevant notes from `05-knowledge/` first... 2. If project-specific, also read related files... 3. Only then synthesize an answer"). Recall = the underlying coding-agent's own `Read`/`Grep`/`Glob` tool use, steered by prose instructions in `CLAUDE.md` and each `SKILL.md`. No bounded snapshot injection, no hook-enforced session-start load — it depends entirely on the live agent choosing to follow the written protocol each turn.
- **Consolidation is a prompt-engineered loop, not a scheduled job.** `knowledge-consolidation/SKILL.md` (877 lines) is a user- or cron-invoked skill (there is no actual cron wiring — "periodic... weekly, monthly, quarterly" is advisory prose) that instructs the agent to scan braindump/brief directories, extract themes via 5 named "pattern detection methods," and write `05-knowledge/consolidated/*-framework.md` / `patterns/*.md` / `timeline/*.md` files from Markdown templates baked into the skill file itself. It self-describes (lines 778-797) as "a loop-until-dry extraction with a completeness critic," terminating at "2 consecutive passes surface nothing new," with a mechanical verifier: "every framework principle links at least one source document. A principle with no `[[source]]` is dropped, not published" and dedup via "check `05-knowledge/consolidated/` before creating a framework." **This mechanical-sounding verifier is itself just another LLM instruction** — there is no hash/canonical-ID dedup floor anywhere in the repo (confirmed: no code exists to run one). Every "deterministic" check in COG is deterministic only insofar as the acting LLM actually executes the named tool call (`ls`, `curl`, `gh`, `test -e`) rather than reasoning about it internally.
- **`memory-hygiene/SKILL.md`** (v3.7.0, new 2026-07-10) is the one skill that does something genuinely different from consolidation: a monthly trust sweep that re-verifies **environment-dependent** claims (paths, URLs, IDs) against the *live* environment via cheap deterministic tool calls (`ls`/`test -e`, `curl -sI`, `gh repo view`), stamps `last_verified`/`confidence` into frontmatter, and *proposes* (never auto-applies) archiving drifted entries. It's explicitly adapted from a cited paper: "From Model Scaling to System Scaling: Scaling the Harness in Agentic AI," Gu, UC Berkeley, arXiv:2605.26112 — **verified via WebFetch**: title and single author (Shangding Gu) confirmed, abstract confirms the paper studies "system scaling" / "scaling the harness" and names "trustworthy memory" as one of three core bottlenecks. The specific framing COG attributes to it — "trust is a runtime decision, not a property of the stored item" — is COG's own paraphrase/adaptation per its own CHANGELOG wording ("adapted from"); I could not confirm that exact framing appears verbatim in the paper (abstract-only fetch, full text not read). Treat the citation's existence as verified, the specific quoted framing as COG's gloss, not a direct quote.
- **`cog-update.sh`** (449 lines, the only substantial code in the repo) is a real, working bash script: adds a `cog-upstream` git remote, diffs a hardcoded `FRAMEWORK_FILES[]` array (every skill/agent/role/doc file) against upstream per-file via `git show remote/branch:file`, and offers interactive/`--dry-run`/`--force`/`--check` update modes with automatic timestamped backups (`file.backup-YYYYMMDD-HHMMSS`) before any overwrite. It hard-codes the "framework vs. content" split (content dirs `00-inbox/`, `01-daily/`...`05-knowledge/` are never in the update array) rather than deriving it from any manifest/gitignore convention.

## Mechanism detail (the HOW)

- **Storage:** flat markdown files under a fixed 7-directory PARA-like vault (`00-inbox` through `06-templates`), each file's frontmatter carrying `type`, `status`, `confidence`, `source_documents`, etc. as free-form YAML the LLM is instructed to populate/read — no schema is enforced by code (verified: no frontmatter parser exists in the repo).
- **Retrieval:** none, mechanically. `CLAUDE.md`'s "Brain-First Knowledge Protocol" (§ lines 62-69) is the entire retrieval "system" — a standing instruction telling the agent which directories to `Read` before answering, backed by nothing that structurally enforces compliance.
- **Consolidation trigger:** user phrase ("consolidate my knowledge") or advisory cadence in the skill's own prose; no scheduler, no hook, no cron file anywhere in the repo.
- **Consolidation output:** Markdown templates for `consolidated/*-framework.md` (Core Principles / Applications / Boundaries / Evolution / Related Frameworks sections, each requiring `[[source]]` links), `patterns/pattern-*.md`, `timeline/*-evolution-*.md`, and one master `consolidation-YYYY-MM-DD.md` report; source docs get their frontmatter `status` flipped `captured` → `consolidated` with a `consolidated_in` backlink; superseded content is moved (never deleted) to `00-inbox/archive/*-archived-YYYY-MM-DD.md`.
- **People CRM (`05-knowledge/people/`):** two-layer profile (Compiled Truth top section + append-only reverse-chronological Timeline), tiered auto-escalation by mention count (Tier 3 stub at 1 mention → Tier 2 at 3+ → Tier 1 full at 8+ or a direct meeting), every claim carrying `[Source: [[path]] | date | confidence: high|medium|low]`. Explicitly credited to `garrytan/gbrain`'s "Compiled Truth + Timeline pattern."
- **Loop vocabulary (`loop-engineering/SKILL.md`):** a shared reference other skills link to instead of restating rules — names 5 termination-condition types (deterministic verifier / hard iteration cap / budget guard / no-progress detection / human escalation) and 7 named loop patterns (ReAct, Reflexion, plan-execute-verify, evaluator-optimizer, orchestrator-workers, loop-until-dry, human-in-the-loop), plus explicit context-rot countermeasures ("externalize state to the vault... never paste one worker's raw output into the next worker's prompt" — named failure mode "narrativisation": the worker treats the prior framing as pre-digested truth instead of independently reading the source).
- **`memory-hygiene` verification budget:** "never spend more than ~1 minute per entry... unverifiable ≠ drifted" — cheap-first checks, propose-only for destructive actions, direct-apply only for stamps/verified corrections.
- **Skill-distillation meta-pattern (`docs/SKILL-DISTILLATION.md`):** separate from memory but worth recording for completeness — "explore once (expensive, strong model) → distill into a layered skill artifact (router/recipes/healing-tree/map-shards) → execute cheap (small model)," with a PASS/FAIL-REAL/FAIL-STALE/BLOCKED verdict taxonomy that explicitly separates "the world changed" (routes to skill maintenance) from "the world is wrong" (routes to a real finding). Field-validated cost claim in the doc: "$4.63/150-turn exploration once → $0.11 per execution," unverified by me (no external benchmark data in-repo to check against, no way to verify a cost claim from static code).

## Relevance to core-memory-kit

**graph_relevance: none.** No graph store exists anywhere in the code; every "graph" hit is a Mermaid diagram, a GraphQL API mention, an Obsidian feature note, or an honestly-unchecked roadmap box. Nothing here bears on ADR-0023's activate/defer/reject decision — it neither confirms nor extends the sweep's "flagships ship no graph" pattern in a new way (COG never claimed graph memory as a shipped feature, so it isn't the same failure class as mem0/MemOS/Letta).

**task95_relevance: strong.** `knowledge-consolidation/SKILL.md` + `memory-hygiene/SKILL.md` are a real-world, independently-arrived-at analog to design.md §21's dream re-curation engine, built entirely as prompt engineering rather than code — a useful contrast case for what our code-enforced version buys over a pure-prompt version:
- Their "loop-until-dry... 2 consecutive passes surface nothing new" mirrors the shape of a termination condition our design should keep explicit for any multi-pass extraction we ever add (currently §21.2 is a single batched call, not a multi-pass loop, so this is a future-relevance note more than an immediate one).
- Their "propose archive, don't delete... only archive after user confirms" is the same posture as §21.6's "No DELETE op exists... archive-not-delete" and §21.2's QUEUE-class adopt-or-discard diff — independent convergence on the same anti-scope choice.
- Their dedup ("check `05-knowledge/consolidated/` first") is **LLM-instructed, not a deterministic floor** — this is the sharpest contrast with §21.2's stage 1 ("Deterministic floor (zero-LLM): canonical-ID / hash dedup... nothing the floor can decide reaches the LLM"). COG has no equivalent floor anywhere; every "mechanical" check in the repo is a tool call the LLM is trusted to actually make. This is a concrete illustration of the risk our zero-LLM floor is designed to close.
- `memory-hygiene`'s live-environment re-verification (stamping `last_verified`/`confidence`, distinguishing "environment-dependent" claims from "preference/judgment" claims, budget-capped at ~1 min/entry) is a genuinely different mechanism from anything in our §21 or ADR-0017/0020: **we don't currently re-verify that a fact's referenced paths/URLs/IDs still exist in the live environment** — our consolidation is about internal-corpus dedup/contradiction, not drift-against-the-world. Worth a look as a possible narrow addition (see Borrow candidates).
- "Every condensed/merged claim keeps source pointers" (our Task-213 pattern, §21.3) parallels COG's blanket citation rule (`[Source: [[path]] | date | confidence]`) and its "verbatim quote or drop the claim" discipline (`CLAUDE.md` "Citation Verbatim & Verifier Pass") — COG's verbatim-quote-or-drop rule is stricter than what I can confirm we currently enforce for provenance pointers; worth checking against Task 213 specifically (not verified in this pass — out of scope for this note).

**Task 233/161/203 (context-compaction) relevance: weak.** COG has no rolling-window/daily-distill/archive pipeline at all — no scheduled compaction exists in the repo (no cron, no `PreCompact` equivalent). The one adjacent idea is "Fresh-Context Isolation" (CLAUDE.md lines 56-58): never paste a prior worker's raw output into the next worker's prompt, because it induces "narrativisation." That's a fan-out/sub-agent hygiene rule, not a compaction mechanism, but the named failure mode is worth remembering if our Task-95 batched-recall call (§21.2 stage 2) ever fans out across multiple sub-calls.

## Borrow candidates

1. **Live-environment drift re-verification as a distinct pass from semantic re-curation** (from `memory-hygiene`). Our §21 dream engine dedupes/merges/resolves contradictions *within* the corpus; nothing re-checks whether a fact's referenced external state (a file path in `context/`, a documented CLI flag, a linked doc section) still exists. A cheap, budget-capped, propose-only sweep — "does the referenced anchor still resolve?" — is a narrow, independently-motivated addition distinct from what §21 already scopes. Not a full lift (COG's version is monthly/manual and prompt-driven; ours would need to be code-enforced per the kit's own deterministic-floor discipline), but the *concept* (separate "is this still true" from "is this a duplicate/contradiction") is clean and currently absent from our design.
2. **"Verbatim quote or drop the claim"** as an explicit citation discipline, if not already covered by Task 213 — worth a follow-up check (not done in this pass) against whether our provenance pointers currently allow a citation with no exact-text backing.
3. **Named failure mode "narrativisation"** — worth citing by name if we ever design a fan-out step inside the §21.2 batched-recall call or any future multi-agent consolidation stage; concrete instance of the general "don't pre-digest for a sub-agent" principle our own `CLAUDE.md` already states differently (fresh-context isolation is not currently named in our docs; a citable term costs nothing to add if the concept resurfaces).

## Reject candidates

1. **The entire prompt-only architecture (no index, no deterministic dedup floor)** — directly contrary to our kit's founding bet (ADR-0002 markdown-as-truth *plus* a rebuildable FTS5/vector index, and §21.2's explicit zero-LLM floor). COG's "search" is "hope the agent follows written instructions" — the D-366 falsify-first discipline and our whole verification-first posture argue against trusting an LLM's self-reported tool-use compliance as a retrieval mechanism.
2. **`cog-update.sh`'s hardcoded framework-file-list update mechanism** — solves a real problem (update framework files, never touch user content) but via a maintained bash array of every shipped path, not a manifest-driven approach; doesn't map to our npm-package distribution model (`npm update` + our own template/registry discipline already solves this differently and, per `validate-pack-completeness.mjs`, more structurally).
3. **Skill-distillation's cost claim ($4.63 → $0.11)** — interesting pattern for testing/execution skills generally, but it's about *executing recurring environment interactions cheaply*, not about memory storage or recall; out of scope for the tasks this pass was asked to compare against. Noting for completeness, not proposing adoption.
