# Mnemo (Qwen Cloud hackathon, Track 1: MemoryAgent) — trust-gated recall + a self-caught evidence-fabrication bug

**Date:** 2026-07-21
**Source:** `<local-wiki>/raw/Mnemo for the Global AI Hackathon with Qwen Cloud — Track 1 MemoryAgent.md` (clipped from `https://dev.to/destinationbubul/mnemo-for-the-global-ai-hackathon-with-qwen-cloud-track-1-memoryagent-5fp2`, published 2026-07-20, author byline "destinationbubul" — a dev.to handle, not further identified)

## What it claims

The author built "Mnemo," a memory agent for a Qwen Cloud hackathon (Track 1: MemoryAgent — "efficient memory storage and retrieval, timely forgetting of outdated information, and recall of critical memories within limited context windows"). The stated differentiator: not "remember more" but a **runtime assessor** that grades pack trust on every recall and gates the agent behind that grade, refusing to act on memories it can't verify or trust. Headline results: against NaiveRAG (47% wrong-fact rate) and RecentOnly baselines across six scenarios, Mnemo reports 0% wrong-fact rate, 100% forgetting precision/recall. The assessor's two-sided error metric (unsafe-acceptance vs over-abstention) took four iterations to become meaningful, ending at 0%/0% on 26 "decisive" cases. The back half of the post is a self-audit: the author discovered a committed "live" transcript was actually silently generated by the offline fake client (`FakeQwen`), had already cited the resulting (fictitious) defect as fact in three documents, and used the discovery to argue that evidence-generating tooling needs the same trust discipline as the memory system itself.

## Mechanism detail (the HOW)

**Three-mode model seam** — every model call routes through one `get_client()`:
- `MNEMO_OFFLINE=1` → `FakeQwen`, a deterministic stand-in (hashed bag-of-words embeddings in a fixed-dim unit vector so overlapping text scores higher cosine; chat pattern-matches structured prompts and returns valid JSON; no network/key/spend).
- Live → real Qwen Cloud (`qwen3-max` for reconciliation/planning, `qwen-plus` for extraction, `text-embedding-v4`@1024d for retrieval, OpenAI-compatible Singapore endpoint; Alibaba OSS for cold tombstone archive; FastAPI on ECS).
- `MNEMO_RECORD=<path>` / `MNEMO_REPLAY=<path>` → cassette record/replay (tee real request/response once, replay deterministically in CI forever; a cassette **miss raises rather than silently degrading** — contrast this with the client-seam bug below).

Cost habits: embeddings persisted at write time, never recomputed on recall; extraction on a cheaper model than reasoning (`qwen-flash` option even cheaper); stable few-shot system-prompt prefixes to benefit from provider-side prompt caching; `max_tokens` capped on chat calls. ~90% of dev work (full test suite, 6-scenario eval harness, CI) runs offline at zero cost; credits spent only on prompt tuning, one live smoke test, demo recording, deployed backend.

**Write path — "record before validate":**
1. Raw turn recorded first (unconditionally).
2. `qwen-plus` extracts candidate memories from it.
3. `qwen3-max` reconciles each candidate against **active memories with overlapping entities** → classifies each as `duplicate` / `contradicts` / `novel`.
4. On `contradicts` (a correction arrives, e.g. a lead-time fact changes from 21 days to 35): the stale memory is **tombstoned immediately** — not left to decay. Tombstones are never hard-deleted; they flush to OSS (cold archive) and **every forget carries a reason**, so recall can explain an absence ("the 21-day fact is gone because a 35-day correction superseded it," not "retrieval missed it").
5. A separate explicit `[FORGET]` command path exists; the article frames the hard part as forgetting *precisely* — not clipping neighboring memories.

**Decay (soft forgetting):** salience = f(importance, reinforcement, corroboration, recency), 14-day half-life, 0.08 threshold — a never-reinforced low-importance memory needs ~a month untouched to cross the line. Notable methodological point: the author's first 5 of 6 scenarios spanned ≤9 simulated days, so decay silently never fired and looked "inert" until a 6th scenario spacing sessions a week apart was added specifically to exercise the mechanism.

**Recall — framed explicitly as a packing problem, not a search problem:** under a hard token ceiling, a planner maximizes decision-relevant recall using **per-type quotas** (semantic 45% / procedural 25% / episodic 30%) so bulk episodic content can't crowd out a critical preference. Pinned memories bypass the quotas but still count against the global token budget. Every inclusion/exclusion decision is logged to a trace (turns "why isn't X in context" into an inspectable decision, not a mystery).

**The assessor — the article's core novelty claim:** on every recall, the packed set is split by verifiability.
- **Verifiable claims** (checkable against a live system of record right now — e.g. a lead time, an inventory level) are **re-grounded via read-only tools before the planner sees them**; if the live source disagrees with the stored memory, the stale value is caught and replaced in-flight. The article's explicit claim: this catches staleness even when *no conversational correction ever arrived* — something a purely conversational/dialogue-driven memory system structurally can't do.
- **Unverifiable claims** (preferences, routines) can't be re-grounded, so they're scored from provenance signals: corroboration, recency, single-source penalty, confirmation-age decay.
- **Pack trust = the MINIMUM over load-bearing memories** (verifiable, or important enough to affect the decision) — deliberately not an average, so low-value episodic color can't dilute a genuinely stale/weak load-bearing fact. Below a threshold (0.55, stated as "statically chosen, not calibrated on held-out data"), the agent **escalates to a human** rather than act on a low-trust pack.

**The two-sided metric, and why it took 4 tries:** unsafe-acceptance alone is gameable (refuse everything → 0% unsafe-acceptance, useless system) — the counter-metric is over-abstention (refusing a sound pack). Reported iteration history:

| gen | n | unsafe-acceptance | over-abstention | what changed |
|---|---|---|---|---|
| 1 | 4 | 0% | 0% | too small to mean anything |
| 2 | 33 | 21% | 11% | expanded set — but 15/33 were two "ladders" (same memory at varying confirmation ages), i.e. one annotator's staleness cutoff double-counted as independent evidence |
| 3 | 26 | 30% | 0% | dropped ladders to keep only decisive-agreement cases — rate got WORSE because errors concentrated |
| 4 | 26 | 0% | 0% | root cause: the scoring rule counted "acted on a should_act=False case" without checking whether the action used the STALE value or the assessor's own CORRECTED value — 3 of the gen-3 "errors" were cases where re-grounding correctly caught and fixed a stale fact, and the metric was punishing the audit layer for doing its job |

**The evidence-fabrication incident (the article's second half):** a committed transcript file (`runs/live_tuning_20260708_202557.jsonl`) showed `qwen-plus` returning `{"candidates": []}` for every extraction call — logged as a real, in-progress live-path defect in the README, an earlier draft of this post, and the live runbook. Three tells eventually surfaced it as fake: (1) a planning-case output was byte-identical to the offline fake's hardcoded plan string; (2) a rephrased-duplicate reconcile case returned `novel` — exactly `FakeQwen`'s behavior (it only flags `duplicate` on exact string equality, something a real model wouldn't do); (3) `{"candidates": []}` is precisely what the fake extractor returns for untagged prose (it only echoes `[SEM]`/`[PROC]`/`[EPI]`-prefixed lines, and the tuning inputs were plain sentences). Root cause: the transcript's `"model"` field was populated from **config** (what the code intended to call) rather than from the actual API response (what answered) — so a silently-degraded-to-fake call still logged a plausible model name. Remediation stated: every transcript record now carries a provenance block (client class, offline flag, replay/record paths, the model string the API itself echoes back, a run id); the tuning script now takes an explicit `--live` flag and **hard-fails** rather than silently falling through to the fake.

Article ships two alternate epilogues (Variant A: the live smoke came back green, the defect never existed; Variant B: it reproduced live, template with `<one-line root cause>` unfilled) — meaning **the version I read is a draft/template with the actual outcome not yet substituted in**. This is worth flagging plainly: the piece's own climax is unresolved in the copy I have.

## Relevance to core-memory-kit

**Task 95 (dream re-curation engine, design.md §21) — strong structural overlap, worth reading against the settled contract, not adopting wholesale:**

- **Contradiction handling.** Mnemo's "tombstone the moment a correction lands, never wait for decay, keep the old value archived with a reason" is *the same shape* as §21.2's AUTO-class "event-time supersede-marks (validity-window close via the 66 engine — both sides retained, archive-not-delete)" and §21.6's "no DELETE op … the kit keeps separate facts + closed windows." Independent convergence on archive-don't-erase-and-explain-why is a mild corroboration signal, not new evidence — the kit's decision was already settled from Sleep-paper/Memora sourcing before this article.
- **LLM-proposes-code-decides.** §21.2 is explicit: "which-wins on a contradiction is decided by EVENT-TIME, never by the LLM." Mnemo's reconcile step instead has `qwen3-max` itself classify `duplicate`/`contradicts`/`novel` and (implicitly) decide the correction — i.e. Mnemo lets the LLM decide which-wins, which is exactly the shape §21.2 rejects. This is a **useful negative data point**: it's a real, recent, differently-sourced design that took the path the kit explicitly closed off, and the kit's own rationale (D-230 bridge discipline — hallucinated/LLM-decided outcomes need code-side validation) is not contradicted by anything in the article; if anything the article's own second-half incident (the LLM ecosystem's evidence being silently unreliable) is an argument FOR keeping high-stakes decisions out of LLM hands.
- **The assessor / pack-trust-as-minimum idea has no equivalent in §21 at all.** The kit's Task 95 contract is about consolidation (merge/dedupe/supersede/insight-surface), not about a runtime gate that scores retrieved-pack trust and can make the agent abstain/escalate. This is the single most transferable NEW idea in the article relative to what's already designed — see borrow candidates.
- **Re-grounding unreachable-by-conversation staleness.** The article's claim that verifiable facts get checked against a live source of truth *even with no conversational correction* has no kit analog — the kit's whole staleness-detection story (Task-66 validity engine, D-352 contradiction handling) is conversation/session-triggered. This is a real gap the article names cleanly, though the kit has no external "systems of record" to check against in the general case (it's a per-project text memory store, not an agent with live tool access to inventory/CRM-type systems) — so the mechanism doesn't port directly, but the PRINCIPLE (some memories are checkable, treat them differently from ones that aren't) might.
- **Two-sided metric discipline.** §21.5's gates ("HC-12 regression gate," "Task-212 stats probe... before/after," "never-silent-skip") are about run-safety, not about a calibrated two-sided error metric for a trust/gating decision. The article's biggest engineering lesson — unsafe-acceptance alone is gameable, you need the over-abstention counter-metric, and you need to write the metric's semantics down BEFORE computing it (their gen-3→gen-4 fix was a scoring-definition bug, not a model bug) — is a process lesson more than an architecture one, but it's directly applicable if/when the kit ever builds an automated trust-gate metric (e.g., for HC-12 or the recuration queue's own approve/reject accuracy).
- **Graph edges (Task 232 / ADR-0023):** no relevance — the article has no entity-relationship/graph structure at all; recall is describes as budget-packing over typed (semantic/procedural/episodic) memory, not graph traversal.
- **Context-compaction (Tasks 233/161/203):** weak/tangential — Mnemo's token-budget packing (per-type quotas under a hard ceiling) is adjacent to "what goes into the context window," but it's a recall-time selection problem over an already-distinct memory store, not a compaction/rewrite-under-pressure problem the way 233/161/203 are framed. Did not deep-dive those task entries since the article gives no compaction-specific mechanism to compare against.

## Borrow candidates

- **The two-sided trust/gate metric pattern (unsafe-acceptance + over-abstention, reported together, never either alone)** — directly applicable wherever the kit scores an automated accept/reject decision (e.g., a future HC-12 regression-gate calibration, or QUEUE-class adopt/discard accuracy if that ever gets measured). The "write the metric's semantics down before computing it" lesson is cheap and general.
- **Provenance-stamped evidence artifacts** (the remediation: every transcript/log record carries what-actually-answered, not what-was-configured-to-answer) — directly relevant to the kit's own audit-log discipline (`design §17` exit-doors, "External calls" door) and to CLAUDE.md's own "fabricated success = no real tool ran" precedent (the kiro-cli capability-config incident cited in this repo's CLAUDE.md is the *same failure class* as Mnemo's `FakeQwen`-silently-answering-as-if-live bug: a graceful-degradation seam producing a plausible fake result with no marker distinguishing it from the real one). Worth a cross-reference in the kit's own docs as independent-corroboration of that CLAUDE.md rule, not as new mechanism to build.
- **"Pack trust = minimum over load-bearing items, not an average"** — a small, concrete aggregation-function idea, cheap to steal if the kit ever needs to roll up multiple per-fact confidence scores into one recall-time decision (avoids a few weak-but-decorative facts diluting one genuinely stale load-bearing one).
- **Per-type recall quotas under a hard token budget** (semantic/procedural/episodic percentages) as a concrete alternative/comparison point to the kit's existing recall-budget design — worth a design.md cross-check (not read as part of this task) rather than a borrow-blind adoption, since the kit's tiers (P/L/U, committed/gitignored) don't map 1:1 onto Mnemo's semantic/procedural/episodic typing.

## Reject candidates

- **LLM-decided contradiction resolution** (Mnemo's `qwen3-max` classifies duplicate/contradicts/novel and appears to drive the tombstone decision) — the kit's §21.2 event-time-decided-by-code rule already rejects this shape explicitly (D-352); nothing in the article weakens that call, and the article's own second-half incident is an argument for keeping it rejected.
- **Statically-chosen, uncalibrated threshold (0.55) presented as a load-bearing gate value** — the article is honest that this isn't calibrated on held-out data; not something to import as a magic number, only as "yes you need SOME threshold and it should be stated as calibrated-or-not."
- **The headline 0% wrong-fact / 0%/0% assessor numbers as evidence of anything generalizable** — self-reported, no repo, n=26, single annotator, and the author's own stated caveat that most of the "clean" margin comes from excluding ambiguous cases rather than model quality. Useful as an illustration of the pattern, not as a benchmark result to cite.
