# Session-buffer compaction under latency + growth — the Task 161 research

> **The question (Task 161 / D-171).** The kit's SessionStart `now.md → today` roll (`compress-session`, design §8.5) spawns `claude --print --model claude-haiku-4-5` to compress the **whole** session buffer under a **50s inner timeout** (under the 60s SessionEnd hook ceiling). As `now.md` grows, the Haiku call gets slower; when it crosses 50s it errors `haiku_timeout`, the buffer is left UNTOUCHED (fail-safe), and the roll retries only at the next SessionStart — by which point the buffer is **bigger**, so the next call is **slower**, so it times out **again**. A compounding degradation spiral that re-instances the Task-105/D-75 unbounded-`now.md` class, gated behind a wall-clock timeout instead of a missing hook.
>
> **RESEARCH-FIRST** (the user: *"real deep research, not just skim over"*; *"clone all of them and research again for the mechanic — don't use past research, it could change"*). **Every mechanic below was re-derived from a FRESH clone on 2026-06-18, not lifted from an earlier note.** That re-derivation already paid off: the prior claude-mem survey claimed the ancestor shares our whole-buffer spiral; the **current** code shows it has since moved to per-turn last-message summarization (no buffer) — a stale-note error the re-clone caught.

## The kit's failure mechanism (grounded, not hypothetical)

- **Code:** [`compress-session.mjs`](../../packages/cli/src/compress-session.mjs) reads the **entire** `now.md` into `buffer` and passes it straight to `backend.compress()` with `timeoutMs: 50_000`. `input_bytes = Buffer.byteLength(buffer)` is **measured but never gated** — there is **no input size cap, no pre-trim, no chunking, no shrink-and-retry**. The same `HaikuViaAnthropicApi.compress` path (same 50s budget) backs `daily-distill` and `weekly-curate` too (design §8.5 table; cut-gate-16 §7 F-4).
- **Data (the kit's OWN dogfood `context/sessions/*.compress.log`, NOT synthetic):** real compressions span **21s–50s+**; **5 historical `haiku_timeout` failures**; `now.md` inputs of **166 KB / 334 KB / 405 KB / 470 KB** all timed out at 50s. At cut-gate-16 a fresh ~10 KB buffer took **71s then 89s** on two isolated runs of the exact compressor invocation — correct 4-section output, just SLOW. So **the compression LOGIC is fine; `claude --print` latency vs the fixed 50s budget on an UNBOUNDED input is the failure.**

**The load-bearing provenance fact for Task 161:** the origin note ALREADY anticipated this failure and wrote a guard that was never wired in —
> *"If compression latency on Haiku 4.5 routinely **> 3 s p95** → switch to async-only (Stop+SessionEnd only, drop rolling window)."* — [option-b note, line 437](2026-05-21-claude-ai-deep-research-option-b.md)

A **3-second** p95 latency budget was specified at design time. We are now measuring **21–89 s**. The latency-kill clause existed in the research and was not implemented — so Task 161 is not discovering a new risk, it is **closing a guard the original design called for and the build dropped.** And note: **LightMem — the system that VALIDATED the rolling-window choice in the first place — bounds its compaction input** (`get_dialated_windows(N=5)` + `truncation=True`, freshly re-derived above). The kit adopted LightMem's *trigger* (every-N-turns rolling window) but **not** its *input bound* — which is precisely the gap this note's 18-system convergence says to close.

## The convergent lesson (17 of 19 bound their input; the 2 that DON'T are claude-remember — a kit precedent — and pre-fix claude-mem, the kit's ancestor before it moved to per-turn)

**Bound the LLM's INPUT before the call — never feed it an unbounded, monotonically-growing buffer.** This is not a niche choice; it is **universal** across every system that compresses at scale, by one of four families:

1. **Structural cap on the buffer** so it can't grow unbounded (MemoryOS `deque(maxlen)`; MemOS `MAX_BUFFER_SIZE`; Letta `message_buffer_limit`; memclaw `MAX_INGEST_WRITES_PER_SESSION=10`; mem0/claude-mem per-turn = no buffer). The input is *inherently* bounded.
2. **Deterministic input truncation/chunking BEFORE the model** (memsearch `MAX_PROMPT_CHARS=80k`; mempalace `CHUNK_SIZE`/`TruncationDetected`; graphiti `truncate_at_sentence`; cognee/honcho/memweave/MemOS truncate-to-limit; TencentDB emergency `truncate-in-place`). A cheap, no-LLM pass floors what the call ever sees.
3. **Partial / windowed compaction** — compress only the OLDEST slice, keep recent verbatim, or sample across history (Letta `PARTIAL_EVICT` 30/70; OpenHands `keep_first`; langmem/LightMem `get_dialated_windows`; squad inbox-deltas). Input is decoupled from total history length.
4. **Batch-size capping** — never extract more than N units per pass regardless of backlog (TencentDB `Max tool pairs per L1 batch = 20`, force-trigger at ≥4; the per-batch sibling of family 1). The per-call input is bounded even when the backlog isn't.

**The kit does NONE of these** — it feeds the *whole* `now.md` to Haiku under a wall-clock budget. **The only two systems that share this unbounded-input antipattern are exactly the two CC-memory PRECEDENTS the kit's mechanic was built on: `claude-remember` (still unbounded today, survives on a 120s timeout + daily-delta staging) and `claude-mem` (which has SINCE abandoned it for per-turn).** Every other mature system — including both direct competitors (memsearch, caura-memclaw) — bounds the input. **So the kit inherited the antipattern from its precedents and never added the bound the rest of the field considers table-stakes; claude-mem already walked away from it. The fix direction is settled by the field: BOUND THE INPUT. "Raise the timeout" (candidate (d)) is the claude-remember approach (120s) — it only delays the cliff, and is used as a *primary* fix by no one.**

## Five concrete mechanisms to steal (ranked by fit; each cites the freshly-verified sources)

1. **Deterministic INPUT CAP before the Haiku call (memsearch `MAX_PROMPT_CHARS=80k` + MemOS + graphiti + cognee/honcho/memweave) — the cheapest, highest-certainty fix.** Cap the bytes/chars handed to `backend.compress()`; if `now.md` exceeds the cap, deterministically trim (keep most-recent verbatim, drop/summarize the oldest overflow) BEFORE the call. Zero extra latency, no LLM, breaks the growth→latency coupling immediately. **This is the single most-converged-on mechanic (≥8 of 16).** Our `input_bytes` is already measured — gate on it.
2. **Partial-evict / windowed compaction (Letta `PARTIAL_EVICT` 30/70 + OpenHands `keep_first` + langmem dilated-window) — the primary structural fix.** Compress only the OLDEST portion of `now.md` down to a target, KEEP recent N turns verbatim; the Haiku input becomes the *evicted slice*, not the whole buffer. Snap the cutoff to a turn boundary (Letta snaps to an assistant message; `now.md` has turn delimiters). Decouples latency from session length.
3. **A `now.md` SIZE/TURN cap that triggers mid-session compaction (MemoryOS `is_full()` + MemOS `MAX_BUFFER_SIZE` + Letta proactive threshold) — drain while small.** Don't wait for SessionStart to roll a huge backlog; when `now.md` crosses a cap mid-session (a cheap check in the PostToolUse appender §16.27), compact the overflow. Makes the spiral *structurally impossible*, not merely survivable.
4. **Compress at SessionEND too, not only SessionStart-lazy (the user's hunch + squad's session-end Scribe ceremony + Letta's eviction-on-step).** Drain at the natural close boundary while the buffer is freshest/smallest. **Composition constraint (design §8.5 / D-42):** SessionEnd already runs `compressSession` CONCURRENTLY with `autoPersona` under the 60s ceiling — a SessionEnd now→today roll must be a *bounded* drain (mechanisms 1–3 keep it cheap), not a second full-buffer 50s call.
5. **`minimum_progress` floor + shrink-and-retry on `haiku_timeout` (OpenHands `minimum_progress=0.1` + `context_scaling=0.8`×5 + mempalace `TruncationDetected`) — defense-in-depth.** On timeout, RETRY with a deterministically-shrunk input (drop the oldest slice, re-call) so a too-large input degrades gracefully instead of spiraling; and don't pay a Haiku call that shrinks <~10%. Task 84 already noted shrink-and-retry — confirmed here from a second source, paired with the min-progress guard.

## What we would NOT take

- **Token-window framing as the TRIGGER.** Letta/OpenHands/langmem compact to fit a *context-window token budget* (the next inference would literally fail). The kit's constraint is a *wall-clock timeout on a batch summarizer* — `now.md` is never fed to a live agent, it's compressed offline — so adopt their **input-bounding mechanics** (cap/truncate/partial-evict/window), NOT their token-pressure trigger. Size the kit's cap in **bytes/turns against measured Haiku latency**, not tokens against a context window.
- **The code.** All Python / TS, tied to their loops, `View`/`deque`/agent-state. The kit is markdown-rolling-window over files. Take the IDEAS + the *constants as starting reference points* (80k chars, 30% evict, 800-char chunks).
- **A pure timeout raise (candidate (d)).** Used by NONE of the 16 — it moves the cliff, doesn't remove it.
- **Abolishing the buffer (mem0/claude-mem per-turn).** Tempting, but the kit's offline batch rollup is a deliberate cost choice (cheap Haiku batch vs per-turn LLM cost); the kit's *auto-extract* already IS the per-turn layer. `compress-session` is the batch layer — bound it, don't delete it.

## Net → the design direction for Task 161 (to settle in design §8.5/§8.2)

**The field is unanimous: BOUND THE COMPACTION INPUT; do not touch the timeout.** The strongest convergent shape for the kit:

- **Primary:** a deterministic **`now.md` byte/turn cap** (mechanism 1+3 — the ≥8-system convergence) so the buffer can't grow unbounded and the Haiku input is always small, **+ partial-evict** (mechanism 2) so even a large buffer only sends its OLDEST slice, keeping recent turns verbatim. Together → the spiral is *structurally impossible*.
- **Defense-in-depth:** **shrink-and-retry on `haiku_timeout`** + a **deterministic pre-trim / keep-last-N** (mechanism 5) so a slow call degrades gracefully.
- **Where it fires:** keep SessionStart-lazy, ADD a **bounded SessionEnd drain** (mechanism 4), composed against the D-42 concurrent-SessionEnd budget (the bounded input is what makes this safe under the 60s ceiling).
- **MUST cover the whole compression family:** the same input-bounding applies to `daily-distill` (timed out at cut-gate-16) and `weekly-curate` — all route through `HaikuViaAnthropicApi.compress` under the shared 50s budget (design §8.5). Their inputs are summaries-of-summaries, so a per-source cap + partial-evict generalizes cleanly.

Stays **within-paradigm** (v0.3.4 patch — fixes a real growing problem, ships no new differentiator) and reuses existing seams (`now.md` append §16.27, `CompressorBackend` §8.3, lazy/SessionEnd composition §8.2/§8.5).
