# OKF (Open Knowledge Format) — convergence + interop assessment

> **What it is.** A Google Cloud open format (v0.1, part of `GoogleCloudPlatform/knowledge-catalog`) for representing **knowledge — "the metadata, context, and curated insight that surrounds data and systems"** — designed to be *"authored by people, generated by agents, exchanged across organizations, and consumed by both."* Its scope is **knowledge EXCHANGE / data-catalog metadata**, NOT agent working memory. But its REPRESENTATION is nearly identical to the kit's — worth a real look (this is the opposite of an orthogonal source).

## The convergence is striking — OKF independently arrived at the kit's design

| OKF (Google, for cross-org knowledge exchange) | claude-memory-kit (for agent memory) |
| --- | --- |
| **Markdown + YAML frontmatter**, deliberately minimal | **Same** — markdown fact files + frontmatter (ADR-0002) |
| *"If you can `cat` a file you can read it; if you can `git clone` a repo you can ship it"* | **The kit's git-native thesis, almost verbatim** (committed `context/` travels with clone) |
| **Concept** = one markdown file = one unit of knowledge | **Granular fact file** = one fact per file |
| Frontmatter: `type` (required) · `title` · `description` · `tags` · `timestamp` · `resource` (URI) | Our frontmatter: `type` · `title` · `tags` · `created_at` · `source_file`/`source_line` · `source_sha1` · `trust` — **near 1:1 on the human fields** |
| **`index.md`** — directory listing, progressive disclosure via links | **`INDEX.md`** — our derived per-fact index |
| **`log.md`** — chronological history, ISO-8601 date headings, **newest-first** | **`DECISION-LOG.md`** — append-only, newest-block-at-top — **independently identical convention** |
| Links = standard markdown, **untyped directed edges, relationship conveyed by prose**; broken links permitted | Our **`[[name]]`** wiki-links — link liberally, a not-yet-existing target is fine — **same philosophy** |
| `okf_version: "0.1"` declared in root `index.md` | (we version the kit, not the bundle — minor) |
| **No fixed `type` taxonomy** — any string, consumers tolerate unknown types | We DO have a fixed taxonomy (`user`/`feedback`/`project`/`reference`) — a real divergence (we're stricter) |
| **Concept ID = file path** minus `.md` (e.g. `tables/users`); semantic versioning; **no content-addressing** | **Content-addressed IDs** (`<tier>-<base32(sha256(canonical))>`, ADR-0007) — the kit's deliberate divergence for cross-machine dedup determinism |

**Two real divergences, both deliberate on our side:** (1) **IDs** — OKF uses path-as-ID (human-readable, simple), we use content-addressing (dedup + cross-machine determinism + consolidation survival — ADR-0007's whole rationale). (2) **type taxonomy** — OKF is open/untyped, we're closed (4 types) because our taxonomy drives routing + trust + the persona gate. Everything else is convergent.

## Why this matters to US — INTEROP, not adoption

The takeaway is NOT "switch the kit to OKF" — our content-addressing + typed taxonomy + trust model are load-bearing and OKF is looser by design (it's an interchange format, not a memory engine). The takeaway is **interop**: OKF is a **Google-backed STANDARD for exactly the shape the kit already produces** (human+agent-authored, git-shippable, markdown+frontmatter knowledge). That's strategically relevant to two lanes where the kit's memory needs to be read by something OTHER than the kit:

1. **Team layer (Task 127, v0.5+ — git-native shared brain, D-119).** The kit's committed `context/` is already "knowledge a team shares via git." If a teammate uses a DIFFERENT tool, an OKF-compatible (or OKF-export) view of `context/` lets their tool read the kit's memory through a neutral standard instead of reverse-engineering our format. OKF is the natural interchange layer for the team story.
2. **Cross-agent (Task 50 / D-157, v0.4).** Same logic for a different agent: a neutral, agent-friendly knowledge format is a candidate lingua franca when the memory must cross tools, not just IDEs.

**Concrete, low-cost possibility (NOT a commitment — a design input):** a `cmk export --okf` that renders the committed tier as an OKF bundle (our typed facts → OKF concepts with `type` carried through; our `INDEX.md`/`DECISION-LOG.md` → OKF `index.md`/`log.md` — the mapping is nearly mechanical given the convergence). Cheap because the formats are already 90% aligned. Slot as a candidate for the Task-127 team-layer design (where interop actually pays off), NOT v0.4.

## What we would NOT do

- **Not replace our format with OKF.** We'd lose content-addressing (dedup/determinism, ADR-0007) and our typed-taxonomy-driven trust/routing/persona logic. OKF is looser by design — right for exchange, wrong as the kit's internal engine.
- **Not adopt path-as-ID.** Content-addressing is a deliberate, load-bearing divergence (ADR-0007).
- **Not chase it for v0.3/v0.4.** Interop pays off at the TEAM boundary (Task 127, v0.5+) and the cross-agent boundary (Task 50, v0.4); export is a design input for those, not a standalone task now.

## Net

**The strongest convergence of the review batch** — Google independently shipped nearly the kit's exact representation (markdown+frontmatter, git-shippable, concept-per-file, index.md + newest-first log.md, untyped prose-relationship links) for cross-org knowledge exchange. **Validation** that the kit's ADR-0002 + git-native + per-fact-file design is the right shape (a major cloud vendor converged on it). **The actionable angle is INTEROP**: OKF is a candidate export/interchange target for the **team layer (Task 127)** and **cross-agent (Task 50)** — recorded as a design input for those lanes, with a sketched `cmk export --okf` as the cheap mechanical path given the 90% format overlap. No new task now; flagged where it pays off.
