# AI Agent Memory Learns Across Sessions: Huawei's JiuwenMemory, and the China Data-Risk Question

**Date:** 2026-07-21
**Source:** `<local-wiki>/raw/AI Agent Memory Learns Across Sessions Huawei Framework Ships With China Data Risk.md` (TechTimes, published 2026-07-02, author Jerry Owens; original URL `techtimes.com/articles/319523/20260702/ai-agent-memory-learns-across-sessions-huawei-framework-ships-china-data-risk.htm`)

## What it claims

Huawei's openJiuwen open-source community released a self-evolving agent-memory framework on July 1, 2026 — called "AutoGenetic Memory" in the lede and "JiuwenMemory" everywhere else in the piece (the article never states outright that these are the same name for the same thing; treat as an unresolved naming ambiguity in the source, not settled here). The architecture bundles five components: a four-layer stratified store (L0 raw transcript → L1 session distillations → L2 episodic/semantic split → L3 user profile), an async "AutoDreaming" consolidation daemon modeled on sleep-stage memory research, a two-tier "MemoryTurbo" write path for latency, a continuously-updated "GraphMemory" knowledge graph, and a "Swarm Memory" pool shared across multi-agent teams. The article's second half pivots to a legal argument: because Huawei and its affiliated communities are Chinese entities, China's 2017 National Intelligence Law obligates them to cooperate with government intelligence requests "on demand," an obligation the article says survives self-hosting, open-source licensing, and any privacy policy — so any sensitive data JiuwenMemory's AutoDreaming daemon structures into profiles carries that exposure by construction, independent of whether a request has ever actually been made.

## Mechanism detail (the HOW, as far as it is knowable — article-only, unverified against source)

- **L0–L3 stack:** L0 = complete timestamped transcript (audit floor). L1 = per-session LLM distillations. L2 = split into **episodic** (dated events/decisions) and **semantic** (background/technical knowledge that "should not be overwritten by daily chatter") — a deliberate two-bucket split, not one pool. L3 = user profile (name, role, timezone, language, preferences), loaded first at session start.
- **AutoDreaming:** a lightweight filter pass flags what's worth processing → a heavier LLM pass extracts/categorizes into L1–L3 → a final stage does dedup + conflict resolution ("if the agent learned a user's preferred database technology in one conversation and the user mentioned switching in another, this stage resolves which version is current"). **No mechanism is given for HOW that resolution happens** — no code, no rule (event-time? recency? LLM judgment?), just the outcome description. This is the same black-box pattern as the ChatGPT "Dreaming" coverage read alongside this note (see the companion notes on `2026-07-21-chatgpt-*`) — a named pipeline stage with no disclosed decision procedure. A "busy backoff" protocol defers processing under load; batch sizes are capped for token predictability.
- **MemoryTurbo:** new conversation data writes immediately to a cache-layer vector store (searchable before extraction runs) while heavier extraction runs async; a semantic-clustering pre-pass groups related conversations by topic before the LLM extraction stage, to cut redundant processing. Claimed 80% perceived-latency reduction + 50%+ further token reduction — **developer-reported, explicitly flagged by the article as not independently benchmarked.**
- **GraphMemory:** entities (people/projects/systems/decisions) + relationships as connected nodes, continuously updated, merging/deduping entity records as new conversations arrive. The article compares it to Graphiti's temporal/validity-window model (closer) vs. Mem0's flat-vector base tier (further; Mem0's own graph tier is paywalled to Pro).
- **Swarm Memory:** a shared org-wide pool multiple agents read from and write into; new agents inherit accumulated domain knowledge instead of starting cold.
- **Benchmarks:** 15% relative LoCoMo accuracy improvement over "OpenClaw's native memory" (relative, not absolute — OpenClaw's own baseline isn't published in what the article reviewed) + 60%+ token/storage reduction, both **developer-reported, no independent third-party benchmark as of publication.** The article itself cites a field-wide **15–37 point gap between self-reported and third-party-verified figures** for this category (via vectorize.io) and contrasts JiuwenMemory's claim against other self-reported numbers (EverOS 93.05%, Mem0 91.6% on LoCoMo) — all in the same self-reported bucket. This is genuinely careful sourcing discipline for a framework-launch article; the caveats are the article doing its own verification hygiene, not something I had to dig for.
- **Storage backends:** SQLite, PostgreSQL, ChromaDB, GaussDB (Huawei's own DB); hosted on GitCode, a Chinese git platform.

## Relevance to core-memory-kit

**Task 95 / design.md §21 (dream re-curation) — weak-to-moderate, no new mechanism.** AutoDreaming's three named stages (filter → extract/categorize → dedup+resolve) match the same generic shape §21.2's three-stage pass already specifies (deterministic floor → LLM pass → op application), but the article supplies **no algorithmic detail** for the middle/resolution stage — not even at the level of detail the ChatGPT Dreaming coverage gave (which at least named "event-time revision" as an example). Nothing here is more specific than what §21 already contracts; no comparison table is warranted the way it was for the ChatGPT notes, because there's no disclosed mechanism to compare against.

**ADR-0023 / Task 232 (graph edges) — a fresh confirming negative exemplar, unverified.** GraphMemory is described exactly as the pattern ADR-0023 already **REJECTS**: an LLM-authored, continuously-updated knowledge graph used as the memory's own store (entity merge/dedup implies LLM judgment, not deterministic parse-what's-already-there). This lines up with the same-day graph-research pass's finding (`2026-07-21-synthesis-graph-vs-adr0023.md`) that headroom, MegaMemory, and Google's always-on-memory-agent all ship this same rejected shape. The difference: those notes cloned and grepped real source; **this note could not** — no repo link was given, so JiuwenMemory's GraphMemory is "described this way," not "verified this way." Treat as directionally-consistent with the standing REJECT line, not as new evidence weight.

**The kit's own no-server / local-first posture (ADR-0002; "the D-64 no-server class" per ADR-0023) — the sharpest actual point of contrast.** JiuwenMemory needs a hosted backend (SQLite/PostgreSQL/ChromaDB/GaussDB, via GitCode) and an agent-framework integration layer; the article's entire legal-risk section exists *because* AutoDreaming structures a user's conversation history into a profile that then lives somewhere subject to a specific jurisdiction's law. core-memory-kit's actual shipped design — plain committed markdown in the user's own git repo, no server, no vendor-hosted data layer, git-clone-portable — never introduces the article's own precondition ("if those structured profiles contain sensitive information... the legal framework... creates exposure that no privacy policy, open-source license, or self-hosting decision can eliminate"). This isn't a China-specific design choice the kit made; it's a side effect of ADR-0002's markdown-source-of-truth stance applying uniformly to every hosted-backend dependency, this one included.

## Borrow candidates

- **The episodic-vs-semantic split inside one tier (L2)** — a clean articulation ("preference/decision facts shouldn't share a bucket with background technical context") worth reusing as prose when explaining the kit's own tier stratification (`context/memory/<type>_<slug>.md`, `MEMORY.md` scratchpad, `archive.md`) — the kit already has an equivalent split by file-type/tier; this is a naming/communication borrow, not a new mechanism.
- **The article's own self-reported-vs-verified-gap discipline** (flagging vendor numbers as unverified, citing the field-wide 15–37 point gap) — a good model for how the kit's own docs should present any future benchmark claim (Task 99, Task-212 stats probe) — cite the caveat in the same breath as the number, not as a footnote.

## Reject candidates

- **JiuwenMemory itself, or any Huawei/openJiuwen-sourced dependency, as a component or reference implementation for the kit** — rejected outright, independent of technical merit, on the jurisdiction-risk grounds the article itself lays out: the kit's committed markdown facts can contain a user's genuinely sensitive project/business context, and introducing any dependency whose maintaining organization is bound to unconditional cooperation with a foreign intelligence-cooperation law is a supply-chain risk class the kit has never accepted for any dependency. Nothing here changes that baseline.
- **GraphMemory's LLM-authored continuously-merging KG pattern** — reject per ADR-0023's already-decided line, consistent with every other 2026-07-21 graph-research note; JiuwenMemory adds no new argument for reopening it.
- **Any of the reported benchmark numbers (15% LoCoMo improvement, 60%+ reduction, 80%/50%+ MemoryTurbo figures) as a target or comparison point** — self-reported by the article's own admission, no independent third-party benchmark exists.
