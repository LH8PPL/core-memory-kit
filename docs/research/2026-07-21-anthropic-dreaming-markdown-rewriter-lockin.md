# Anthropic Dreaming Is a Markdown Rewriter — The Vendor Lock-In Is Real

**Date:** 2026-07-21
**Source:** [Medium — Jaroslaw Wasowski, "Anthropic Dreaming Is a Markdown Rewriter — The Vendor Lock-In Is Real"](https://medium.com/@wasowski.jarek/anthropic-dreaming-is-a-markdown-rewriter-the-vendor-lock-in-is-real-1d5f44d290bf) (published 2026-05-27); local clip at `<local-wiki>/raw/Anthropic Dreaming Is a Markdown Rewriter — The Vendor Lock-In Is Real.md`

## What it claims

Anthropic's "Dreaming" feature (Managed Agents, research preview, announced Code with Claude SF, May 6–7 2026) packages agent memory consolidation into a single vendor API. The article decomposes it into **five architectural primitives** — (1) reflection scheduler, (2) memory schema, (3) fact-conflict resolution, (4) recall/reconsolidation separation, (5) redact-as-compliance — and argues that every agent-memory architecture must implement all five *somewhere*; Dreaming is the first vendor product to expose all five as one purchasable thing. It maps Dreaming against Letta sleep-time compute (arXiv:2504.13171), Mem0, APEX-MEM (arXiv:2604.14362/ACL 2026), and Vertex AI Memory Bank on a "topological spectrum" from flat (Mem0) → hierarchical/Git-backed (Letta) → vendor control-plane (Dreaming, Vertex). It cites a diagnostic benchmark (Yuan et al., "Diagnosing Retrieval vs. Utilization Bottlenecks in LLM Agent Memory," MemAgents Workshop @ ICLR 2026) as evidence that **retrieval method dominates write-strategy** for QA accuracy on LoCoMo, concluding write-path consolidation (what Dreaming sells) is secondary to retrieval quality. It closes with a DIY blueprint (3 databases + scheduler + evaluator LLM + reconciliation prompt) as the sovereignty-preserving alternative, and frames the whole piece around a headline claim that Harvey (legal AI) saw 6x task-completion gains from Dreaming (Anthropic-published, methodology undisclosed).

## What the evidence actually shows

I fetched the primary sources directly (not just trusting the article) — both the official Anthropic docs and the cited benchmark paper. Verdict: **the article's mechanism description is substantially accurate**, with several specific numbers/paths slightly off, one primitive's framing overstated, and one of the article's own diagrams internally contradicting its own body text.

**Confirmed exactly against `platform.claude.com/docs/en/managed-agents/dreams` and `.../memory` (fetched 2026-07-21):**
- A dream is an async job (`POST /v1/dreams`) taking a `memory_store` input + 1–100 `session_ids`, a `model`, and optional `instructions` (**exactly** capped at 4,096 chars — confirmed).
- Beta header is exactly `dreaming-2026-04-21` (plus `managed-agents-2026-04-01`) — confirmed verbatim.
- Input store is never modified; output is a wholly separate new memory store — confirmed verbatim ("the dream itself never deletes or modifies its inputs").
- Every memory mutation creates an immutable **memory version** (`memver_...`) — confirmed verbatim, this is real and it is Git-commit-shaped.
- The redact endpoint is real: `POST /v1/memory_stores/{store_id}/memory_versions/{version_id}/redact` (article said `.../versions/{ver_id}/redact` — close but the real path segment is `memory_versions`, not `versions`). Confirmed it "scrubs content out of a historical version while preserving the audit trail (who did what, when)" — matches the article's GDPR framing.
- Execution time "minutes to tens of minutes" — confirmed verbatim.
- Automatic-vs-review-gate control mode — confirmed conceptually: docs describe leveraging vs. discarding the output store; the article's "automatic/review gate" binary is a reasonable paraphrase (not a literal API field name I could find).

**Inaccurate, incomplete, or stale relative to today's primary source:**
- **Model whitelist is stale.** Article says "whitelisted to `claude-opus-4-7` or `claude-sonnet-4-6`" — correct for a May-2026 snapshot, but the docs I fetched today list five supported models (`claude-fable-5`, `claude-opus-4-8`, `claude-opus-4-7`, `claude-sonnet-5`, `claude-sonnet-4-6`). Expected drift, not an article error, but a reader relying on the article today would be wrong.
- **"Redact returns 200 OK + `redacted_at` timestamp"** — I could not verify this specific field name; the docs page I fetched shows the redact call but not its full response schema (that's on a separate API reference page I did not fetch). Unverified, not confirmed false.
- **Pricing ("$5/M input, $25/M output for Opus 4.7")** — not verifiable from the pages I fetched; the Dreams doc only says "billed at standard API token rates for the model you select" without a rate table on that page. Unverified.
- **Harvey's "6x" claim and its May 6, 2026 announcement** — not independently verified; I did not find/fetch an Anthropic press source for this specific number. The article itself flags the methodology as undisclosed, which I can't add to or subtract from.
- **Primitive 1 (reflection scheduler) is overstated as something Anthropic "chose."** The Dreams API has **no schedule/trigger field at all** — `POST /v1/dreams` just runs whenever you call it. There is no cron, no calendar object, nothing server-side that decides *when*. The "calendar trigger (dev-set)" framing is really "you, the developer, decide when to call this — same as the DIY path." This matters: of the article's five primitives, #1 is the one Anthropic does **not** actually own as a distinct product surface — it's bring-your-own-scheduler on both the vendor and DIY paths. The article's own summary line ("Anthropic took the write-path... the read-path stays on your side") is accurate for primitives 2–5 but primitive 1 doesn't cleanly belong to the vendor column either.
- **Memory schema is "Markdown files"** — docs say "workspace-scoped collection of **text documents**"; Markdown is the convention in every example (`.md` paths) but not a documented hard requirement. Minor overclaim.
- **Undisclosed but real and load-bearing (article omits these entirely):** per-memory cap 100 KB (~25k tokens); store cap 2,000 memories; max 8 memory stores per session; **memory versions are retained only 30 days** (with an ambiguous "recent versions always kept" exception) — so the "Git commit, point-in-time recovery" analogy is time-bounded, not eternal, unless you export. The article's framing implies indefinite audit history; the real product doesn't guarantee that past 30 days.
- **A version that is the current head of a live memory cannot be redacted** — you must overwrite/delete first, then redact the orphaned old version. The article doesn't mention this two-step requirement, which matters for anyone building a "one-call GDPR erasure" workflow on top of it.
- **`instructions` is synthesis guidance, not a line editor** — docs explicitly warn imperative line-level edits ("change sentence X to Y") "generally produce no change." The article's example ("Focus on migrating coding-style preferences") is the right *kind* of instruction but doesn't convey this limitation.

**Yuan et al. benchmark (verified via arXiv:2603.02473, not 2604.14362 as the article's APEX-MEM citation number might suggest confusion with — see Honest gaps):** confirmed real, ICLR 2026 MemAgents workshop, by Boqin Yuan et al. Abstract confirms: retrieval-method spread ~20 points (57.1%–77.2%) vastly exceeds write-strategy spread (3–8 points); "raw chunked storage... matches or outperforms expensive lossy alternatives" is a verbatim-adjacent paraphrase of the actual abstract language. The article's specific cell values (77.9%/81.1%/77.3%/73.3% etc., visible in the article's own infographic) were not independently cross-checked cell-by-cell against the PDF (I read the abstract/search snippet, not the full paper's results table) — treat those exact percentages as article-reported, not independently confirmed to the decimal.

**Internal inconsistency inside the article itself:** the article's own second table image ("Agent Memory Architecture Map," 5 systems × 6 dimensions) states Anthropic Dreaming's Trigger is "Implicit, continuous during interaction, post-session" and its Conflict Resolution is "Model-internal weighting, recency bias, **reinforcement learning**" — both of which contradict the article's own body text (explicit dev-called async job; LLM-judge-in-consolidation-pass) *and* the primary docs (no RL mentioned anywhere; no implicit/continuous trigger — it's an explicit API call). This table looks like filler/templated content that wasn't fact-checked against the rest of the piece. Treat that specific image's content as unreliable.

## Mechanism detail (the HOW)

**Dream job shape** (confirmed against primary docs):
```
POST /v1/dreams
  inputs: [
    {type: "memory_store", memory_store_id: <immutable input>},
    {type: "sessions", session_ids: [<1..100 transcript ids>]}
  ]
  model: one of 5 whitelisted Claude models
  instructions: <=4096 chars, synthesis guidance only (not a line-editor)
→ returns {id: "drm_...", status: "pending", outputs: []}
```
- Lifecycle: `pending → running → completed | failed | canceled`. On `failed`/`canceled`, the **partial output store is preserved** (not discarded) — you can inspect what got written before the stop, but there's no documented *automatic* resume; you'd have to kick off a fresh dream.
- Once `completed`, `outputs[]` contains a `memory_store` reference to the new store; you either **attach it** to future sessions (leverage) or **archive/delete it** (discard). The input store is untouched either way.
- Memory-store internals: each memory = a path-addressed text file (`/preferences/formatting.md` style), max 100 KB; store max 2,000 memories; max 8 stores attached per session, mounted read-only or read-write at `/mnt/memory/<slug>/`.
- Every write (by the agent live, by a human via the API, or by a completed dream) → new immutable `memver_...` row, retained 30 days (recent ones may live longer). Redact: `POST .../memory_versions/{id}/redact` — works only on **non-head** (historical) versions; nullifies content, keeps the actor/timestamp audit trail.
- **No built-in scheduler.** Anthropic ships zero cron/trigger machinery — "when to dream" is entirely the caller's responsibility, exactly like the article's own DIY blueprint.
- Security-relevant detail the article doesn't dwell on: a `read_write`-attached memory store processing untrusted input (user prompts, fetched web content, third-party tool output) is explicitly flagged by Anthropic's own docs as prompt-injection-writable — "a successful prompt injection could write malicious content into the store. Later sessions then read that content as trusted memory." This is Anthropic naming the exact threat class core-memory-kit's Poison_Guard / `screenBeforeCommittedWrite` exists to defend against.

## Relevance to core-memory-kit

Read against `specs/design.md` §21 ("Dream re-curation engine," Task 95) rather than a guess:

- **§21.1 explicitly cites "Anthropic Dreams' shape"** as one of the two design ancestors (alongside the Sleep paper's second phase) — this article is directly on-topic for that citation's accuracy. The shape holds up: raw-transcript + fact-corpus input → LLM restructuring → merge/supersede/insight-surface output is the same skeleton Dreaming ships.
- **Task 95's AUTO/QUEUE op-class split (§21.2 step 3) mirrors Dreaming's automatic-vs-review-gate control mode** — this article's confirmation that Anthropic ships exactly that binary (auto-attach vs. developer-approves) is independent validation that the split is the right shape, not an invented complexity.
- **Task 95 deliberately diverges from Dreaming on primitive 3 (fact-conflict resolution) — and this article's own critique explains why that divergence is warranted.** The article calls Dreaming's conflict resolution "the least-documented primitive... the place where vendor lock-in runs deepest — you can't see exactly what the LLM learns." §21.2's explicit rule — "the LLM proposes; CODE decides... which-wins on a contradiction is decided by EVENT-TIME, never by the LLM, and the LLM never counts recurrence" — is precisely the opacity problem this article flags, addressed structurally rather than left as an opaque LLM judgment. Worth citing this article alongside D-352 as the concrete "here's what happens if you don't do this" case.
- **Task 95's resumability (§21.4, ADR-0020: per-batch durable units, artifact-derived resume point) is stronger than what Dreaming documents.** Dreaming preserves partial output on failure but the docs describe no automatic resume — a failed dream just leaves an inspectable partial store; restarting is manual. This is a genuine point where the kit's design already exceeds the vendor's published behavior — useful as a confidence data point, not a gap to close.
- **The redact / memory_version primitive (#5) has no current analog in Task 95's scope**, and design.md §21.6's anti-scope note ("No DELETE op exists... only the human-only `cmk purge --hard`") is a *narrower* compliance posture than Dreaming's per-version redact-with-audit-trail. If GDPR/compliance-grade erasure ever becomes a stated requirement (it isn't currently, per the source-of-truth table), Dreaming's redact-on-historical-version-only pattern (can't redact a live head) is a concrete mechanism to borrow rather than reinvent.
- **The 30-day memory-version retention limit is a caution for any "audit trail = permanent" assumption** — core-memory-kit's own audit log (`​.locks/audit.log`) and archive tiers are file-based and not time-boxed the way Anthropic's hosted version history is; this article's finding doesn't change kit design, but it's a reminder not to model Dreaming's audit trail as strictly superior to the kit's append-only log — it's actually less durable past 30 days unless exported.
- **The prompt-injection-into-memory-store warning (docs, not the article) is a direct external confirmation of the exact threat model Poison_Guard/`screenBeforeCommittedWrite` (Task 216, §21.3) was built for** — Anthropic names the identical failure mode ("later sessions read [injected] content as trusted memory") for their own product. Good corroborating citation if design.md ever wants an external validation point for why the screen exists.
- **Task 232 / ADR-0023 (graph-recall activate-edges)** — no bearing. Dreaming's memory schema is a flat, path-addressed directory of text files; nothing in the primary docs or the article describes graph structure, edges, or activation spreading. `graph_relevance: none`.
- **Task 95 / design §21 is the correct comparison point; Tasks 233/161/203 (context-compaction) are a weaker fit** — Dreaming operates on a durable *store between sessions*, not a live context window being compacted mid-session, so it maps to the offline-consolidation family, not the compaction family.

## Borrow candidates

- **Redact-only-on-non-head-version rule.** If/when the kit ever adds hard-erasure-with-audit-trail beyond `cmk purge --hard`, the "can't redact the live head, must supersede first" rule is a clean, easy-to-reason-about invariant worth copying.
- **Automatic-vs-review-gate as the exact same binary Task 95 already chose (AUTO/QUEUE)** — no new borrow needed, but this article is citable evidence that the binary is externally validated, not an invented complexity.
- **The prompt-injection-into-memory-store framing as a citable external corroboration** for why `screenBeforeCommittedWrite` exists — useful for design.md prose, not a mechanism to copy (the kit's screen is already more specific than Anthropic's blanket warning).
- **Explicit per-store/per-memory numeric caps (100 KB/memory, 2,000 memories/store) as a reference data point** when next tuning the kit's own per-file/per-tier caps (design §7.1's cap-coordination rule) — not a value to copy verbatim (different tech stack) but a sanity-check that hard numeric ceilings on a memory store are normal industry practice, not overcautious.

## Reject candidates

- **"Reflection scheduler" as a vendor-owned primitive** — reject the framing as stated. Verified against primary docs: Anthropic ships no scheduler/trigger surface at all for Dreams; it's 100% developer-side, identical to what the kit already does with its idle/cron machinery (§21.4). Don't treat this article as evidence the kit is missing a "primitive" here — it isn't; the vendor doesn't have one either.
- **Adopting Markdown-as-the-schema as if it were an Anthropic mechanical requirement** — reject as unnecessary; the kit already made this exact choice for the kit's own reasons (design §1.3 etc.), and Anthropic's own docs don't even mandate it (just "text documents").
- **Trusting the article's second infographic (5 systems × 6 dimensions) for any Anthropic-Dreaming-specific facts** — reject that specific image as a source; it self-contradicts the article's own body text and the primary docs on both trigger type and conflict-resolution mechanism (see "internal inconsistency" above). The other five images in the article were internally consistent with the body text and matched primary sources everywhere I could check.
