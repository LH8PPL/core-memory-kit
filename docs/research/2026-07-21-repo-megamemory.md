## What it claims (README)

An MCP server ("the LLM is the indexer, no AST parsing") that gives coding agents a persistent per-project knowledge graph of **concepts** (feature/module/pattern/config/decision/component) connected by typed **relationships** (connects_to/depends_on/implements/calls/configured_by), stored in `.megamemory/knowledge.db` (SQLite). Loop: `understand → work → update`. 9 MCP tools. Web graph explorer (d3-force + Canvas). Two-way `merge`/`conflicts`/`resolve` CLI for reconciling `knowledge.db` across git branches (SQLite files don't merge in git). Brute-force cosine similarity search, "fast enough for graphs with <10k nodes." In-process embeddings via Xenova/all-MiniLM-L6-v2, no API keys, no network calls after first model download.

## What the evidence actually shows (code-level verification)

This is a **real graph store**, not a README word — verified directly against `src/db.ts` and `src/types.ts`:

- `src/db.ts` lines 51–98: `CREATE TABLE nodes (...)` and `CREATE TABLE edges (from_id, to_id, relation, ...)` with actual `FOREIGN KEY` constraints (`parent_id → nodes(id) ON DELETE SET NULL`, edge endpoints `ON DELETE CASCADE`), plus 10 indexes including `idx_edges_from`/`idx_edges_to`/`idx_edges_relation`.
- Edges are actually **traversed**, not just stored: `getOutgoingEdges`/`getIncomingEdges` (db.ts:423–453) do real `JOIN nodes n ON e.to_id = n.id` queries, and `src/tools.ts` `buildNodeWithContext` (line 58) assembles `children` + `edges` (outgoing) + `incoming_edges` + `parent` for every concept returned to the agent — this is genuine graph-context assembly on every read, not cosmetic.
- Retrieval path confirmed in `tools.ts` `understand()` (line 109): embed the query (`embeddings.ts`), brute-force cosine over ALL active node embeddings (`getAllActiveNodesWithEmbeddings` + `findTopK`, no ANN index — matches the README's own "brute-force… <10k nodes" caveat exactly), then hydrate each hit with full graph context. No FTS/BM25 keyword layer at all — semantic-only recall.
- Embeddings confirmed real and in-process: `src/embeddings.ts` uses `@xenova/transformers` `pipeline("feature-extraction", "Xenova/all-MiniLM-L6-v2", {quantized:true})`, 384-dim, mean-pooled, normalized — matches the README claim exactly, no API calls.
- Schema is versioned (`SCHEMA_VERSION = 4`, `PRAGMA user_version`) with real migrations (v1 base tables → v2 merge columns → v3 timeline table → v4 unique-edge-index + dedup cleanup), guarded by `BEGIN IMMEDIATE` + re-check-after-lock — this is careful, not hand-waved.

**Files verified for the "is graph real" question specifically:** `src/db.ts` (schema+queries), `src/types.ts` (Node/Edge types), `src/tools.ts` (`buildNodeWithContext`, `understand`, `link`), `src/index.ts` (9 MCP tool registrations matching README's table exactly: understand, get_concept, create_concept, update_concept, link, remove_concept, list_roots, list_conflicts, resolve_conflict).

## Mechanism detail — the HOW

**Write path (agent-driven, no automatic capture):**
- `create_concept` (tools.ts:150): slugify name→id (`makeId`, optional parent-prefix), reject if id exists or parent missing, embed `"{kind}: {name} — {summary}"`, `insertNodeAndEdges` in one transaction (edges only inserted if the target already exists — dangling edges silently dropped, not queued).
- `update_concept`: re-embeds only if `name`/`summary` changed.
- `link`: validates both endpoints exist, `INSERT OR IGNORE` on the unique `(from_id,to_id,relation)` index (v4 migration added this — dedupes accidental double-links).
- `remove_concept`: **soft-delete** (`removed_at`/`removed_reason` set, row kept) but **hard-deletes all edges touching it** (`softDeleteNode` → `DELETE FROM edges WHERE from_id=? OR to_id=?`) — so edge history does NOT survive a concept's removal, only the node's own history does.

**There is no automatic extraction, no hook, no background job, no cron, no consolidation pass anywhere in the codebase.** Grepped the full `src/` tree: zero scheduling, zero "dream"/"consolidate"/"curate" logic, zero deterministic-dedup floor. The two moments memory gets written are entirely agent/user-initiated via slash commands:
- `commands/bootstrap-memory.md`: a prompt instructing the agent to `git ls-files | wc -l | sort` the repo, read the top ~150 files by size, and manually `create_concept`/`link` the top 10–15 concepts. One-shot, run by hand.
- `commands/save-memory.md`: a prompt instructing the agent to reflect on "this session," call `understand` per topic to check for existing concepts (dedup by hand, in-context, no embedding-threshold code path — the LLM decides similarity by reading, not by any θ/cosine gate), then `create_concept`/`update_concept`/`link`/`remove_concept`. Also one-shot, run by hand (or by the user invoking `/user:save-memory`).

**Merge/conflict engine** (`src/merge.ts`, `src/tools.ts` listConflicts/resolveConflict) exists **because SQLite files can't be diffed/merged by git** across branches. `MergeEngine.merge()` opens two `.db` files read-only, walks all node IDs, and for each: identical-content → keep one copy clean; content differs → both kept as `id::left`/`id::right` under a shared `merge_group` UUID with `needs_merge=1`; edges to conflicted nodes deferred and remapped in a second pass to avoid FK violations. `resolve_conflict` (an MCP tool) is explicitly a **agent-verifies-against-current-codebase** flow: the agent reads both `::left`/`::right` versions, checks the real source, and writes back a resolved summary + a `reason` string documenting what was verified — not a blind pick-a-side.

**Web explorer** (`web/index.html`, 2133 lines, confirmed via grep to use `d3` not the `Cytoscape.js` the CHANGELOG claims — see contradictions below): single-file d3-force + Canvas visualization, colored by kind, sized by edge count, with a `timeline` scrubber backed by `db.ts` `getNodesAtTime`/`getEdgesAtTime` (time-travel queries: nodes/edges whose `created_at ≤ t` and not yet `removed_at`). The `timeline` table (schema v3, `insertTimelineEntry` in `src/timeline.ts`) logs every MCP tool call (tool, params JSON, result summary, is_write, is_error, affected_ids) — but this table is **only consumed by `stats.ts` and `web.ts`**, never exposed as its own MCP tool (no `mk_timeline`-equivalent for the agent to query mid-session; confirmed via grep — `getTimelineEntries`/`getTimelineTicks` calls appear only in `stats.ts`/`web.ts`, not `index.ts`/`tools.ts`).

## Compare against core-memory-kit's real contracts

**Task 95 (dream re-curation engine, design.md §21):** relevance is **none**, verified, not assumed. §21's mechanism contract is: (1) a deterministic zero-LLM dedup floor, (2) exactly ONE batched LLM call per run that returns `add/update/supersede/none` + validated target_ids + an AUTO-vs-QUEUE op-class split with a reviewable diff queue (`context/queues/recuration.md`), (3) event-time-decided contradiction resolution (never LLM-decided), (4) resumability per ADR-0020, (5) provenance/source-trust tagging. MegaMemory has **zero** of these five: no dedup floor (dedup is "the agent reads `understand` output and decides in-context" — `commands/save-memory.md` step 3), no batching (every write is one MCP call at a time, agent-paced), no AUTO/QUEUE split (every write applies immediately, no review queue), no scheduling of any kind (both capture moments are user/agent-invoked slash commands, not hooks or crons), no supersession/validity-window concept at all (only hard soft-delete + merge-conflict tracking, which is a different problem — reconciling two DIVERGED branches, not consolidating one growing corpus over time). This is a fully agent-driven "you type `/save-memory`, the agent decides what's worth keeping" model — the mem0-style "manual LLM judge at write time" pattern the kit's own research base already evaluated and moved past (deterministic floor precisely because per-write LLM judgment doesn't scale/dedupe reliably at volume).

**ADR-0023 (graph edges — activate/defer/reject):** relevance is **weak but confirmatory**. ADR-0023 (accepted 2026-07-19, two days before this read) already surveyed 9 systems + Letta and settled on: markdown stays the single source of truth (ADR-0002), a lightweight `edges` table gets DERIVED from `related:` frontmatter + `[[slug]]` mentions at reindex time (rebuildable exactly like FTS), and it explicitly **REJECTS** "LLM-extracted typed KGs as a store" citing drift + non-deterministic rebuild. MegaMemory is a clean, well-built instance of precisely the rejected pattern: SQLite (not markdown) is the source of truth; nodes/edges are **authored directly by the LLM** via `create_concept`/`link` tool calls (not derived from any human-legible file); a fresh `megamemory install` + normal git usage means the graph and the markdown/code it describes CAN silently diverge with no rebuild path back to ground truth (contrast: core-memory-kit's `edges` table is disposable/rebuildable from markdown at any time). MegaMemory's own `merge`/`conflicts`/`resolve` CLI is the concrete cost of that choice made visible: **because the graph isn't derived from a git-mergeable text format, they had to hand-build a custom two-way SQLite merge engine** to solve a problem ADR-0002's markdown-first choice sidesteps entirely (git already merges the `.md` files that would drive edge-derivation). This is useful **evidence for** ADR-0002/0023's reasoning, not a challenge to it — a concrete "here's what it costs to NOT do it the way we chose."

**Tasks 233/161/203 (context-compaction, daily-distill):** no relevant mechanism found. MegaMemory has no rolling-window/compaction concept at all — nodes accumulate indefinitely (soft-delete only, no archival tier, no distillation).

## Borrow candidates

- **Time-travel queries as a pattern** (`getNodesAtTime`/`getEdgesAtTime`, db.ts:891–912): "what did the graph look like at timestamp T" via `created_at ≤ t AND (removed_at IS NULL OR removed_at > t)`. Cheap, general SQL pattern — if core-memory-kit's `cmk timeline`/`mk_timeline` (already shipped, per MCP.md) ever needs a "graph state as-of" view once ADR-0023's edges table lands, this is a clean two-line WHERE-clause shape worth 1:1 copying rather than re-deriving.
- **`resolve_conflict`'s "verify against current codebase, don't pick a side blindly" contract** (tools.ts:354, the MCP tool description text): a crisp phrasing of an invariant core-memory-kit's own conflict-queue (Task 25/25b) already enforces structurally — worth as a one-line doc/description-text polish reference for `mk_queue_resolve`'s tool description, not a mechanism change.
- **The `edges(from_id,to_id,relation)` UNIQUE index added retroactively in schema v4** (db.ts:157–170), with an explicit pre-migration `DELETE ... WHERE id NOT IN (SELECT MIN(id) ... GROUP BY from_id,to_id,relation)` dedup step before adding the constraint: a clean small pattern for "add a uniqueness constraint to an existing table that may already have dupes" — directly reusable shape if/when ADR-0023's `edges` table needs the same retrofit.

## Reject candidates (with reasons)

- **The whole storage architecture (SQLite-as-source-of-truth for the graph)** — rejected already, explicitly, by ADR-0002 + ADR-0023, for reasons this repo's own merge-engine complexity now empirically illustrates (a from-scratch two-way conflict-resolving merge algorithm exists ONLY because the format isn't git-mergeable).
- **LLM-authored edges as the sole edge source** — rejected by ADR-0023 in favor of frontmatter/mention-derived edges; MegaMemory's `link`/`edges` params are exactly the pattern named and rejected there ("LLM-extracted typed KGs as a store").
- **Manual slash-command-only capture (`/save-memory`, `/bootstrap-memory`)** as a substitute for automatic extraction — core-memory-kit already has Stop-hook auto-extract as the default with explicit-phrase override as the exception (design.md §6.0); MegaMemory has the inverse (explicit-only, no default automatic path), which is a strictly weaker capture guarantee for the "user forgets to run the command" case. Not a design worth adopting even partially.
- **Brute-force cosine with no keyword/FTS layer at all** — MegaMemory's `understand()` is semantic-only; core-memory-kit's FTS5+sqlite-vec RRF-fused hybrid (ADR-0015) is already a strictly more robust retrieval design and the graph-memory sweep (ADR-0023's own research base) found hybrid beats semantic-only on exact-token queries. No reason to narrow to MegaMemory's simpler shape.
