## What it claims

(Per the article/docs, now checked against code.) Markdown is the source of truth; SQLite + LanceDB are rebuildable indexes. Hybrid retrieval is "one LanceDB query = BM25 + vector ANN + scalar filter." A `reflect_episodes` "Reflection" pass merges episode clusters between sessions, offline, self-evolving. Cases distill into Skills with no manual curation ("self-evolving skills"). Scoped by `user_id`/`agent_id`/`app_id`/`project_id`/`session_id`.

## What the evidence actually shows

**Markdown-source-of-truth: confirmed.** `src/everos/core/persistence/markdown/` writes `.md` files (daily-log append for episodes/facts/foresights/cases, single-file rewrite for `user.md`, one-directory-per-skill for `SKILL.md`); LanceDB rows carry `md_path` back-pointers.

**"Single query" hybrid framing: contradicted by code.** The retrieval-mechanism claim in the prior docs-only note ("single LanceDB query = BM25 + vector ANN + scalar filter") is **not what `SearchManager` does**. `sparse_recall` (LanceDB FTS/tantivy BM25 via `MatchQuery`/`BooleanQuery`) and `dense_recall` (LanceDB cosine ANN) are **two separate async calls, run in parallel** (`asyncio.gather(recaller.sparse_recall(...), recaller.dense_recall(...))`, `search/manager.py:509-514`) against the same `where`-filtered table. Each call is a genuine single hybrid-scoped LanceDB query (BM25+filter, or ANN+filter) — but the **fusion of the two result sets is application code**, not a LanceDB primitive: three distinct fusion strategies dispatched by `resolve_pipeline()` (`search/adapter.py`) — `"hierarchy"` (episode/atomic_fact: RRF-ordered heap expansion + LR-calibrated global top-N competition, `search/hierarchy.py`), `"vector_anchored"` (agent_case: `everalgo.rank.arank`, alpha=0.7), `"skill_hybrid"` (agent_skill: RRF → cross-encoder rerank → optional LLM verify, `search/skill_hybrid.py`). This is real, non-trivial, hand-written retrieval-fusion engineering — just not the "one query" shape the marketing implies.

**The actual ranking/extraction algorithms live OUTSIDE this repo.** `everalgo-{core,rank,clustering,agent-memory,user-memory,knowledge,boundary}` are separate packages on PyPI (`uv.lock` lines 444-499+; sdist+wheel, not vendored, not present anywhere under this repo's tree — verified `find . -iname '*everalgo*'` returns nothing). RRF, the LR-calibration formula (`cosine_to_lr_score`), the LLM-refined clustering (`cluster_by_llm`), `AgentSkillExtractor`, `EpisodeReflector` — all imported `from everalgo...`, none of their implementations are in the EverOS git repo. **I did not fetch/read `everalgo`; this note's mechanism claims about those algorithms are limited to call-shape (what's passed in/out), not internals.**

**Reflection is real, auto-apply, and disabled by default — confirmed at config + code level.** `src/everos/config/default_ome.toml:51-53`: `[strategies.reflect_episodes]` is **commented out**, with `enabled = false`, `cron = "0 2 * * 1"` (Monday 02:00) as the documented default — matches the 2026-07-21 docs-only note exactly, now verified against the shipped config file rather than `how-memory-works.md`. `src/everos/memory/reflection/orchestrator.py` (1094 lines, fully read): `ReflectionOrchestrator.run()` → `_select_candidates` (clusters with ≥2 members) → `_process_cluster` → `_reflect_cluster` (calls `everalgo`'s `EpisodeReflector.areflect`, INIT or UPDATE mode) → `_write_and_reextract` (writes the merged episode straight to markdown via `episode_writer.append_entries`, no staging) → `_deprecate` (patches `deprecated_entries` into the **source files'** frontmatter, sets `deprecated_by` on the LanceDB rows for both episodes and their atomic facts, under a partition lock). **No review/approval/queue step exists anywhere in this flow** — confirms the docs-only note's inference; this is now a code-verified claim, not a documentation-silence inference. Non-destructive in the archive-not-delete sense (originals stay on disk, tagged `deprecated_by`), but **lossy in content**: the LLM narrative-merges N episodes into one and atomic facts are *re-extracted from the merged text* (not preserved verbatim), so per-episode detail can be compressed away even though the raw file survives.

**Self-evolving skills: a real, multi-stage, auto-apply pipeline.** `AgentCaseExtracted` event → `trigger_skill_clustering.py` (quality-gated at `quality_score < 0.2`, embeds `task_intent`, calls `everalgo.clustering.cluster_by_llm` to merge into an existing cluster or mint a new one) → emits `SkillClusterUpdated` → `extract_agent_skill.py` (selects existing skills in-cluster — scalar fetch if ≤10, cosine top-K if not — hydrates `supporting_cases` from `source_case_ids` lineage, calls `everalgo.agent_memory.AgentSkillExtractor.aextract`, writes each emitted skill straight to `SKILL.md` via `AgentSkillWriter`). Auto-apply, no review gate, same as Reflection. `SKILL.md` carries `source_case_ids` as a lineage/provenance list.

**Graph: confirmed NOT a real graph store — README-and-marketing-silent-on-this too, so this isn't even a contradicted claim, just a negative check for ADR-0023 purposes.** `grep -rli graph src/` → 6 hits, all false positives on substrings (`lexicographic`, `paragraph`, `historiography`) except one docstring in `_parent_type.py` ("Three values cover the current provenance graph") describing a 3-value `ParentType` enum (`memcell`/`episode`/`cluster`) — a discriminator field, not a traversable structure. `grep -rli "node\|edge\|traversal\|networkx\|neo4j\|kuzu"` turns up only: (a) `knowledge_topic` rows called "L2 topic node" — a taxonomy tree for the separate `knowledge/` doc-ingestion feature, no cross-links; (b) `search/hierarchy.py`'s two-level episode→atomic_fact parent/child expansion, which is a **retrieval-time ranking structure** (a heap + a dict of `topn` candidates, `episode_to_facts: dict[str, list[FactCandidate]]`), rebuilt per-query from `parent_id`/`entry_id` metadata — not a persisted, queryable, typed edge store. No graph library dependency (no networkx/neo4j/kuzu in `pyproject.toml`). **EverOS ships no graph.**

## Mechanism detail (the HOW)

```
POST /add → unprocessed_buffer (SQLite) → boundary detection / flush
  → LLM extracts MemCell
  → UserMemoryPipeline (SYNC): writes episode .md, response blocks on this
  → AgentMemoryPipeline (ASYNC): emits AgentPipelineStarted
  → OME (Offline Memory Engine) strategies fire off events:
       extract_atomic_facts / extract_foresight / extract_user_profile
       extract_agent_case → trigger_skill_clustering → extract_agent_skill
       reflect_episodes (disabled by default, cron 0 2 * * 1)
  → Cascade daemon (in-process filesystem watcher, watchdog) sees the .md
    write → md_change_state queue (SQLite, durable) → worker diffs by
    content_sha256, re-embeds only changed entries → upserts LanceDB rows
```

Retrieval (`search/manager.py`, fully read, 800 lines): per-kind (episode/atomic_fact/agent_case/agent_skill), `resolve_pipeline(method, kind)` picks KEYWORD (BM25-only), VECTOR (ANN-only, plus a MaxSim atomic-fact→episode max-pool variant), HYBRID (parallel BM25+ANN → kind-specific fusion, no LLM rerank by default), or AGENTIC (multi-query + cross-encoder + LLM sufficiency loop, in `search/agentic.py`, not read in depth this pass).

Reflection (`reflection/orchestrator.py`): Select (clusters, ≥2 members) → Merge (LLM narrative merge, `everalgo`) → Re-extract (atomic facts from the merged text) → Deprecate (frontmatter `deprecated_entries` + LanceDB `deprecated_by` on both episodes and facts, partition-locked). Auto-apply; scheduled but off by default.

## Relevance to core-memory-kit

**graph_relevance: none.** No node/edge model, no graph library, no traversal query surface anywhere in 33K LOC. Confirms (does not newly establish) the pattern ADR-0023's 9-repo sweep already found — "the field's own 'graph memory' flagships under-deliver in shipped code" — EverOS isn't even a claimant here (it never markets itself as graph-based), so this is a clean negative data point rather than a caught overclaim. Nothing here revisits ADR-0023's ACCEPT/DEFER/REJECT split; the activation-slice decision (parse `related:`/`[[slug]]` into an `edges` table, recursive-CTE supersession walk, `cmk links`) has no EverOS counterpart to borrow from or contrast against beyond what the ADR already cites.

**task95_relevance: moderate — code confirms the docs-only note's inference, and sharpens the divergence from design.md §21.** Reflection's Select→Merge→Re-extract→Deprecate is the closest real-world analog to the dream re-curation engine (design.md §21), and now confirmed at code level (not inferred from doc silence) to **auto-apply with zero review gate** — directly the shape §21.6's anti-scope line rejects ("no in-entry trail-lossy merges... no unscreened raw-verbatim to the LLM"). Two concrete, code-verified contrasts worth carrying into Task 95's build:
1. EverOS's merge is genuinely **lossy at the content layer** (LLM narrative-compresses N episodes → 1, atomic facts re-extracted from the compressed text, original detail only recoverable by going back to the deprecated raw file) even though it's **non-destructive at the storage layer** (nothing deleted, `deprecated_by` pointer). §21's design keeps this pair of properties but adds a THIRD the EverOS code doesn't have: a QUEUE-class staging step before any lossy/generative op lands (§21.2 step 3) — EverOS writes the merge directly with no equivalent to `context/queues/recuration.md`.
2. EverOS's provenance pattern (`deprecated_entries` frontmatter map old→merged id, `source_members_json`/`source_case_ids` lineage lists, partition-locked writes) is a clean, independently-arrived-at confirmation of the shape §21.3 already specifies (source pointers on every condensed claim) — reinforcing, not adding to, an already-settled mechanism.

**Tasks 233/161/203 (context-compaction): convergent-design confirmation, no new mechanism, now with cron-config verification.** Daily-log append (`episode-<YYYY-MM-DD>.md`) matches the kit's `today-<date>.md` shape. The cascade daemon's `content_sha256`-keyed, entry-level selective re-embed (only touch what actually changed) — confirmed for real this pass by reading `cascade/worker.py`'s call shape (not fully traced line-by-line) — is the same "bound the work to what changed" instinct as ADR-0020's resumability contract, applied to index maintenance. Worth a deliberate comparison against the kit's own reindex granularity, not an assumed gap.

## Borrow candidates

- The `deprecated_entries: {old_entry_id: merged_entry_id}` frontmatter map as a **file-local** provenance pointer (distinct from a global `superseded_by` FK) — worth comparing against however the kit's supersession chain currently threads through multiple originals merging into one target, if that many-to-one shape ever comes up for facts (Task 95's `target_ids[]` many-to-many already covers this at the recall/dedup layer per design §21.2 step 2 — EverOS's version is storage-layer, a different composition point).
- `content_sha256`-keyed selective re-embedding at entry granularity (not file granularity) — a concrete pattern to check the kit's reindex path against, not an assumed win.
- The partition-lock discipline (`get_partition_lock("reflection_deprecate", f"{app}:{project}:{cluster}")`) around any read-decide-write cycle that could race across concurrent OME strategy runs — a clean small pattern for any future concurrent-writer surface in the kit.

## Reject candidates

- **Auto-apply lossy merge with no review step** — directly conflicts with the kit's already-settled §21.2 QUEUE class + §21.6 anti-scope. Confirmed real in code, not just plausible from doc silence; still a reject, now with more confidence.
- **LanceDB as a fourth storage engine** — no new information this pass; FTS5+sqlite-vec already cover the same ground per the 2026-06-25 note's original verdict.
- **Splitting the "algorithm" layer into a separately-versioned package family (`everalgo-*`)** — reasonable for EverOS's product shape (multiple consumers, PyPI distribution cadence), but not a fit for the kit's single-consumer, in-repo, markdown-first posture; noted only because it explains why this pass could not verify the fusion-formula internals, not proposed as something to copy.
