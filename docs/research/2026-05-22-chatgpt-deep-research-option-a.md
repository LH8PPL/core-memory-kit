# Comparative Analysis of LLM Memory Systems for In-Repo Project Memory

## Executive assessment

The field has clearly split into **two major camps** by May 2026. One camp treats memory as a managed service or database-backed substrate that extracts salient facts and retrieves them with vector or graph methods. That is the center of gravity for Mem0, Letta, Cognee, Zep/Graphiti, and LangMem. The other camp is pushing toward **local-first, more auditable memory**, including markdown files, raw-event retention, session-state virtualization, or explicit provenance. That counter-current is most visible in AnthropicŌĆÖs Claude Code memory, Basic Memory, Claude Code CMV, Nautilus Compass, and several recent papers such as Codebase-Memory and True Memory. For **your exact target**ŌĆöan **in-repo, per-project memory system** with durable facts stored as **human-readable markdown** and a **per-fact archive**ŌĆöthere is still real whitespace. AnthropicŌĆÖs official Claude Code memory is the closest mainstream reference point, but its auto memory lives under `~/.claude/projects/<project>/memory/`, not inside the repo, and uses a concise `MEMORY.md` entrypoint plus topic files rather than a fact-ledger model. Basic Memory is the closest open-source practitioner system to your philosophy: it is local-first, markdown-native, hand-editable, semantic-searchable, and MCP-native across multiple clients. ## Baseline systems

The verified baseline systems cluster into a few recognizable patterns. Mem0 emphasizes a **universal, multi-level memory layer** with user, session, and agent state. Letta emphasizes **stateful agents** with explicit memory blocks and continual self-improvement. Cognee and Zep sit more in the **graph / memory-control-plane** family. LangMem focuses on **hot-path memory tools plus background memory management** for LangGraph-based agents. AnthropicŌĆÖs Claude Code memory is different: it separates **human-authored instruction files** from **auto memory learned from corrections**, both tied to project scope. ## Synthesis for your design

### Design patterns that appear to be winning

- **Layered memory beats a single monolithic memory bucket.** Mem0 explicitly describes user/session/agent layers; Claude Code separates `CLAUDE.md` guidance from project auto memory; EverMemOS separates episodic traces, MemCells, MemScenes, and profiles; MemOS broadens the layering idea even further across plaintext, activation, and parameter memory. The recurring lesson is that ŌĆ£memoryŌĆØ is not one store; it is several stores with different lifetimes and compression rules. ŅłĆciteŅłéturn9view5Ņłéturn29view1Ņłéturn24academia2Ņłéturn24academia3Ņłü

- **A short startup entrypoint plus expandable deeper memory is becoming a standard pattern.** Claude Code loads `CLAUDE.md` and only the concise `MEMORY.md` entrypoint at session start, with topic files read on demand. Basic Memory similarly keeps canonical markdown while using search and links to surface the relevant subset. LangMem formalizes the same idea in framework form by splitting hot-path memory tools from background management. ŅłĆciteŅłéturn29view0Ņłéturn29view1Ņłéturn21view1Ņłéturn51view1Ņłü

- **Hybrid retrieval is outperforming single-mode retrieval.** Cognee combines embeddings and graphs; Zep/Graphiti is explicitly temporal KG plus episode provenance; Mem0 offers a graph-memory variant on top of extracted memory; AnchorMemŌĆÖs entire contribution is the pairing of anchored facts with associative contexts. The practical pattern is that dense retrieval alone is rarely enough once time, relations, or evolving facts matter. ŅłĆciteŅłéturn9view2Ņłéturn9view4Ņłéturn39academia8Ņłéturn22view1Ņłü

- **Background consolidation is increasingly separate from foreground recall.** LangMem exposes a background memory manager; EverMemOS has semantic consolidation; Mem0ŌĆÖs paper centers extraction, consolidation, and retrieval as separate stages. This suggests that a strong design should not force every write path to carry the full consolidation cost. ŅłĆciteŅłéturn51view1Ņłéturn24academia2Ņłéturn39academia8Ņłü

- **Provenance is moving from ŌĆ£nice to haveŌĆØ to ŌĆ£architectural requirement.ŌĆØ** Graphiti keeps episodes as ground-truth provenance for derived facts; Nautilus adds a Merkle integrity log; A-MemGuard and MemLineage both push the idea that future memory systems need explicit chains of trust and origin, not just retrieval scores. ŅłĆciteŅłéturn9view4Ņłéturn23view1Ņłéturn37academia2Ņłéturn37academia4Ņłü

### Controversial design choices where systems disagree

- **Extraction at ingest vs raw-event retention is the central fight.** Mem0, LangMem, and many graph systems extract and consolidate at write time. Nautilus Compass argues against LLM extraction at index time entirely. True Memory goes further and explicitly argues that ŌĆ£extraction at ingestion is the wrong primitive,ŌĆØ preferring retrieval-centered recall over preserved verbatim events. MemMachine and Codebase-Memory also strengthen the verbatim-or-structural-first side of the argument. ŅłĆciteŅłéturn39academia8Ņłéturn51view1Ņłéturn20academia2Ņłéturn20academia3Ņłéturn20academia1Ņłéturn26academia0Ņłü

- **Opaque service storage vs human-readable files remains unresolved.** Claude Code and Basic Memory show that markdown files are viable and useful. Most other production-oriented systems still optimize around opaque stores or managed services, largely because hybrid retrieval, concurrency, and background compaction are easier there. This is probably the single most relevant fork for your design. ŅłĆciteŅłéturn29view0Ņłéturn21view1Ņłéturn9view5Ņłéturn10academia0Ņłü

- **Passive capture vs explicit, tool-mediated capture varies a lot.** Claude Code auto memory is passive/learned, while `CLAUDE.md` is manually authored. LangMem exposes explicit manage/search tools, while also supporting background extraction. CMV is almost entirely active and user-invoked. Different systems disagree on whether ŌĆ£rememberingŌĆØ should feel automatic or deliberate. ŅłĆciteŅłéturn29view1Ņłéturn51view1Ņłéturn22view0Ņłü

- **Memory for a single assistant vs portable memory across agents is another divide.** Claude Code memory is tightly Claude-shaped. Supermemory MCP, Nautilus Compass, Basic Memory, and MemOS/EverOS all push harder toward memory as a reusable substrate across clients, CLIs, or agent ecosystems.
