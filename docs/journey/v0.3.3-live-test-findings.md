**Why record it anyway (the signal):** the model emitted a
required-string-missing first call **three times in one short session** — a
pattern, not a one-off. For `mk_remember` the transcript shows the model
reliably providing the **title/slug** while sometimes omitting **`text`**.

**RESOLUTION (the user's call, 2026-06-17): log it, no code change.** The
deciding evidence is that the **same `-32602`-required-string-undefined error
hit Context7's `resolve-library-id` twice** — a third-party tool the kit can't

### 🚩 FINDING F-V0.3.3-2 — CUT-BLOCKER: `cmk remember` leaks the username into the fact FILENAME + INDEX when no `--title` is given (home-path sanitized in the body but NOT before slug derivation)

**Probe (C4):** `cmk remember "venv at C:\Users\<you>\proj\.venv" --type project` →

- **Body: sanitized** ✓ — shows `venv at ~\proj\.venv` (no username).
- **Filename: LEAKED** ✗ — `project_venv-at-c-users-<you>-proj-venv.md` (the real username in the committed filename).
- **INDEX.md: LEAKED** ✗ — the auto-generated link `[venv at ~\proj\.venv](project_venv-at-c-users-<you>-proj-venv.md)` carries the username in the path.

**Why it's a cut-blocker:** a committed fact filename + INDEX pointer containing the user's real username is exactly the **public-repo username-leak class** the kit's whole privacy subsystem exists to prevent. `git add context/` ships it. (The body redaction is correct, which masks it on a quick look — you only catch it by reading the FILENAME, the G4 standing-rule's reason.)

**Root cause (verified in `remember-core.mjs` rememberRich):** the slug is `slugifyFact(title)` where `title = sanitizePrivacyTags(text).slice(0,80)` — **`<private>` IS stripped before the title/slug (the v0.3.1 fix, lines 57-67), but `sanitizeHomePaths` is NOT.** Home-path abstraction (`C:\Users\you` → `~`) runs later inside `writeFact` on the body + frontmatter title, AFTER the slug was already derived from the un-sanitized title. So the same ordering fix v0.3.1 applied for `<private>` was never applied for home paths in this spot. Same class, missed sibling.

**The fix (ready):** in `rememberRich`, apply `sanitizeHomePaths` to the headline/title in the SAME place `sanitizePrivacyTags` already runs (before `slugifyFact`), so the slug only ever sees the abstracted `~` form. Mirror it in `richFactTitle` (the caller-message preview) for consistency. One-spot fix + a regression test (a username in the text → assert it appears in NEITHER the filename NOR INDEX.md). **Verify the MCP `mk_remember` rich path too** (it shares `rememberRich`, so the fix covers both surfaces).

**Status: RESOLVED (2026-06-17, branch `fix-remember-slug-homepath-leak`).** Fixed in `rememberRich` + `richFactTitle`: a `clean = (s) => sanitizeHomePaths(sanitizePrivacyTags(s))` helper now runs on the headline + title BEFORE `slugifyFact`, so the slug only ever sees the `~`-abstracted form (`sanitizeHomePaths` already handles the `C:\Users\…` backslash form — `sanitize.mjs` line 25, `[\\\\/]`). Both surfaces fixed by one change (`cmk remember` + `mk_remember` share `rememberRich`). Live-re-tested via the repo bin: `cmk remember "venv at C:\Users\<u>\proj\.venv"` → filename `project_venv-at-proj-venv.md`, username absent from filename/INDEX/body, body abstracted to `~`. Regression test added (`cli-private-tag-write-path.test.js`), proven to fail without the fix (stash-revert-rerun = red) and pass with it.

**Skill-review widened it to THREE sites (the real scope).** The `cmk remember` fix alone would have been half a fix: the code-review-excellence pass found the **same slug-leak in `auto-extract.mjs`** (B1 — and *higher-stakes*: it runs every turn automatically, no `cmk remember`, so a home path in Haiku's candidate title leaks the username into a committed filename with zero user action — its comment even wrongly claimed "writeFact already sanitizes") and in **`import-claude-md.mjs`** (I1 — a CLAUDE.md rule with a `C:\Users\you\` path). Resolution: instead of three inline copies, a **single central `sanitizeForTitle` helper** in `sanitize.mjs` (= `sanitizeHomePaths ∘ sanitizePrivacyTags`), used at all three slug-derivation sites — so a fourth caller can't re-miss the class. Each site has a regression test with a fixture self-guard + TDD-red-proof. (The lesson echoes the v0.3.1 `<private>` fix: when a sanitize-before-slug ordering bug is found at one site, AUDIT every `slugifyFact(...)` caller — the class travels.)

**A test-fixture trap worth recording (the meta-lesson):** my ad-hoc `node -e` / bash-heredoc probes while diagnosing this MANGLED the backslashes + `${}` interpolation (`C:\\Users\\${user}` collapsed to `C:Users${user}`), which made the fix look both broken AND vacuously-passing at different moments — nearly causing me to "fix" a non-problem or distrust a working fix. The resolution was (a) a **fixture self-guard** (`expect(text).toContain('\\Users\\<user>\\')`) so the test proves it actually contains the raw username, and (b) the **stash-revert-rerun** to prove red-without-fix / green-with-fix unambiguously. Shell heredocs are not a reliable way to construct backslash-bearing test input; assert the fixture, and prove the red.
