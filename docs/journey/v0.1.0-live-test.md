## Findings

### Finding 1 (NOT a bug — scope is correct; corrected 2026-05-29 after the user's review)

**Initial (wrong) framing**: I flagged this as a "doc gap — README over-promises search over sessions+transcripts." the user pushed back: *"the docs are correct, maybe we referred it to a later version?"* He was right on both points. I'd cited the README from memory of its **old** (v0.0.1) text instead of checking the current file — the "don't summarize from memory when the primary source is available" trap.

**Corrected, verified facts:**

- `cmk search` indexes `<tier>/MEMORY.md` + `<tier>/memory/*.md` (granular archive) across P/L/U tiers. It deliberately excludes the rolling-window session files (`today-*.md`, `recent.md`, `archive.md`) — `index-rebuild.mjs` line 13: `Caller-skipped: today-{date}.md`.
- Observed: facts auto-extracted to MEMORY.md (`test:ci`, the 30s/504 rule) are searchable; facts that only existed in the session-compression layer (PostgreSQL, Drizzle, receipts — never auto-extracted to MEMORY.md in this test) are not. **Consistent, correct behavior.**
- The **current README does NOT claim** session/transcript search — verified by grep, no such phrasing. (The old v0.0.1 README did; Task 41's rewrite removed it.)
- The fuller vision — search "across project memory, user memory, **sessions, and transcripts**" — lives in **requirements.md US-9**, and it's tied to **Layer 5b semantic search**, which is **explicitly deferred to v0.1.x** (ADR-0008 + design §16). So searching sessions/transcripts is a correctly-deferred later-version feature, NOT a v0.1.0 claim and NOT a doc bug.

**Net**: no action needed for v0.1.0. v0.1.0 keyword-search-over-durable-memory is correct and correctly scoped. The session/transcript search arrives with Layer 5b (v0.1.x/v0.2) per US-9. Nothing over-promises.

**Meta-lesson** (worth keeping): this is a live instance of the "did you check?" / "don't summarize from memory" discipline. I asserted a README claim from stale memory; the primary source (current README) disproved it. Always read the current file.

### Finding 2 (observation, not a bug) — compress-session is project-scoped, not persona-scoped

In the seeded session, the user said "I always prefer terse responses, skip explanations." That preference did **not** appear in `today-*.md` — because `compress-session` extracts project session state (Decisions / Open Questions / Files / Active Threads), not user-persona facts. The persona fact is correctly the domain of (a) auto-extract → USER.md when stated in a turn, or (b) the deferred Task 45 auto-persona. So its absence from the session compression is correct separation of concerns, but worth knowing: **a preference mentioned only in passing during a session, never in a captured turn, can fall through** until auto-persona (Task 45) ships. The auto-extract test (separate turn) DID capture the `test:ci` preference correctly, so the path works when the turn is processed.

## Harness notes (not kit issues)

- Driving the hook-only modules (`runAutoExtract`, `injectContext`) required `node` against the global install's `src/` with `file://` URLs (Windows ESM) — the published package doesn't expose these as `cmk` verbs because they're plugin-hook-driven. A real user gets them automatically via the Claude Code plugin's hooks; this test invoked them directly to exercise the same code.
- Git-Bash `/tmp` ≠ Windows-`node` `C:\tmp` bit me once on the turn-file path; fixed by using a project-local absolute path. Pure test-harness artifact.
