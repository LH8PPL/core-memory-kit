## 2026-06-04 — AUTOMATED live-test harness (`npm run live-test`) — the manual guide, scripted

Built [`scripts/live-test.mjs`](../../scripts/live-test.mjs): the scripted equivalent of the manual guide. It installs the **real packaged tarball** into an isolated global prefix (never touches the user's global `cmk`), then drives `claude -p` through the whole flow and asserts **files AND conversation outputs at every stage** — install + `cmk doctor` 0-fail + reads every scaffolded file; SESSION-1 staged build where the agent **actually builds** (per-stage artifact checks); explicit `cmk` probes (terse / rich Why-How / Poison_Guard reject + no-write / home-path sanitize); skill-path safety via **tool-use inspection** (no hand-edit of memory, no leak, audited); SESSION-2 recall (tool-use proves memory vs code-read); SESSION-3 cold-open (reads the built files). Raw per-run dumps → [`live-test-runs/`](live-test-runs/), one TIMESTAMPED file per run (tracked; a run-to-run trail to spot drift/regressions — e.g. the recall probe's variance is only visible across runs).

**Result:** the shipped, deterministic capabilities are **green every run** — real-artifact install, **Task 69's safe skill scaffolds + routes through `cmk` (no hand-edit, no leak)**, slim CLAUDE.md, capture B1–B4 incl. the stated rule promoting at `trust:high`, C1–C4 (incl. C3 reject-and-write-nowhere + C4 home-path→`~`), and the **cold-open wedge**. Best definitive run: **29/0**.

**Two real findings the automation surfaced** (the value of running it repeatably):

1. **In-project recall is variable (~50/50) — the known Task-75 / D-35 active-recall gap, not a regression.** Asked "what are my standing rules" in a fresh session, one run answered entirely from the injected persona (**0 tool calls**), the next **globbed the project code** instead. So D1/D2 are classified as **non-gating recall probes** (reported, but they don't fail the suite — the gap is tracked as Task 75, deferred to v0.3). Until found, the agent "described" builds instead of doing them, so early runs showed "scaffold-only, no source." Verified via probe.

## DECISION PENDING — Layer 5b semantic-recall backend (do the bake-off, THEN pick)

The video that started the kit ("Master Claude Memory…", Simon Scrapes) names **RECALL as the most important function + the biggest Claude-native gap**, solved by semantic search over complete capture. We deferred exactly this (Layer 5b). It's the value-defining piece — build it after F1/F2, with the RIGHT backend. **Backend is UNDECIDED — run a bake-off first, do NOT just inherit the video's pick.**

**Candidates (lightest → heaviest):**
1. **`sqlite-vec`** — vector similarity *inside the kit's existing SQLite/FTS5 index*; no server, hybrid in one place (sqlite-memory pattern). **Best architectural fit** (we already run SQLite). Needs a local embedding model (sentence-transformers / small GGUF). NB: **Anthropic has NO embeddings API** — can't use Claude for vectors.
2. **`qmd`** (Tobias Lütke, MIT) — Node, markdown-native, **MCP-native**, GGUF embeddinggemma, zero-server. What the user already runs in liorwiki for pure semantic. Caveat: file/chunk granularity ≠ our per-bullet granularity; ~1.6 GB model.
3. **Chroma** — **liorwiki chose this** for *filtered* semantic (pure-Python, `pip install chromadb`, no server, metadata filtering, all-mpnet-base-v2 768-dim). But it's Python (kit is Node) + a separate index. NB: our FTS5 already does the metadata filtering Chroma adds — so we need *less* than the wiki did (mostly pure semantic).
4. **memsearch + Milvus** — the video's pick + our original Layer-5b design (§9.3.1). Heaviest (Milvus server/Docker). liorwiki + our own node-only/no-server ethos (Task 62) argue it's the **wrong weight class** for a single-user local kit.

**The bake-off (the "tests/checks" the user wants):** ~30–50 memory-shaped facts + ~10 queries incl. **synonym/paraphrase** queries where keyword *should* miss and semantic *should* hit (e.g. "package manager preference" → finds a pnpm fact). Index into FTS5 + each candidate; compare **recall@5 + latency + install weight**. The kit already exposes the `semanticBackend` DI seam + `reciprocalRankFusion` in [search.mjs](../../packages/cli/src/search.mjs), so the winner drops in. Harness FTS5 baseline buildable now; qmd/Chroma legs need model downloads on the dev machine (or run in liorwiki where qmd+Chroma are already indexed).

**Recommendation (pre-test, to be confirmed by the bake-off):** lean **sqlite-vec** (stays in our DB, node-only ethos, no server) OR **qmd** (already proven in liorwiki, MCP-native) over Chroma (Python dep) or Milvus (server). **Decide after the bake-off, not before.** Tracked: design §9.3.1, DECISION-LOG (2026-05-31 search thread), backlog (post-F1/F2, post-friend-validation).
