## DECISION PENDING — Layer 5b semantic-recall backend (do the bake-off, THEN pick)

The video that started the kit ("Master Claude Memory…", Simon Scrapes) names **RECALL as the most important function + the biggest Claude-native gap**, solved by semantic search over complete capture. We deferred exactly this (Layer 5b). It's the value-defining piece — build it after F1/F2, with the RIGHT backend. **Backend is UNDECIDED — run a bake-off first, do NOT just inherit the video's pick.**

**Candidates (lightest → heaviest):**
1. **`sqlite-vec`** — vector similarity *inside the kit's existing SQLite/FTS5 index*; no server, hybrid in one place (sqlite-memory pattern). **Best architectural fit** (we already run SQLite). Needs a local embedding model (sentence-transformers / small GGUF). NB: **Anthropic has NO embeddings API** — can't use Claude for vectors.
2. **`qmd`** (Tobias Lütke, MIT) — Node, markdown-native, **MCP-native**, GGUF embeddinggemma, zero-server. What Lior already runs in liorwiki for pure semantic. Caveat: file/chunk granularity ≠ our per-bullet granularity; ~1.6 GB model.
3. **Chroma** — **liorwiki chose this** for *filtered* semantic (pure-Python, `pip install chromadb`, no server, metadata filtering, all-mpnet-base-v2 768-dim). But it's Python (kit is Node) + a separate index. NB: our FTS5 already does the metadata filtering Chroma adds — so we need *less* than the wiki did (mostly pure semantic).
4. **memsearch + Milvus** — the video's pick + our original Layer-5b design (§9.3.1). Heaviest (Milvus server/Docker). liorwiki + our own node-only/no-server ethos (Task 62) argue it's the **wrong weight class** for a single-user local kit.

**The bake-off (the "tests/checks" Lior wants):** ~30–50 memory-shaped facts + ~10 queries incl. **synonym/paraphrase** queries where keyword *should* miss and semantic *should* hit (e.g. "package manager preference" → finds a pnpm fact). Index into FTS5 + each candidate; compare **recall@5 + latency + install weight**. The kit already exposes the `semanticBackend` DI seam + `reciprocalRankFusion` in [search.mjs](../../packages/cli/src/search.mjs), so the winner drops in. Harness FTS5 baseline buildable now; qmd/Chroma legs need model downloads on the dev machine (or run in liorwiki where qmd+Chroma are already indexed).

**Recommendation (pre-test, to be confirmed by the bake-off):** lean **sqlite-vec** (stays in our DB, node-only ethos, no server) OR **qmd** (already proven in liorwiki, MCP-native) over Chroma (Python dep) or Milvus (server). **Decide after the bake-off, not before.** Tracked: design §9.3.1, DECISION-LOG (2026-05-31 search thread), backlog (post-F1/F2, post-friend-validation).
