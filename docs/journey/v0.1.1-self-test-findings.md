## v0.2 headline capability — "Claude remembers its own positions and stays consistent" (the user, 2026-05-30)

**The scenario the user wants:** in a new session — *"2 days ago we discussed FastAPI, you said doing it that way was a mistake; now you're saying the opposite."* Claude should **know it, own it, and reconcile** (explain the change or self-correct). This is the concrete form of his "what about the AI side of the conversation?" point, and it's arguably the kit's most compelling value prop.

**Honest current status (do NOT overstate to users):**
- ✅ Raw material exists — `context/transcripts/{date}.md` captures BOTH user and assistant turns with timestamps (capture-prompt + capture-turn). Claude's "that's a mistake" recommendation IS on disk.
- ❌ Not recall-able — transcripts are NOT injected at session start and NOT in the search index (`cmk search` indexes facts/scratchpads only). So today Claude can't reliably retrieve what it told you 2 days ago.
- ❌ Claude's stated positions/recommendations are NOT captured as first-class, timestamped, contradiction-checkable facts (only user prefs + project facts are). Nothing for the contradiction engine to catch on reversal.

**Three pieces to deliver it (v0.2):**
1. Capture Claude's stated positions ("recommended X", "advised against Y") as timestamped decision-facts (AI-side capture, beyond user prefs).
2. Make them recall-able — index transcripts + a decision-history, and/or inject a compact "recent decisions" digest at SessionStart.
3. Contradiction-awareness — prior position in context so Claude notices + reconciles a reversal. Reuses the **optimistic auto-promote + auto-supersede-on-contradiction** engine (the user chose "Optimistic" 2026-05-30).

**Relationship:** subsumes/extends Task 55 (behavioral patterns); builds on the v0.1.x sweep (#R recall, write-path via `cmk remember`, auto-drain the review queue). Sweep makes *facts* recall; this makes *the dialogue + Claude's own positions* recall. **Scope: v0.2.** Capture in design §16 + a Task when the sweep lands.

**Related decision (2026-05-30): auto-drain the review queue (no manual `cmk queue review`).** The user: *"i dont want to do anything, i want it to be automatic."* AI-side facts must flow automatically. Chosen posture: **Optimistic** — auto-promote AI-inferred facts; auto-supersede when a contradicting fact arrives (reuses Task 25 conflict detection); the automatic daily-distill/weekly-curate passes drain the queue. No manual promotion step, ever. This is a v0.1.x behavior change (part of the write-path/auto-memory family) feeding the v0.2 capability above.

## Findings

### 🔴 #0 — `cmk search` returns "no results" despite facts existing (index never built on fresh install)

Surfaced 2026-05-30. All four Checkpoint searches (`layered`, `port 8000`, `type hints`, `thin routes`) returned `cmk search: no results` — but `type context\MEMORY.md` showed the facts present, including the **literal** string "thin routes" in bullet P-WGQAZFVC.

**Root cause (verified in code):** the `cmk search` handler ([subcommands.mjs:182-184](../../packages/cli/src/subcommands.mjs#L182-L184)) does `openIndexDb()` → `searchAction()` with **no reindex first**. The FTS5 index is populated only by (a) `reindexBoot`/`reindexFull` — never called by `search` — or (b) the chokidar runtime watcher, which only runs inside a long-lived process not active during a one-shot CLI call. Auto-extract writes facts to `MEMORY.md` + fact files but **does not trigger a reindex**. So on a fresh install the index is empty and every `cmk search` says "no results." The indexer *does* cover MEMORY.md scratchpad bullets ([index-rebuild.mjs:72-75](../../packages/cli/src/index-rebuild.mjs#L72-L75) `kind: 'scratchpad'`), so once reindexed the facts WILL be findable.

**Severity 🔴:** search is a core feature, and silent "no results" reads as "nothing was captured" — actively misleading (capture was actually working). Masks success as failure.

**Fix:** `cmk search` should call `reindexBoot` (cheap mtime/sha1 diff per design §9.2) before querying — OR auto-extract/capture-turn should reindex after writing. The boot-reindex-on-search option is simplest + idempotent; design §9.2 already names boot-reindex as a strategy, it's just not wired into the search command.

**Confirm:** `cmk reindex --full` then re-run the searches → should now hit. (Pending the user's confirmation.)

**Does NOT affect recall:** SessionStart injects `MEMORY.md` directly (not via FTS), so the Session-2 recall test is independent of this bug.

**Confirmed 2026-05-30:** `cmk reindex --full` → `cmk search "thin routes"` now returns the MEMORY.md bullet. Diagnosis locked. **But reindex surfaced #0b ⬇.**

### 🔴 #0b — Inline-written `feedback_*.md` fact files use the wrong frontmatter schema → rejected by the kit's own indexer

`cmk reindex --full` skipped **all three** inline-written fact files:
```
skipped feedback_code-style-types-comments.md: invalid or missing id
skipped feedback_layered-backend-architecture.md: invalid or missing id
skipped feedback_python-venv-313-only.md: invalid or missing id
```
Their frontmatter is `name: / description: / metadata: {type:}` — the **generic Claude-Code auto-memory format** (the one in the harness/system-prompt memory convention) — but the kit's indexer requires top-level `id / type / title`. So the main agent, hand-writing fact files via the `Write` tool following the scaffolded CLAUDE.md, **produced files the kit's own index cannot ingest.** Only the auto-extract→`MEMORY.md` path (provenance bullets) indexed.

**Root cause:** the kit relies on the agent freehand-writing fact files in the kit schema, but the agent falls back to the generic auto-memory format it knows from its system prompt. **The robust fix is to stop relying on freehand writes:** the scaffolded CLAUDE.md / memory-write guidance should route durable captures through `cmk remember` (programmatic — assigns a valid `id`, writes the correct schema + provenance) rather than a raw `Write`. This ALSO would fix #1 (machine-path routing) and #2 (tier) and #3 (dedup) at once, since a single programmatic entry point can enforce all of them. **This is likely the highest-leverage fix on the list** — the freehand-write path is the common cause behind #0b/#1/#2/#3.

### 🔴 #R — Recall: new session re-derived everything from the code, did NOT visibly use injected memory (NEEDS confirmation)

The Session-2 recall test (the product's whole thesis). Asked *"What port are we on, how is this project structured, and what's our backend?"* in a fresh session, the agent responded *"I'll explore the project to answer those three questions"* and ran `ls -la` / `Glob` / `Read config.py`. It produced correct answers — **by reading the codebase, not by recalling the injected snapshot.** Likewise `/health` landed in `api/views.py` correctly, but right after the agent *read* views.py — not demonstrably from the thin-route preference.

**This suggests SessionStart injection (cmk-inject-context) did not fire or did not surface to the agent.** Not yet conclusive — the agent *could* have had the snapshot and chosen to verify against code. **Pending diagnostic:** ask the fresh session *"Without reading any files or running any tools, what do you already know about this project and how I like to work?"* — answerable ⇒ injection worked; "I don't have prior context" ⇒ injection failed (🔴, core mechanism).

If injection failed, candidate causes to check: did the SessionStart hook fire in the new session? does `cmk-inject-context` emit the snapshot in the shape Claude Code injects (additionalContext)? is it erroring silently (spawn/path)? Check any inject log under `context/.locks/`.

**Update 2026-05-30 — snapshot generation PROVEN WORKING; bug isolated to the firing/ingestion handoff.** Ran the installed hook directly against the test project (`'{}' | cmk-inject-context` with cwd = `C:\Temp\live-test-1`). It returned a **fully-populated** `hookSpecificOutput.additionalContext` containing all three tiers AND the 4 captured MEMORY.md bullets verbatim (venv/3.13, no-3.10, layered-doctrine, name-the user). So:
- ✅ **Snapshot build works** — the hardest part. `injectContext()` + the bin emit the correct Anthropic SessionStart shape with real content.
- 🔴 **The snapshot is not reaching the agent's context at session start.** In the recall session the agent surfaced none of those 4 bullets; its "about you" knowledge traced entirely to global `~/.claude/CLAUDE.md` + the scaffolded project CLAUDE.md.

**Failure is one of (A/B/C):**
- **(A) hook not firing** when the VS Code extension starts a new chat via "+" (vs. a fresh `claude` CLI launch). Plausible — needs confirming SessionStart fires on "+".
- **(B) cwd mismatch** — the bin ignores the stdin payload's `cwd` and uses `process.cwd()` ([cmk-inject-context.mjs:53](../../packages/cli/bin/cmk-inject-context.mjs#L53)). If Claude Code spawns the hook with a different cwd, `injectContext` reads an empty/wrong project → empty snapshot. **Robustness fix regardless: parse the stdin payload and use `payload.cwd`/workspace root, not `process.cwd()`.**
- **(C) Claude Code not ingesting** `additionalContext` from a settings.json (non-plugin) SessionStart hook. Needs primary-source verification against the hooks docs (don't assume — this is the plugin-layout-bug class).

**Decisive disambiguator (pending):** fresh session, FIRST message: *"Without reading any files: do I have a rule about which Python version to use, and how do I structure FastAPI backends?"* — answered ⇒ injection works (earlier was a non-contaminated false alarm); "I don't know" ⇒ injection broken, chase A/B/C. **This is the single most important open question in the whole self-test** — it's the read-half of the kit's core loop.

**RESOLVED 2026-05-30 — ROOT CAUSE FOUND. The injected snapshot is 70% template noise; real facts are buried/truncated.** Clean fresh-session test: agent said *"a local-tier snapshot that's all placeholder examples (node binary at /usr/local/bin/node, etc.) — no real facts populated yet."* So injection **fires and Claude Code DOES ingest** `additionalContext` (rules out A and C) — but only the *front* of the snapshot (local-tier placeholders) reached the agent; the project-tier MEMORY.md real facts did not surface.

Measured the actual injected `additionalContext` (`'{}' | cmk-inject-context` in the test project):
| Metric | Value |
| --- | --- |
| Total bytes | 11,288 |
| Bytes inside `<!-- -->` template/comment blocks | **7,921 (70%)** |
| `(example)` placeholder seed bullets injected | 6 |
| Local tier (placeholders) offset | byte 10 (FIRST) |
| Real MEMORY.md venv fact offset | byte 5,244 (~halfway) |
| User tier offset | byte 6,830 |

**Root cause:** `inject-context.mjs` injects **raw file contents** — every file's multi-line `<!-- format explanation -->` header + every `(example)` seed bullet — ordered placeholder-heavy local tier FIRST, real MEMORY.md facts buried at byte 5,244. The agent hit a wall of scaffolding and concluded "no real facts." (Snapshot is 11,288 < the kit's 13,000 cap, so the kit didn't drop tiers; the loss is noise-burial and/or a Claude-Code-side additionalContext truncation that kept only the front placeholder slice.)

**Fix (all in `inject-context.mjs` snapshot assembly) — highest-impact fix for the friend:**
1. Strip the `<!-- … -->` format-explanation headers from the *injected* snapshot (keep them in the on-disk files for human editors). Kills ~70% of payload.
2. Skip placeholder/seed bullets (`(example)`, 2020-01-01 dates) — they mislead the agent into "nothing real here."
3. Reorder: lead with real facts (populated MEMORY.md + real user-tier), put empty/placeholder tiers last or omit. After stripping, snapshot drops ~11KB → ~1.5KB of pure real facts.
4. (Robustness) parse stdin payload `cwd` instead of `process.cwd()` per #R(B).

**Verify after fix:** re-run the fresh-session diagnostic; agent should immediately cite Python-3.13-only + layered-FastAPI without reading files.

### 🔴 #1 — Machine-specific content written into the *committed* project tier (privacy + portability + tiering)

**Three problems in one:**
- **Privacy** — the username `tamir.bn-sh` (+ machine config) ships in git. Same class as the pre-public scrub we just did.
- **Portability** — absolute paths are wrong on any other machine; violates Tenet T2.

**Root question / fix direction:** the routing guidance (scaffolded CLAUDE.md / memory-write skill / auto-extract) needs to either (a) **detect absolute home paths** in a captured body and redact/abstract them, (b) **route machine-specific facts to `context.local/`**, or both. Likely a poison-guard-style guard: a path-detector that re-routes or redacts `C:\Users\<name>\…` / `/home/<name>/…` / `/Users/<name>/…`. **High priority — privacy regression in the exact area we just hardened.**

  > `(P-EPJF7SXX) User's Python setup has version mismatch: python=3.13.5 but pip installed to Python 3.10; workaround is to reinstall deps into 3.13 or use explicit interpreter path` — `proposed_trust: medium`

**🟡 Gap A — auto-extract captures intermediate reasoning, not final state.** The queued entry reflects turn-1's understanding, which the session itself superseded. By the time you `cmk queue review`, that entry is *factually stale* — you'd discard it, but only because you happen to know the later correction. ### spawn — `shell:true` + args array breaks Windows paths with spaces (DEP0190)

Surfaced 2026-05-30 via a `DEP0190` DeprecationWarning on `cmk doctor`.
- Cosmetic at [doctor.mjs:65](../../packages/cli/src/doctor.mjs#L65) (`memsearch --version` — constant args, just the warning).
- **Real bug** at [compressor.mjs:213](../../packages/cli/src/compressor.mjs#L213): `--mcp-config <mcpConfigPath>` where the path is under `tmpdir()` (`C:\Users\<username>\…`). `shell:true` concatenates args unescaped → cmd.exe re-tokenizes on the space in a `First Last` username → `--mcp-config` gets a truncated path → auto-extract/compression **silently fails** for any Windows user whose profile dir contains a space. Did not surface in self-test (`tamir.bn-sh` has no space).
- **Fix:** resolve the bin's real `.cmd`/`.exe` via PATHEXT and spawn it **without** `shell:true` (eliminates DEP0190 *and* the unquoted-arg breakage). Shared spawn helper + `validate-spawn-discipline.mjs` (already on the PR-D2 roadmap). TDD + stress 5/5 (spawn surface).
- **Doors:** Door 3 (external calls — argv shape); the fix's test must assert the spawned argv keeps the path as a single token.

### UX — `cmk install` over-shares bookkeeping at info level

"scaffolded 5 file(s), skipped 4 existing" reads like a problem on a *fresh* folder — the "4 skipped" are the **user tier** at `~/.claude-memory-kit/` (cross-project, outside the folder, already on disk from a prior install), not files in the new folder. Debug-level inventory leaking into info-level output.
- **Principle:** outcome over inventory (Unix "silence is golden" / clig.dev / RFC 5424 levels).
- **Fix (print layer only — `subcommands.mjs`; `install()` already returns `created`/`skipped`):** first run → `✓ cmk install: <folder> ready — context/ scaffolded, hooks wired. Restart Claude Code to activate.` (no counts); re-run → `context/ already present (your edits preserved), hooks refreshed.`; `--verbose` → full per-tier breakdown.
- DEP0190 does *not* belong in the info/warning/error model — it's our own internal spawn misuse leaking through Node. Fixing the spawn bug removes it; don't demote it to debug.
