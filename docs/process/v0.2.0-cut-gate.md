The final human gate before publishing v0.2.0 (D-24/D-25). The **wedge** (your
cross-project persona + project memory, portable, so a new project cold-opens
already knowing how you work) is the differentiator; this run confirms it *feels*
real on a fresh machine-state, and checks the three things automation can't.

[`../journey/live-test-runs/`](../journey/live-test-runs/). What you add by hand:

## 2. Session 1 — build it, stating preferences (watch R1 + R2)

Build a small real thing across **one** Claude Code session. Each stage has a
**Build** prompt and a **Say it out loud** preference (a real engineering opinion,
stated naturally — **never** "remember this"). End each turn normally (the Stop
hook fires auto-extract each time). Don't start a new session between stages.

**Stage 0 — baseline.**
- *Build:* "Create a minimal Python web chat UI: a FastAPI server with a WebSocket
  endpoint and a single static `index.html`. Plain HTML/JS, no framework. Put the
  server in `app.py`." → reply "yes, run it" if it offers to start the server.

**Stage 1 — refactor to layers (the big one).**
- *Build:* "Refactor this into a layered FastAPI project — `app/{api,services,repositories,schemas,core}/`
  and `app/main.py`. WebSocket route into `api/`, connection/broadcast logic into a
  service, Pydantic message schemas. Keep it on port 8000."
- *Say:* "How I build backends: FastAPI is the delivery layer, not the brain. Routes
  stay thin and orchestrate; business logic lives in services; data access in boring
  repositories; Pydantic schemas are the boundary contracts. I want this even for a
  small app — I'd rather pay the structure cost now than fight it in six months."

**Stage 2 — swap to Claude + typing rule.**
- *Build:* "Change it to a single-user chat with Claude via the Claude Agent SDK
  (`claude-agent-sdk`) — a `ClaudeAgentService` wrapping `ClaudeSDKClient`, each
  WebSocket connection its own session."
- *Say:* "Type hints on every function signature — I'm on Python 3.12+. Comments
  explain why, not what. And tests first: boundary test, watch it fail, then implement."

**Stage 3 — stream + async rule + the universal rule (B4).**
- *Build:* "Stream Claude's output to the browser as it arrives — push JSON frames over
  the WebSocket; the client appends to the live bubble."
- *Say:* "Async all the way down — nothing blocking in the event loop." **Then state
  one cross-project rule** (this is the wedge's auto half): "From now on, in **every**
  project I work on, always use `uv` for packages, never `pip`, and always run `ruff`
  before committing."

- [ ] **R1 — no console flash (Task 81).** At every **SessionStart**, NO black `node`
  window flashes to the foreground. (Visual — only you can see this.)
- [ ] **R2 — no `cmk` permission prompt (Task 79).** When the agent runs an explicit
  `cmk remember` / `cmk lessons promote`, Claude Code does **not** ask "Allow this
  bash command?" (the install allow-lists `Bash(cmk:*)`).

**The Task-69 explicit-skill check (the security point of this release):**
- [ ] **G5** Mid-build, say: *"remember this: my local cache is at C:\Users\<you>\cache\app."*
  Then verify the agent **ran `cmk remember`** (not a hand-edit), and open
  `context\MEMORY.md` / the fact file — it shows `~\cache\app`, **never your username**.
  (If it hand-edited a memory file, or your username leaked, that's a blocker.)

**Capture checks (read the files):**
- [ ] **G6** `cmk search "layered"` / `type context\MEMORY.md` → your stated decisions
  captured **without** "remember this".
- [ ] **G7 — the wedge fills:** `type %USERPROFILE%\.claude-memory-kit\HABITS.md` (and
  USER/LESSONS) → your cross-project style is there; a **stated** universal rule
  (e.g. "from now on, always use uv") landed at **`trust: high`** on its own.
