# v0.1.1 self-test — step-by-step walkthrough

A literal, copy-along guide to test the kit by **building a small real thing across two sessions**, so the memory features get exercised *organically*. The build vehicle **follows the stages of [*Build Your Own Claude Code Web UI in 280 Lines of Python*](https://generativeai.pub/build-your-own-claude-code-web-ui-in-280-lines-of-python-7658422a8464)** (Jes Fink-Jensen) — a two-pane streaming chat UI with FastAPI + WebSockets + plain HTML/JS — **with two deliberate twists**:

1. **Backend = Claude (Claude Agent SDK), not Ollama.** The article wraps a local Ollama model; we swap in the [Claude Agent SDK](https://code.claude.com/docs/en/agent-sdk/python) (`claude-agent-sdk`) so it streams from Claude instead. You'll learn the article's technique while using a backend you already have.
2. **Architecture = your layered FastAPI style, not single-file.** The article keeps everything in one `main.py` (its whole point is "280 lines, read in one sitting"). You build it the way *you* build backends — layered (`api/ services/ repositories/ schemas/ core/`), thin routes, logic in services — per your own [FastAPI Architecture](https://medium.com/algomart/fastapi-architecture-how-to-build-backends-that-dont-fight-you-six-months-later-e477c91be19f) notes. **This conflict is intentional**: voicing your standards against the article's defaults is exactly the durable-preference signal the kit should capture.

The point isn't to *finish* the app — it's to do real, opinionated work in session 1 (stating decisions + preferences out loud, never saying "remember this") and prove Claude **recalls it in session 2**. You don't have to do all 7 stages; **even stages 1–4 give a strong memory test** — stop whenever you've had enough.

**Prereqs:**

- Global `@lh8ppl/claude-memory-kit@0.1.1` installed (it is — `cmk --version` → `0.1.1`).
- Python 3.12+ on PATH. The kit will `pip install claude-agent-sdk fastapi uvicorn` (or `uv add` — your call) during the build.
- **Claude Agent SDK auth**: you're already logged into Claude Code, and the SDK uses the `claude` CLI transport by default — so it works out of the box. If it ever asks for a key, `set ANTHROPIC_API_KEY=...` (Windows) and retry.
- ~45–60 min for session 1 (fewer stages = less); ~15 min for session 2.

## SESSION 1 — install + build + capture

**a.** Create a fresh folder:
```bash
mkdir c:\temp\lior-test-1
```
**b.** Open VS Code in it: `File → Open Folder… → c:\temp\lior-test-1` (or `code c:\temp\lior-test-1`).

**c.** Open a terminal in VS Code and install the kit into this project:
```bash
cmk install
```
Expect: `scaffolded … files, … hooks=wired`. Then run `cmk doctor` — HC-2 should PASS.

**d.** **No need to close VS Code.** Claude Code's file watcher picks up the hooks `cmk install` just wrote to `.claude/settings.json` automatically ([per the hooks docs](https://code.claude.com/docs/en/hooks): *"Direct edits to hooks in settings files are normally picked up automatically by the file watcher"*). The event hooks (`UserPromptSubmit`, `PostToolUse`, `Stop`) activate in your **current** session — just keep going. **Verify** they loaded by typing `/hooks` in the panel (read-only list; you should see the 5 `cmk-*` hooks). The only hook that needs a fresh session is `SessionStart` (memory injection) — and that's tested in Session 2 below, which is a new session anyway.

**e. Continue from what you already built.**

**Stage 0 — the first prompt (already sent).** The exact prompt that kicked off the build, kept here for the research trail:
- **Build (sent):** *"Create a minimal Python web chat UI: a FastAPI server with a WebSocket endpoint and a single static index.html. Use FastAPI + uvicorn, no frontend framework — plain HTML/JS. Put the server in `app.py`."*
- **What Claude produced:** a **single-file broadcast chat** — `app.py` (FastAPI + a `ConnectionManager` that broadcasts each message to all connected clients), `static/index.html` (plain HTML/JS), `requirements.txt`. Port 8000, host 127.0.0.1. Human-to-human chat room, no AI backend, everything in one file. *(Note: no preferences were stated in this prompt, so whatever the kit captures from Stage 0 is pure project-fact extraction — FastAPI, WebSocket, single `app.py`, port 8000.)*

The rest of the test is **redirecting + refactoring** this toward the article's real shape — and "we built X, now change it to Y because I prefer Z" is the richest thing the kit can capture. Each stage below has a **Build** prompt and a **Say it out loud** preference (a real engineering opinion — state it naturally, *never* "remember this"). Spread them out; mix in your own.

> **Right now**, Claude is asking *"Want me to install the deps and start the server to verify it works?"* — reply **"yes, run it"** (seeing the broadcast chat work in two browser tabs is a satisfying baseline), **then** start Stage 1.

**Stage 1 — set your architecture standards + refactor to layers.** *(This is the big one — it openly overrides the single-file shape Claude just produced.)*
- **Build:** *"Before we go further: refactor this into a layered FastAPI project — `app/{api,services,repositories,schemas,core}/` + `app/main.py`. Move the WebSocket route into `api/`, the connection/broadcast logic into a service, and add Pydantic message schemas. Keep it running on port 8000."*
- **Say it out loud:** *"How I build backends: FastAPI is the delivery layer, not the brain. Routes stay thin and just orchestrate; business logic lives in services; data access in boring repositories; Pydantic schemas are the boundary contracts; wiring via `Depends`. I want this even for a small app — I'd rather pay the structure cost now than fight it six months from now."*

**Stage 2 — swap the backend to Claude (the article's whole point).**
- **Build:** *"Now change it from a broadcast room to a single-user **chat with Claude**, using the **Claude Agent SDK** (`claude-agent-sdk`). Add a `ClaudeAgentService` in `services/` that wraps `ClaudeSDKClient` with `include_partial_messages=True`. Each WebSocket connection gets its own session with its own history."*
- **Say it out loud:** *"Type hints on every function signature — I'm on Python 3.12+. Comments explain *why*, not *what*; drop the ones that just narrate the code."*

**📍 Checkpoint A (after stage 2)** — run the capture check in box **f**. You should already see your **layered-architecture**, **Claude-Agent-SDK**, and **type-hint** preferences captured.

**Stage 3 — stream it token-by-token.**
- **Build:** *"Stream Claude's output to the browser as it arrives. On the server, iterate the SDK's `receive_response()` — handle `AssistantMessage`/`TextBlock` for complete blocks and `StreamEvent` (the `content_block_delta` → `text_delta`) for partial tokens — and push JSON frames over the WebSocket. On the client, append to the live message bubble as frames arrive."*
- **Say it out loud:** *"Async all the way down — nothing blocking in the event loop. If something blocks (a sync call, heavy CPU), offload it with `asyncio.to_thread` so one slow turn can't freeze other connections."*

**📍 Checkpoint B (after stage 3)** — you now have a working streaming Claude chat UI, layered your way. Re-run box **f**; the async/streaming preference should appear too.

**Stages 4–6 (optional, if you want the full article experience).**
- **Stage 4 — sessions:** *"Persist + list sessions via a `repositories/` layer (JSON history files); add a sidebar session switcher. Repositories stay boring — read/write only, zero business logic."* → **Say:** *"Persistence is a repository concern. I never let storage details leak into services or routes."*
- **Stage 5 — persona picker:** *"Replace the article's Ollama 'skills' with Claude Agent SDK **system prompts** — a sidebar picker that switches the agent's persona instantly, without a model round-trip."* → **Say:** *"Configuration over conversation — if the UI can set state directly, don't make the user ask the model to do it."*
- **Stage 6 — context + commands:** *"Add slash-commands-as-buttons and a context/usage readout from the SDK's `ResultMessage` (`usage`, `total_cost_usd`)."* → **Say:** *"Tooling I expect on a real project: `uv` for packages, `ruff` for lint+format, `pytest` for tests. Set those up if we go further."*

End the turn normally after each stage (the **Stop** hook fires auto-extract each time — more turns = more capture).

> **⚠️ Session boundary — read this.** Stages 0 through 6 all happen in **ONE single Claude Code session** (the one you've had open since `cmk install`). Do **not** start a new session between stages. You only start a new session **once**, *after* you've done as many stages as you want — that's step **g** below, which begins SESSION 2 (the recall test). Rule of thumb: **build in Session 1 → new session → ask-it-to-remember in Session 2.**

**f. Checkpoint — did it capture?** In the terminal (re-run at Checkpoints A and B):
```bash
cmk search "layered"
cmk search "port 8000"
cmk search "type hints"
cmk search "thin routes"
type context\MEMORY.md      # (Windows; or: cat context/MEMORY.md)
```
✅ **Expected:** your decisions + preferences — **Claude Agent SDK backend, layered FastAPI architecture, thin routes, port 8000, type hints, comments explain why, no-framework frontend, async-all-the-way, uv/ruff/pytest** — show up as facts in `context/MEMORY.md`, **without** you ever saying "remember this." The engineering-style ones (type hints, thin routes, async) are the high-value test: those are *durable user preferences*, not project trivia, so watch whether they get flagged for the user tier.

**g. ⬅️ THIS is when you start a new session.** You've finished building (did as many stages as you wanted, all in the one session). **Now** click the **new-chat / "+" button** in the Claude Code panel — that ends Session 1 and starts Session 2. (No need to close VS Code; the new chat is what fires the `SessionStart` memory-injection hook. Closing VS Code entirely also works and is closer to a real "next day," but isn't required.) **After this click, you are in Session 2 → do the recall test below.**

## SESSION 2 — the payoff (recall)

*You should have just clicked "+" for a new session (step g). You're now testing whether this fresh session remembers Session 1 — without you re-explaining anything.*

**a.** In the **same** folder (`c:\temp\lior-test-1`), in your fresh Claude Code session (the new chat from step g — or reopen the folder if you closed VS Code).

**b.** Without re-explaining anything, ask something that needs session-1 context:
- *"What port are we on, and how is this project structured?"* (does it recall **port 8000** + the **layered `api/services/repositories/schemas/core`** architecture, and that the backend is the **Claude Agent SDK**?)
- *"Add a `/health` endpoint."* (does it put a **thin route** in `api/`, **type-hinted**, comments-only-for-why, **without** being re-told your style?)
- *"Add a new persona option."* (does it use the SDK **system-prompt** approach from stage 6, not re-invent it?)

✅ **Expected (the whole product working):** Claude already knows the port, the layered architecture, the Claude-Agent-SDK backend, the thin-routes / type-hints / async preferences — because SessionStart injected the snapshot. **No re-briefing.** The strongest signal: it follows your *engineering style* on the `/health` endpoint without you restating it. That's scenarios #5 + #6.

If it *doesn't* remember → that's the #1 thing to report (copy what it said + paste `context/MEMORY.md`).
