CORRECTION to P-3F7L7JZ4 (stale auto-extracted mis-diagnosis): the Kiro capture-prompt WIRING IS NOT MISSING. dispatchKiroHook (kiro-hook-dispatch.mjs:46,71-85) already routes userPromptSubmit→capturePrompt since Task 50.N.1 (PR #228). The REAL root cause of the Kiro-IDE user-turn-not-captured bug is: env.USER_PROMPT (the documented Kiro IDE prompt-delivery mechanism, kiro-hook-bin.mjs:67-70) arrives EMPTY on Kiro IDE 1.0's v1 hook format — a Kiro platform bug (GitHub #9619 UserPromptSubmit-no-longer-fires-post-1.0, #6188/#7375 USER_PROMPT empty/wrong-shape) — and capture-prompt.mjs:80-82 silently no-ops on the empty prompt.

**Why:** A same-day auto-extracted fact (P-3F7L7JZ4, trust:medium) mis-diagnosed the Kiro user-turn bug as 'the capture-prompt hook is never invoked — the fix is to invoke it.' Two independent sonnet research agents (web + our own research base) both proved that WRONG: the dispatcher DOES call capturePrompt on userPromptSubmit/promptSubmit (PROMPT_CAPTURE_EVENTS set at kiro-hook-dispatch.mjs:46, called at :71-85, wired by Task 50.N.1 PR #228 shipped 2026-06-25). The stale fact likely came from reading only the IDE hook JSON's name/description ('recall') without tracing the dispatcher. The ACTUAL root cause, now authoritative: (1) env.USER_PROMPT is the DOCUMENTED Kiro IDE prompt mechanism (kiro.dev/docs/hooks/types/ 'the user prompt can be accessed via the USER_PROMPT environment variable') and the kit reads it correctly (kiro-hook-bin.mjs:67-70); (2) BUT it was only ever verified on the LEGACY 0.x .kiro.hook surface (probe P-CJYGTQYR 2026-06-21, pre-IDE-1.0) + an AWS article about the general pre-1.0 hook system + kiro-cli's stdin 'prompt' field — NEVER on Kiro IDE 1.0's v1 .json/UserPromptSubmit format; (3) on IDE 1.0 v1 it arrives EMPTY (Kiro's own open bugs #9619/#6188/#7375/#4620); (4) capture-prompt.mjs:80-82 silently no-ops (reason:'empty-prompt') with no log — the D-269 silent-empty-since-v0.4.0 class exactly. So the user turn never lands in the transcript/now.md → auto-extract's bi-turn extraction starves → nothing_durable → 0 write_source:auto-extract facts on Kiro IDE.

**How to apply:** THE FIX (do NOT re-wire the dispatcher — it's correct): work around Kiro's broken USER_PROMPT env var by sourcing the user prompt from the TRANSCRIPT, which is already proven to work on IDE 1.0. Concretely: (A) primary fix — in kiro-hook-bin.mjs:67-70, add a 3rd fallback after env.USER_PROMPT: if still empty, read the user turn from the transcript via readKiroTurn({projectRoot,env}).userText (the SAME reader wrappedCapture uses at line 52, which parseKiroIdeV1Messages proved returns the real userText from messages.jsonl payload.type==='user'). CAVEAT to verify: TIMING — does UserPromptSubmit fire BEFORE the prompt is written to messages.jsonl? If yes, the transcript-read at UserPromptSubmit time won't have it → then the better fix is (B): capture the user turn at the STOP hook instead, where wrappedCapture ALREADY reads turn.userText (line 52-56) and builds user_message — ensure captureTurn writes BOTH the user_message and assistant_message to the transcript/now.md as a bi-turn (check capture-turn.mjs handles user_message). Option B is likely more robust since the Stop-hook transcript definitely has the completed turn. (C) also add a diagnostic log at capture-prompt.mjs:80-82 empty no-op (don't ship silent — D-269 lesson). TEST: the current tests/cli-kiro-hook-bin.test.js only test non-empty USER_PROMPT happy path; add an empty-USER_PROMPT test that asserts the transcript fallback fires.
