Survey deep-read: Memoria (matrixorigin, Rust) DOES learn from failure at the CODE level — a YES, and oracle-free/transferable. The `memory_feedback` MCP tool accepts signals useful/irrelevant/outdated/wrong; a `wrong` signal increments a per-memory denormalized counter `feedback_wrong` in `mem_memories_stats` (storage/store.rs record_feedback ~L4216). At every hybrid search (search_hybrid_from_scored_scoped, store.rs ~L5720), the loaded per-memory feedback is applied: feedback_delta = useful - 0.5*(irrelevant+outdated+wrong); final_score *= (1.0 + feedback_weight * feedback_delta).clamp(0.5, 2.0). So a `wrong`/negative signal directly DOWN-RANKS that specific memory (toward the 0.5 floor) and re-sorts results. Unit-tested behaviorally: scoring.rs test_default_scoring_adjustment asserts "2 wrong -> 0.9x penalty" and test_scoring_clamp asserts extreme-negative clamps to 0.5x. There is ALSO a second-level meta-loop: memory_tune_params / AdaptiveTuner.tune_params adjusts per-user feedback_weight from accumulated feedback ratios (negative_ratio>0.5 -> feedback_weight*0.9, min 0.05). Signal is USER/AGENT-supplied relevance feedback — NO ground-truth oracle (no benchmark reward, no unit-test pass, no web final-state) — so it IS transferable to a session host. Governance quarantine_low_confidence is SEPARATE and does NOT key off `wrong` feedback (it's a confidence*time-decay < 0.2 DELETE); `wrong`'s effect flows through the rerank path, not pruning. Honest caveat: `confidence_weight` param is stored/clamped/tuned-display but I found no ranking-path READ of it (unlike feedback_weight which is read at store.rs:5727 + service.rs:1862) — partially-inert/reserved, but does NOT weaken the finding since the wrong->down-rank loop runs fully through feedback_weight. Updated portfolio: Memoria joins memclaw/ReasoningBank/MemRL as YES. Novel signal type vs known portfolio: a 4-valued explicit relevance signal (useful/irrelevant/outdated/wrong) that both (a) directly down-ranks the specific offending memory per-retrieval AND (b) meta-tunes the global feedback weight — the "outdated" value in particular is a staleness signal distinct from plain contradiction/user-correction.

**Why:** Fixes the untrustworthy-denominator problem in the U-Mem-driven learns-from-failure survey: Memoria was the strongest NEW candidate but code-unverified (README/DEV-article confirmed the transferable signal but never documented whether `wrong` actually decrements/down-ranks/prunes). A code read settles it as a YES with a verbatim mechanism, and the discriminator (oracle-free, user-supplied signal) makes it directly relevant to claude-memory-kit's own deferred trust_score-in-ranking question — Memoria is a working precedent for exactly the loop cmk defers: contradiction/relevance feedback that dampens AND feeds ranking.

**How to apply:** When citing Memoria as a learns-from-failure YES: cite the two load-bearing sites — (1) storage/store.rs `search_hybrid_from_scored_scoped` feedback-delta rerank block (the per-retrieval down-rank, the closed loop) and (2) service/scoring.rs `DefaultScoringPlugin::adjust_score` + `tune_params` (the meta-tune). Say CODE-level (not README). Classify the signal as transferable (user/agent relevance feedback, no ground-truth oracle). Distinguish it from governance quarantine (confidence-decay housekeeping, NOT feedback-driven). For cmk's own design: Memoria's feedback_weight*(useful-0.5*negative) clamp(0.5,2.0) multiplier is a concrete, oracle-free template for turning cmk's already-computed trust_score into a ranking multiplier — the exact thing cmk currently computes-but-doesn't-rank-on.
